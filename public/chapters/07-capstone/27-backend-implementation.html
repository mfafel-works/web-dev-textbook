Chapter 27

Backend Implementation

Create API endpoints, design your database, handle forms, and add authentication — and understand why each decision matters.

This chapter builds the backend for your capstone project. We'll cover API structure, database design, form handling, and authentication basics. Each section explains the why behind our implementation choices.

Project Structure

A well-organized backend makes maintenance easier and helps others understand your code.

backend/ ├── public/ # Static files (if any) ├── src/ │ ├── config/ # Configuration (DB, env) │ ├── controllers/ # Request handling logic │ ├── models/ # Database models (if using ORM) │ ├── routes/ # Route definitions │ ├── middleware/ # Auth, logging, error handling │ ├── services/ # Business logic │ ├── index.php # Entry point │ └── db.php # Database connection ├── .htaccess # Apache routing (if applicable) ├── composer.json # PHP dependencies └── phpunit.xml # Test configuration
Why This Structure?

Separation of concerns: routes handle URL routing, controllers handle request/response, services handle business logic. This makes testing easier, allows multiple controllers to share logic, and keeps files focused. As your project grows, this structure scales — unlike putting everything in one file.

Database Design

A well-designed database is the foundation of a reliable application. Spend time here — fixing schema mistakes later is expensive.

Designing Your Schema

Start by identifying your entities. For a typical web app with user accounts, content, and contact forms, you might need:

  • Users: id, email, password_hash, name, created_at
  • Posts/Content: id, title, content, author_id, status, created_at, updated_at
  • Contact Submissions: id, name, email, message, created_at, status
SQL schema example
CREATE TABLE users (
  id INT AUTO_INCREMENT PRIMARY KEY,
  email VARCHAR(255) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  name VARCHAR(255),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE posts (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  content TEXT,
  author_id INT NOT NULL,
  status ENUM('draft', 'published') DEFAULT 'draft',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (author_id) REFERENCES users(id)
);

CREATE TABLE contact_submissions (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  email VARCHAR(255) NOT NULL,
  message TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

API Endpoints

Your API is the contract between frontend and backend. Design it thoughtfully — changing it later breaks existing clients.

RESTful Design

MethodEndpointAction
GET/api/postsList all posts
GET/api/posts/{id}Get single post
POST/api/postsCreate new post
PUT/api/posts/{id}Update post
DELETE/api/posts/{id}Delete post
PHP endpoint example
<?php
header('Content-Type: application/json');
header('Access-Control-Allow-Origin: *');

$method = $_SERVER['REQUEST_METHOD'];
$uri = parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH);

// Simple router
if ($method === 'GET' && $uri === '/api/posts') {
    $stmt = $pdo->query("SELECT * FROM posts WHERE status = 'published'");
    echo json_encode($stmt->fetchAll());
} elseif ($method === 'POST' && $uri === '/api/posts') {
    $data = json_decode(file_get_contents('php://input'), true);
    $stmt = $pdo->prepare("INSERT INTO posts (title, content, author_id) VALUES (?, ?, ?)");
    $stmt->execute([$data['title'], $data['content'], $data['author_id']]);
    echo json_encode(['id' => $pdo->lastInsertId()]);
}

Form Handling

The backend needs to validate and process data from forms. Never trust client-side validation alone.

Contact form endpoint
<?php
header('Content-Type: application/json');

$input = json_decode(file_get_contents('php://input'), true);

// Validate
$errors = [];
if (empty($input['name'])) {
    $errors[] = 'Name is required';
}
if (empty($input['email']) || !filter_var($input['email'], FILTER_VALIDATE_EMAIL)) {
    $errors[] = 'Valid email is required';
}
if (empty($input['message'])) {
    $errors[] = 'Message is required';
}

if ($errors) {
    http_response_code(400);
    echo json_encode(['errors' => $errors]);
    exit;
}

// Store in database
$stmt = $pdo->prepare("INSERT INTO contact_submissions (name, email, message) VALUES (?, ?, ?)");
$stmt->execute([$input['name'], $input['email'], $input['message']]);

http_response_code(201);
echo json_encode(['message' => 'Submission received']);
Why Validate on the Backend?

Client-side validation improves UX (instant feedback), but malicious users can bypass it by sending requests directly to your API. Backend validation is your last line of defense — always validate there too.

Authentication

If your app needs user accounts, authentication keeps unauthorized users out of protected areas.

Password Security

Never store plain-text passwords. Use PHP's password_hash for secure storage.

Password hashing
// Creating hash (registration)
$password = $_POST['password'];
$hash = password_hash($password, PASSWORD_DEFAULT);
$stmt = $pdo->prepare("INSERT INTO users (email, password_hash) VALUES (?, ?)");
$stmt->execute([$email, $hash]);

// Verifying (login)
$stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?");
$stmt->execute([$email]);
$user = $stmt->fetch();

if ($user && password_verify($password, $user['password_hash'])) {
    // Valid login - create session
    $_SESSION['user_id'] = $user['id'];
}

Sessions

Sessions store user state on the server. PHP handles this natively.

Session-based auth
// Start session (at top of any file needing auth)
session_start();

// Check if logged in
function requireAuth() {
    if (empty($_SESSION['user_id'])) {
        http_response_code(401);
        echo json_encode(['error' => 'Unauthorized']);
        exit;
    }
}

// Protect routes
requireAuth();

// Now you can safely access $_SESSION['user_id']
