Final Considerations

Test your application, deploy to production, and reflect on what you've built — with a deep-dive into two deployment options.

This final chapter covers testing strategies, deployment options (with detailed guidance on my top pick and the most common industry standard), and reflection questions to help you internalize what you've learned.

Testing Strategies

Testing isn't about proving your code works — it's about building confidence that it keeps working as you make changes.

Types of Tests

  • Unit tests: Test individual functions in isolation. Fast, focused.
  • Integration tests: Test how components work together. More realistic but slower.
  • End-to-end (E2E) tests: Simulate real user interactions. Slowest but most realistic.

When to Test

New feature? → Write test first (TDD) Bug fix? → Write test to reproduce bug, then fix Legacy code? → Add tests when you touch it, not before
The Testing Pyramid

Most tests should be unit tests (fast, cheap). Fewer integration tests. Fewest E2E tests (slow, expensive). A typical project might have 70% unit, 20% integration, 10% E2E. Don't test everything — test the stuff that matters: business logic, critical paths, things that would cause real problems if broken.

Deployment Option 1: Cloudflare Pages (My Top Pick)

Cloudflare Pages is my personal favorite for modern web projects. It combines performance, simplicity, and an excellent free tier.

Why Cloudflare Pages?

  • Global CDN: Your site is served from 300+ locations worldwide
  • Automatic HTTPS: Free SSL with custom domain support
  • Zero config: Connect your GitHub repo, it builds and deploys automatically
  • Workers: Serverless functions included if you need backend logic
  • Free tier: Generous limits suitable for personal projects and portfolios

Deploying a React App

Connect GitHub repository
1. Go to cloudflare.com → Pages
2. Click "Create a project"
3. Select your GitHub repository
4. Configure build settings:
   - Production branch: main
   - Build command: npm run build
   - Build output directory: dist
Adding environment variables
In Cloudflare Pages dashboard:
Settings → Environment Variables

Add:
VITE_API_URL = https://your-api.example.com
VITE_APP_NAME = My App
The "Why" Behind This Choice

Cloudflare Pages removes the need to manage servers, configure Nginx, or handle SSL certificates. It just works. The free tier is generous enough for most personal projects and portfolios. When your project grows, you can add Cloudflare Workers for backend logic — same platform, no new infrastructure to learn.

Backend Integration

If using PHP with Cloudflare Pages, you have two options:

  • Workers: Deploy PHP via Cloudflare Workers (requires third-party like Workers KV)
  • Separate hosting: Keep frontend on Cloudflare Pages, backend on traditional PHP host

Deployment Option 2: Traditional PHP Hosting (Industry Standard)

Traditional shared or VPS hosting with PHP remains the most common deployment method for PHP applications. It's what you'll encounter at most web hosting companies.

Common Providers

  • Shared hosting: SiteGround, Hostinger, Bluehost, GoDaddy — cheap, limited control
  • VPS: DigitalOcean, Linode, Hetzner — more control, more responsibility
  • Managed WordPress: WP Engine, Kinsta — if using WordPress

Deploying via FTP/SFTP

Manual upload process
1. Build your frontend: npm run build
2. Connect to hosting via SFTP (FileZilla, Transmit)
3. Upload contents of /dist to public_html/ or /htdocs/
4. For PHP backend: upload to same directory
5. Configure .htaccess for routing if needed

The .htaccess File

Example .htaccess for clean URLs
<IfModule mod_rewrite.c>
  RewriteEngine On
  RewriteBase /
  
  # Redirect to index for SPA
  RewriteCond %{REQUEST_FILENAME} !-f
  RewriteCond %{REQUEST_FILENAME} !-d
  RewriteRule ^ index.html [,L]
</IfModule>
The "Why" Behind Traditional Hosting

Many companies still run on traditional PHP hosting because it's familiar and "just works." Shared hosting is cheap, PHP runs everywhere, and there's no infrastructure to manage. The tradeoff: less control, potential resource limits, and occasionally outdated software. For PHP-centric projects or when working with legacy code, this knowledge is essential.

Reflection Questions

Take time to think about what you've built and what you've learned. These questions help solidify the knowledge.

Questions to Ask Yourself
  • What was the hardest part? What did it teach you?
  • What would you do differently? Now that you understand more, what would change?
  • Which decision felt right immediately? Which one surprised you?
  • What would you add next? Feature ideas, performance improvements, tests?
  • How did the "why" questions change your approach? Did understanding rationale change how you built things?

Food for Thought: What's Next?

You've built a complete project from scratch. Here's where you could go from here:

Skills to Build Next

  • Testing: Learn PHPUnit for PHP, Vitest/Jest for JavaScript
  • TypeScript: Add type safety to your JavaScript projects
  • Database optimization: Learn indexing, query optimization, caching
  • CI/CD: Set up GitHub Actions for automated testing and deployment
  • Docker: Containerize your app for consistent environments
  • Monitoring: Add error tracking, performance monitoring in production

Topics Worth Exploring

  • GraphQL as an alternative to REST
  • WebSockets for real-time features
  • Static site generation (Next.js, Astro)
  • Headless CMS options (Strapi, Sanity)
  • Serverless architecture in depth
