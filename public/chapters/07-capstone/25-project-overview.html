Chapter 25

Project Overview

Build an SPA + config-driven CMS + contact system with pagination, validation, and bot protection — step by step

This chapter walks through building a complete single-page application from scratch. The project uses no frameworks — just HTML, CSS, and JavaScript — with a JSON-based content system that behaves like a headless CMS. Every decision is explained: not just what to write, but why.

Project Overview and Architecture

We are building a content-driven SPA. The architecture follows this pattern:

config.json → fetch layer → routing engine → render system → DOM output

Each layer has a single responsibility:

  • config.json — stores all content: pages, routes, metadata, settings
  • Fetch layer — loads config.json at startup, caches it in memory
  • Routing engine — maps the current URL to a page key
  • Render system — converts page data into HTML and inserts it into the DOM

The project folder structure looks like this:

my-spa/ ├── index.html # SPA entry point — loaded once ├── config.json # All content lives here ├── css/ │ └── style.css # Application styles ├── js/ │ ├── app.js # Init: fetch config, start router │ ├── router.js # URL → page mapping │ └── renderer.js # JSON → DOM rendering └── assets/ # Images, fonts, etc.

Key rule: The config.json is the source of truth. The frontend never hardcodes content — it always reads from config. This means you can later swap the JSON file for a Strapi CMS API without touching your frontend code.

Step 1: Define the Content Schema

The entire site is driven by a single JSON file. Here is the schema structure:

config.json — content schema
{
  "site": {
    "title": "My SPA Site",
    "description": "A config-driven single-page application"
  },
  "routes": {
    "/": "home",
    "/about": "about",
    "/contact": "contact"
  },
  "pages": {
    "home": {
      "title": "Home",
      "blocks": [
        { "type": "hero", "text": "Welcome to My Site" }
      ]
    },
    "about": {
      "title": "About",
      "blocks": [
        { "type": "text", "content": "This site is built with a config-driven CMS pattern." }
      ]
    },
    "contact": {
      "title": "Contact",
      "blocks": [
        { "type": "text", "content": "Get in touch using the form below." }
      ]
    }
  }
}

Three core concepts here:

  • routes — maps URL paths to page keys (makes routing a pure lookup — no business logic)
  • pages — the actual content, keyed by page identifier
  • blocks — an array of content sections. Each block has a type that maps to a renderer function. This is how we build a scalable component system without a framework.

The HTML file is minimal — it provides the app container and loads the scripts:

The #app div is where all page content will be rendered. The #nav div will be populated dynamically from the config. Nothing is hardcoded in the HTML.

Step 3: Load Config at Runtime

The fetch layer is simple — load the JSON file once and keep it in memory:

app.js — fetch and init
let CMS;

async function init() {
  const res = await fetch('config.json');
  CMS = await res.json();

  buildNav();
  navigate();
}

function buildNav() {
  const nav = document.getElementById('nav');
  nav.innerHTML = Object.keys(CMS.routes).map(path => {
    const key = CMS.routes[path];
    return `<a href="#${path}">${CMS.pages[key].title}</a>`;
  }).join('');
}

window.addEventListener('hashchange', navigate);

init();

Three things happen at startup: the config loads, the navigation is built from route data, and the current route is rendered. On every hash change, navigate() runs again.

Step 4: Build the Router

The router is a pure mapping layer — it takes the current URL and returns a page key. No rendering logic, no business rules:

router.js — hash-based routing
function getRoute() {
  const hash = window.location.hash.slice(1) || '/';
  return hash;
}

function resolvePage(path) {
  const key = CMS.routes[path];
  if (!key) return null;  // 404
  return CMS.pages[key];
}

function navigate() {
  const path = getRoute();
  const page = resolvePage(path);

  if (!page) {
    renderNotFound();
    return;
  }

  renderPage(page);
}

If the route doesn't exist, resolvePage returns null and we render a 404. Notice the router never touches the DOM — it only resolves data. The renderer handles presentation.

Step 5: Build the Render Engine

The render engine converts page data into HTML. It uses a block system — each block type maps to a function that produces the right markup:

renderer.js — block-based rendering
function renderPage(page) {
  const html = page.blocks.map(renderBlock).join('');
  document.getElementById('app').innerHTML =
    `<h1>${page.title}</h1>${html}`;
  document.title = `${page.title} — ${CMS.site.title}`;
}

function renderBlock(block) {
  switch (block.type) {
    case 'hero':
      return `<section class="hero">
        <h2>${block.text}</h2>
      </section>`;
    case 'text':
      return `<section class="text-block">
        <p>${block.content}</p>
      </section>`;
    case 'image':
      return `<figure>
        <img src="${block.src}" alt="${block.alt}">
      </figure>`;
    default:
      return `<p>Unknown block type: ${block.type}</p>`;
  }
}

function renderNotFound() {
  document.getElementById('app').innerHTML =
    `<h1>404</h1>
     <p>Page not found.</p>
     <a href="#/">Go home</a>`;
}
Why This Pattern Matters

This block system is the same mental model that React, Vue, and other frameworks use — just without the framework. You define components (block types), map data to them, and compose them into pages. The difference is you wrote it yourself, so you understand every line. Later, when you learn React, you will recognize this pattern immediately — it's just switch(block.type) becomes <Component />.

Step 6: Navigation Generation

The navigation is built dynamically from the config. It uses hash links (#/about) so the SPA routing works without a server. The active state is highlighted based on the current hash:

Active navigation highlighting
function buildNav() {
  const currentPath = getRoute();
  const nav = document.getElementById('nav');
  nav.innerHTML = Object.entries(CMS.routes).map(([path, key]) => {
    const active = path === currentPath ? ' class="active"' : '';
    return `<a href="#${path}"${active}>${CMS.pages[key].title}</a>`;
  }).join('');
}

// Update active state on navigation
function navigate() {
  const path = getRoute();
  const page = resolvePage(path);
  if (!page) { renderNotFound(); return; }
  renderPage(page);
  buildNav();  // re-render nav to update active class
}

Step 7: Adding New Content

To add a new page, you only touch config.json:

Adding a new page — config.json only
{
  "routes": {
    "/": "home",
    "/about": "about",
    "/contact": "contact",
    "/blog": "blog"           // ← new route
  },
  "pages": {
    "blog": {                 // ← new page
      "title": "Blog",
      "blocks": [
        { "type": "text", "content": "Welcome to the blog." },
        { "type": "image", "src": "assets/blog-header.jpg", "alt": "Blog header" }
      ]
    }
  }
}

No HTML changes. No routing code changes. No navigation code changes. The navigation builds itself from the routes object. The router resolves the new route automatically. The renderer handles any existing block type. This is decoupled architecture in action.

Form submission handler

app.js — form handling with validation
function setupContactForm() {
  const form = document.getElementById('contact-form');
  if (!form) return;

  form.addEventListener('submit', async (e) => {
    e.preventDefault();

    const formData = new FormData(form);
    const data = {
      name: formData.get('name').trim(),
      email: formData.get('email').trim(),
      message: formData.get('message').trim()
    };

    // Client-side validation
    if (!data.name || !data.email || !data.message) {
      showFormStatus('All fields are required.', 'error');
      return;
    }
    if (!data.email.includes('@')) {
      showFormStatus('Please enter a valid email.', 'error');
      return;
    }

    // Disable button to prevent double-submit
    const btn = form.querySelector('button');
    btn.disabled = true;
    btn.textContent = 'Sending...';

    try {
      const res = await fetch(data.endpoint, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(data)
      });

      if (!res.ok) throw new Error('Server error');
      showFormStatus('Message sent!', 'success');
      form.reset();
    } catch (err) {
      showFormStatus('Something went wrong. Try again.', 'error');
    } finally {
      btn.disabled = false;
      btn.textContent = 'Send';
    }
  });
}

function showFormStatus(msg, type) {
  const el = document.getElementById('form-status');
  if (el) { el.textContent = msg; el.className = type; }
}

Note the submit event is on the <form> element, not a click on the button — this catches Enter-key submissions. The Turnstile widget div is included in the HTML but its verification happens server-side (covered next).

FormData vs JSON: Choosing the Right Encoding

The code above uses new FormData(form) to extract field values, then sends them as application/json. This works for text fields but silently discards file inputs — JSON.stringify cannot serialize File objects. If your form includes <input type="file">, you must send the raw FormData directly and omit the Content-Type header (the browser sets multipart/form-data with the correct boundary automatically). Setting Content-Type: application/json while passing a FormData body will also break — the boundary parameter is missing. Either use JSON for text-only payloads or FormData for file uploads; never mix the two approaches.

Choosing a backend for the form

You have two good options for handling form submissions:

For this project, the backend/ folder contains a PHP endpoint that receives the POST, validates the Turnstile token, and forwards the message via Resend's API. The key architectural point: the frontend doesn't care which backend handles the form — it just sends JSON to the configured endpoint.

Step 9: Bot Protection with Turnstile

Cloudflare Turnstile is a user-friendly CAPTCHA replacement. Users check a box (or pass invisibly), and you verify the token server-side. It's free, privacy-respecting, and works without showing distorted text images.

Add the widget to your form

The Turnstile widget div is already in the contact form HTML above. Replace YOUR_SITE_KEY with your actual site key from the Cloudflare dashboard:

HTML widget (already included in form)
<div class="cf-turnstile" data-sitekey="0x4AAAAAA..."></div>

The Turnstile script is loaded automatically from Cloudflare's CDN when the widget is rendered.

Submit the token with your form data

When the form is submitted, Turnstile automatically includes a token in the form data under cf-turnstile-response. The frontend sends it along with the rest of the data:

Token is included automatically — no extra frontend code
// The FormData object already contains cf-turnstile-response.
// Just send everything to your backend as shown in Step 8.

Verify the token server-side

The backend must verify the token before processing the form. This is critical — never trust the frontend alone:

PHP — Turnstile verification
function verifyTurnstile($token) {
  $ch = curl_init(
    'https://challenges.cloudflare.com/turnstile/v0/siteverify'
  );
  curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
  curl_setopt($ch, CURLOPT_POSTFIELDS, [
    'secret' => 'YOUR_SECRET_KEY',
    'response' => $token
  ]);
  $result = json_decode(curl_exec($ch), true);
  curl_close($ch);
  return $result['success'] === true;
}

Common failure point: SPA routing breaks the token lifecycle

In an SPA, the page doesn't reload when navigating. If a user navigates away from the contact page and back, the Turnstile widget may have an expired token. The fix: ensure the token is revalidated on every submission. The widget handles this automatically if you don't manipulate its DOM container — just don't cache the rendered form HTML across navigations.

In our SPA, each navigation re-renders the page content, so the widget is re-initialized fresh every time. This is the correct behavior.

Never Trust Frontend Validation Alone

Client-side validation is for user experience — it catches typos and missing fields instantly. Server-side validation is for security — it catches everything else. Turnstile verification happens server-side because a bot can easily bypass anything running in the browser. Always verify the token on your server before accepting the form submission.

Step 10: Sliding Window Pagination

A blog, portfolio, or product listing needs pagination. Our block-based system handles this naturally — pagination is just another block type. The content data lives in config.json, the renderer displays a window of items, and JavaScript handles page switching.

Config Schema

Add a paginated-list block to any page. The data array can hold hundreds of items — the pagination logic only renders a slice:

config.json — paginated list block
{
  "type": "paginated-list",
  "perPage": 6,
  "items": [
    { "title": "Post 1", "date": "2026-01-15", "summary": "..." },
    { "title": "Post 2", "date": "2026-01-20", "summary": "..." }
    // ... many more items
  ]
}

Render Engine: Paginated List Handler

The renderer gets the block data, identifies the current page from a data attribute, and renders only the items for that page:

renderer.js — paginated list render
function renderPaginatedList(block) {
  const page = block._page || 1;
  const perPage = block.perPage || 10;
  const totalPages = Math.ceil(block.items.length / perPage);
  const start = (page - 1) * perPage;
  const end = start + perPage;
  const pageItems = block.items.slice(start, end);

  let html = '<div class="paginated-list" data-block="' + block._key + '">';

  // Render items
  html += '<div class="paginated-items">';
  for (const item of pageItems) {
    html += '<article class="paginated-item">';
    html += '<h3>' + escapeHtml(item.title) + '</h3>';
    html += '<time>' + item.date + '</time>';
    html += '<p>' + escapeHtml(item.summary) + '</p>';
    html += '</article>';
  }
  html += '</div>';

  // Pagination controls
  html += '<nav class="pagination" aria-label="Pagination">';
  html += buildPagination(totalPages, page, block._key);
  html += '</nav>';

  html += '</div>';
  return html;
}

The Sliding Window Algorithm

Displaying all page numbers works for 10 pages but breaks at 100. The sliding window shows a fixed set of page numbers around the current page with ellipsis for gaps. The algorithm:

Always show: first page + last page + window around current page + ellipsis between gaps
renderer.js — sliding window pagination
function buildPagination(totalPages, currentPage, blockKey) {
  if (totalPages <= 1) return '';

  const windowSize = 3; // Pages on each side of current
  let pages = [];

  // Always include page 1
  pages.push(1);

  // Calculate window start/end
  let windowStart = Math.max(2, currentPage - windowSize);
  let windowEnd = Math.min(totalPages - 1, currentPage + windowSize);

  // Adjust window if it overlaps with edges
  if (windowEnd - windowStart < windowSize * 2) {
    if (currentPage < totalPages / 2) {
      windowEnd = Math.min(totalPages - 1, windowStart + windowSize * 2);
    } else {
      windowStart = Math.max(2, windowEnd - windowSize * 2);
    }
  }

  // Add ellipsis before window if needed
  if (windowStart > 2) {
    pages.push('...');
  }

  // Add window pages
  for (let i = windowStart; i <= windowEnd; i++) {
    pages.push(i);
  }

  // Add ellipsis after window if needed
  if (windowEnd < totalPages - 1) {
    pages.push('...');
  }

  // Always include last page
  if (totalPages > 1) {
    pages.push(totalPages);
  }

  // Build HTML
  let html = '';
  for (const p of pages) {
    if (p === '...') {
      html += '<span class="pagination-ellipsis">&hellip;</span>';
    } else if (p === currentPage) {
      html += '<span class="pagination-current" aria-current="page">' + p + '</span>';
    } else {
      html += '<button class="pagination-page" data-page="' + p + '" data-block="' + blockKey + '">' + p + '</button>';
    }
  }

  return html;
}

Pagination Event Handler

Clicking a page button triggers a re-render of only that paginated block — no full page navigation, no full re-render:

app.js — pagination click handler
document.addEventListener('click', function(e) {
  const btn = e.target.closest('.pagination-page');
  if (!btn) return;

  const page = parseInt(btn.dataset.page);
  const blockKey = btn.dataset.block;
  const block = findBlockByKey(blockKey);

  if (!block) return;

  block._page = page;
  const container = document.querySelector('[data-block="' + blockKey + '"]');
  if (container) {
    container.outerHTML = renderPaginatedList(block);
  }
});
Why Sliding Window Instead of "Previous / Next" Only

Simple "Prev / Next" navigation forces the user to click through every intermediate page to reach a faraway page. Numbered pagination with a sliding window gives direct access to any page while staying compact. The window size of 3 on each side (7 visible numbers + 2 ellipsis) is a good default — it fits on mobile and provides enough context for the user to orient themselves.

Edge Cases Handled

  • Single page: No pagination controls rendered at all
  • Fewer pages than window: Window collapses naturally — no duplicate pages or negative indices
  • First page: Window starts at 2, ellipsis appears before last page if needed
  • Last page: Window ends at totalPages - 1, ellipsis after first page if needed
  • No items: totalPages is 0, no pagination rendered
  • Concurrent paginated blocks: Each block has a unique data-block key, so pagination is independent per block
