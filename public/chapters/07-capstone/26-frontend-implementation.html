Chapter 26

Frontend Implementation

Build the user interface with React, Vite, components, routing, and state management — and understand why each choice matters.

This chapter walks through building the frontend of your capstone project. We'll cover setup, component architecture, routing, state management, and connecting to your backend. Every decision explains the why — not just what to code.

Project Setup with Vite

Vite has become the standard build tool for modern frontend development. Its speed and developer experience make it the natural choice for new projects.

Creating a new project
npm create vite@latest my-app -- --template react
cd my-app
npm install
npm run dev

Why Vite?

Earlier build tools like Webpack parse all files upfront — slow on large projects. Vite uses native ES modules in development (instant server start), only bundling for production. The difference is noticeable: projects that took 30+ seconds to start now start in under 1 second.

Why Not Create React App?

Create React App (CRA) is no longer recommended by the React team. It uses Webpack under the hood, meaning slow builds. Vite is the official recommended alternative. If you've used CRA before, Vite's workflow will feel familiar but noticeably faster.

Component Architecture

Components are the building blocks of React applications. But organizing them effectively requires thought — the difference between a maintainable app and a tangled mess often comes down to structure.

Folder Structure

src/ ├── components/ # Reusable UI components │ ├── Button.jsx │ ├── Card.jsx │ └── Header.jsx ├── pages/ # Page-level components (routes) │ ├── Home.jsx │ ├── About.jsx │ └── Contact.jsx ├── hooks/ # Custom hooks │ └── useAuth.js ├── services/ # API communication │ └── api.js ├── App.jsx # Main app component ├── main.jsx # Entry point └── index.css # Global styles

Component Design Principles

  • Single responsibility: Each component does one thing well. A Button handles buttons — not also fetching data.
  • Prop-driven: Components receive data via props, not from global state directly (except context).
  • Presentational vs Container: Presentational components handle UI only; containers handle data and logic. This separation makes testing and reuse easier.

Routing

Client-side routing gives your app a native feel — pages change without full reloads. React Router is the standard solution.

Installing React Router
npm install react-router-dom
App.jsx with routing
import { BrowserRouter, Routes, Route } from 'react-router-dom';
import Home from './pages/Home';
import About from './pages/About';
import Contact from './pages/Contact';

function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
        <Route path="/contact" element={<Contact />} />
      </Routes>
    </BrowserRouter>
  );
}

Why Client-Side Routing?

Traditional server-side routing sends a request to the server for each URL — slow, requires full page reload. Client-side routing intercepts URL changes in JavaScript, swaps components instantly. The user experience is smoother, and your app feels more like a native application.

State Management

As your app grows, managing state becomes critical. The key is choosing the right level of complexity for your needs.

Local State with useState

Simple state example
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Count: {count}
    </button>
  );
}

Global State with useContext

For state that multiple components need (user authentication, theme, cart), React Context avoids prop-drilling through many layers.

Creating context
const AuthContext = createContext();

function AuthProvider({ children }) {
  const [user, setUser] = useState(null);

  const login = (userData) => setUser(userData);
  const logout = () => setUser(null);

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}
When to Add State Libraries

If you find yourself passing props through 4+ component levels, or if multiple unrelated components need the same data, reach for Context. Only reach for Redux, Zustand, or similar when you have complex derived state, middleware needs, or time-travel debugging requirements. For most projects, Context + useState is sufficient.

Connecting to the Backend

Your frontend needs to communicate with your API. Centralize this communication rather than scattering fetch calls throughout components.

API service module
const API_BASE = import.meta.env.VITE_API_URL;

async function fetchJSON(endpoint, options = {}) {
  const response = await fetch(`${API_BASE}${endpoint}`, {
    headers: {
      'Content-Type': 'application/json',
      ...options.headers,
    },
    ...options,
  });

  if (!response.ok) {
    const error = await response.json().catch(() => ({}));
    throw new Error(error.message || `HTTP ${response.status}`);
  }

  return response.json();
}

export const api = {
  get: (endpoint) => fetchJSON(endpoint),
  post: (endpoint, data) => fetchJSON(endpoint, { 
    method: 'POST', 
    body: JSON.stringify(data) 
  }),
  // ... put, delete
};

Why a Service Layer?

Direct fetch calls in components create duplication, make error handling inconsistent, and complicate changes (like adding authentication later). A service layer centralizes: base URL configuration, headers, error handling, and response transformation. One place to change, everywhere benefits.

Form Handling

Forms are where users input data. Good form handling improves user experience dramatically.

Controlled form component
function ContactForm() {
  const [formData, setFormData] = useState({ name: '', email: '', message: '' });
  const [errors, setErrors] = useState({});
  const [submitting, setSubmitting] = useState(false);

  const validate = () => {
    const errs = {};
    if (!formData.name.trim()) errs.name = 'Name is required';
    if (!formData.email.includes('@')) errs.email = 'Valid email required';
    if (!formData.message.trim()) errs.message = 'Message is required';
    return errs;
  };

  const handleSubmit = async (e) => {
    e.preventDefault();
    const errs = validate();
    if (Object.keys(errs).length) {
      setErrors(errs);
      return;
    }

    setSubmitting(true);
    try {
      await api.post('/contact', formData);
      alert('Message sent!');
      setFormData({ name: '', email: '', message: '' });
    } catch (err) {
      alert('Failed to send message');
    } finally {
      setSubmitting(false);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input 
        value={formData.name} 
        onChange={(e) => setFormData({...formData, name: e.target.value})}
        placeholder="Your name"
      />
      {errors.name && <span className="error">{errors.name}</span>}
      {/* more fields... */}
      <button type="submit" disabled={submitting}>
        {submitting ? 'Sending...' : 'Send'}
      </button>
    </form>
  );
}
