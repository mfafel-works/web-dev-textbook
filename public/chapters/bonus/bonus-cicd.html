CI/CD Fundamentals

Automated testing, GitHub Actions, and deployment automation

CI/CD stands for Continuous Integration / Continuous Deployment. It means: every time you push code, your tests run automatically and your site deploys automatically. No manual steps, no forgetting to run tests, no "it works on my machine" issues.

Why Automate?

Manual deployments are error-prone:

  • You forget to run tests before pushing
  • You deploy without checking for errors
  • Different machines have different setups
  • Rollbacks are manual and slow

CI/CD fixes all of this. You push code → tests run → if passing, deploys automatically.

What CI/CD Actually Is

CI (Continuous Integration): Every push triggers automated tests. Bad code gets caught before merging.

CD (Continuous Deployment): Passing code automatically deploys to production (or staging).

GitHub Actions Basics

GitHub Actions is free for public repos and included in GitHub. You define workflows in YAML files:

Basic workflow: .github/workflows/test.yml
name: Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test

This workflow: runs on every push to main and every PR, checks out code, sets up Node.js, installs dependencies, runs tests.

Automated Deployment

Add deployment after tests pass:

Deploy workflow: .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: cloudflare/pages-action@v1
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          projectName: my-app
          directory: dist

The needs: test means deployment only runs if tests pass. This is the key: bad code never deploys.

Environment Variables in CI

Never hardcode secrets. Use GitHub secrets:

Using secrets in workflows
steps:
  - run: npm run build
    env:
      API_URL: ${{ secrets.API_URL }}
      DATABASE_URL: ${{ secrets.DATABASE_URL }}

Go to Settings → Secrets and variables → Actions in your GitHub repo to add secrets.

Pull Request Previews

Every PR gets its own preview deployment:

Preview deployment
on:
  pull_request:
    types: [opened, synchronize, closed]

jobs:
  deploy-preview:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: cloudflare/pages-action@v1
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          projectName: my-app
          directory: dist
          # Creates preview-123.pr-preview.pages.dev
          branch: ${{ github.event.pull_request.number }}-pr-preview

Team members can test the actual deployed site before merging.

Getting Started

Start simple: add tests + CI first, add deployment later.

  1. Create .github/workflows/test.yml in your repo
  2. Push code — watch tests run automatically
  3. Once tests pass reliably, add deployment workflow
  4. Protect your main branch: require tests to pass before merging
Start Small

You don't need everything at once. A simple "run tests on push" workflow catches bugs immediately and only takes a few minutes to set up.
