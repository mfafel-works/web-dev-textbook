Debugging Your Project

Console techniques, DevTools, and solving common issues in your capstone project

Bugs are inevitable. The difference between junior and senior developers is not who makes fewer mistakes — it's who finds and fixes them faster. This bonus covers the debugging toolkit you need for your project.

The Console

The browser console is your first line of defense. Open DevTools (F12 or Cmd+Option+I) and click Console.

Console logging
// Basic logging
console.log("User logged in:", user);

// Warning - something might be wrong
console.warn("API response took 3 seconds");

// Error - something is wrong
console.error("Failed to save:", error);

// Table - for arrays/objects
console.table(users);
Console Tips

Right-click in console → "Clear console" to start fresh. Filter by Errors/Warnings to focus. Click any log to see the full stack trace.

Reading Error Messages

Errors tell you what's wrong — once you know how to read them:

Example error
TypeError: Cannot read property 'name' of undefined
    at handleSubmit (app.js:45:5)
    at HTMLFormElement.onclick (index.html:12:1)

Break this down:

  • TypeError: What type of problem — type mismatch, function call on wrong type
  • Cannot read property 'name' of undefined: What happened — you tried to access .name on something that's undefined
  • at handleSubmit (app.js:45:5): Where it happened — line 45, column 5, in handleSubmit function

Fix: Check if the variable is defined before accessing its properties.

Network Tab

For API-related bugs, the Network tab shows every request your app makes:

  • Check request succeeded: Look for 200 status codes
  • Check response body: Click a request → Response tab to see what the API returned
  • Check request headers: See what's being sent to the API
  • Timing: See how long each request took
Common API Issues

CORS error: Your frontend is on a different domain than your API. Fix by adding CORS headers to your PHP backend.

404 on API call: Check the URL path — maybe missing /api/ prefix or wrong endpoint.

500 from API: Check your PHP error logs — syntax error, database issue, or missing file.

Breakpoints

For complex bugs, pause execution and step through code:

  1. Open Sources tab in DevTools
  2. Click a line number to add a breakpoint (red dot)
  3. Trigger the action in your app
  4. Browser pauses at the breakpoint
  5. Use the right panel to inspect variables, step through lines
When to Use Breakpoints

Use console.log for quick checks. Use breakpoints when: you don't know where the bug is, you need to inspect many variables, or the bug happens in a complex flow.

Systematic Debugging

When stuck, follow this process:

1. Reproduce — can you make the bug happen consistently? → No: Might be race condition or environment-specific → Yes: Continue 2. Isolate — remove code until bug disappears 3. Fix — make the change, test it works 4. Verify — bug is gone, nothing else broke
The Most Important Skill

Reading error messages is learnable. The key is: don't panic, read carefully, look at the line number, check what's undefined or null. Most bugs follow patterns — the more you debug, the faster you recognize them.
