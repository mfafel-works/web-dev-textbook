SVG Diagram Coordinates

How to plan and compute x/y values for decision trees, flowcharts, and inline diagrams

Inline SVG diagrams power the decision trees and flowcharts throughout this textbook. Getting the coordinates right is the hardest part. This bonus explains the coordinate system we use, the formulas behind every x and y, and provides a template you can reuse for your own diagrams.

Every SVG Starts With a viewBox

The viewBox defines the coordinate grid. It's an invisible canvas — elements can go anywhere inside or even outside it, but only what's inside the viewBox is visible.

viewBox format
<svg viewBox="0 0 730 450" style="width:100%;max-width:730px;height:auto;">

The four numbers are:

ValueMeaning
0minimum X (left edge, always 0)
0minimum Y (top edge, always 0)
730width of the coordinate grid in SVG units
450height of the coordinate grid in SVG units

The max-width should match the viewBox width so the diagram renders at its intended aspect ratio. The viewBox height should be tall enough to contain all elements plus ~30px of bottom padding.

The Three Positioning Rules You Must Know

1. Text y is the baseline, not the top

When you write <text x="100" y="200">, the text sits on top of y=200 — the baseline is at y=200, and letters extend upward. For a 15px font at y=200, the text occupies roughly y=185 to y=200. Descenders (g, j, p, q, y) extend slightly below.

Text placement rule of thumb
text_y = box_y + (box_height / 2) + (font_size / 3)

// For a 36px box with 15px font:
text_y = 145 + (36 / 2) + (15 / 3) = 145 + 18 + 5 = 168

2. Rect origin is top-left

A <rect x="60" y="145" width="230" height="36"> starts at the top-left corner (60, 145) and extends 230px right and 36px down. The center is at x + width/2 = 175.

3. Arrowheads appear at x2, y2

A <line x1="175" y1="181" x2="175" y2="205" marker-end="url(#arrow04)"> draws the arrowhead at the end point (175, 205). This means the arrowhead points into whatever is at that position — the line should end a few px inside the target box.

The Grid Paper Approach

Before writing SVG, sketch your diagram on graph paper or in your head using this process:

  1. Choose vertical tracks — pick x positions for each column (left, center, right)
  2. Choose horizontal floors — pick y positions for each row (boxes, decision points)
  3. Place elements at intersections — a box sits at (column_center, floor), arrows connect between them
  4. Verify the budget — add up all heights + gaps to make sure everything fits in the viewBox

Start With the Floor Plan

Plan your Y floors first. List every element from top to bottom, estimate its height, add 15-25px gaps, and sum it all up. If the total exceeds your viewBox height, reduce gaps or shift elements closer together. Only then compute the X positions.

Common Pitfalls

  • Text outside the box — your text_y is probably the same as the box_y. Remember text y is the baseline, not the top. Use the formula above.
  • Arrow misses its target — the x2,y2 of a line should point at the box center, and the y should be a few px inside the box edge. If the arrowhead is outside the box, extend y2 by 10-15px.
  • Text overflows the pill — measure your text width before setting the pill width. A rough estimate: text_width ≈ chars × font_size × 0.65. Leave 10-15px padding on each side.
  • ViewBox cuts off the bottom — the last element's bottom edge (y + height, or text_y + 3) needs to be at least 15px less than the viewBox height. Add viewBox="0 0 W H+30" for breathing room.
  • Dark mode colors don't work — use CSS variables (var(--color-text, #1a1a1a)) instead of hardcoded colors so the diagram adapts to theme changes. The <defs> markers should use variables too.
