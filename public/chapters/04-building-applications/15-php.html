PHP

Form handling, sessions, MySQL, and pragmatic server-side programming

PHP powers 75% of the web (WordPress, Laravel, Wikipedia, Facebook's original backend). It is not fashionable but it is practical: you write a PHP file, put it on a server, and it runs. No build step, no process manager, no cold starts. This chapter covers the PHP you actually need for modern web work.

PHP Fundamentals

PHP is a server-side scripting language designed for the web. You embed PHP code inside HTML using <?php ... ?> tags. The server executes the PHP and sends the resulting HTML to the browser.

Basic Syntax

PHP embedded in HTML
<!DOCTYPE html>
<html>
<body>
  <h1>Welcome, <?= $username ?>!</h1>
  <!-- <?= is shorthand for <?php echo -->

  <?php if ($loggedIn): ?>
    <p>You are logged in.</p>
  <?php else: ?>
    <p>Please <a href="login.php">log in</a>.</p>
  <?php endif; ?>
</body>
</html>

Variables and Arrays

PHP variables start with $. No let or const — just $. Arrays can be indexed or associative:

PHP variables and arrays
<?php
$name = "Alice";
$age = 30;
$isAdmin = true;

// Indexed array
$users = ["Alice", "Bob", "Charlie"];

// Associative array (like a JS object)
$user = [
  "name" => "Alice",
  "email" => "alice@example.com",
  "role" => "admin"
];

echo $user["name"]; // "Alice"

// Superglobals — available everywhere
$method = $_SERVER["REQUEST_METHOD"];  // GET, POST, etc.
$input  = $_POST["email"];             // Form input
$query  = $_GET["search"];            // URL query parameter
?>

Control Structures

PHP supports the expected if, foreach, while, and switch. The colon syntax (if (condition): ... endif;) is idiomatic when mixing with HTML — it avoids searching for curly braces.

Working with Forms

Forms are PHP's bread and butter. A PHP file receives form data and processes it — typically validating, sanitizing, storing, and responding.

Form handler with validation
<?php
$errors = [];

if ($_SERVER["REQUEST_METHOD"] === "POST") {
    $name = trim($_POST["name"] ?? "");
    $email = trim($_POST["email"] ?? "");

    // Validation
    if ($name === "") {
        $errors[] = "Name is required";
    }
    if (!filter_var($email, FILTER_VALIDATE_EMAIL)) {
        $errors[] = "Valid email is required";
    }

    if (empty($errors)) {
        // Sanitize for output
        $safeName = htmlspecialchars($name, ENT_QUOTES, "UTF-8");
        $safeEmail = htmlspecialchars($email, ENT_QUOTES, "UTF-8");

        // Save to database, send email, etc.
        // ...

        header("Location: /thank-you.php");
        exit;
    }
}
?>

<?php if (!empty($errors)): ?>
  <ul class="errors">
    <?php foreach ($errors as $error): ?>
      <li><?= htmlspecialchars($error) ?></li>
    <?php endforeach; ?>
  </ul>
<?php endif; ?>

Key points: validate on the server (never trust the frontend), use htmlspecialchars() for output escaping (prevents XSS), always redirect after successful POST (prevents duplicate submissions on refresh).

Modern PHP 8+ Features

PHP has evolved significantly. Modern PHP (8.0+) includes named arguments, match expressions, constructor property promotion, readonly properties, enums, and union types. Features like match replace verbose switch statements: $result = match($status) { 200 => 'OK', 404 => 'Not Found', default => 'Unknown' }; PHP 8.1 added enums and fibers, 8.2 added readonly classes and standalone types. Write PHP 8 — treat older versions as legacy.

Sessions and Authentication

PHP sessions are file-based by default — when you call session_start(), PHP creates a unique session ID, sends it as a cookie, and stores session data on the server.

Login flow with sessions
<?php
session_start();

// Login
if ($_SERVER["REQUEST_METHOD"] === "POST") {
    $email = $_POST["email"];
    $password = $_POST["password"];

    // Fetch user from database
    $stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?");
    $stmt->execute([$email]);
    $user = $stmt->fetch();

    if ($user && password_verify($password, $user["password_hash"])) {
        $_SESSION["user_id"] = $user["id"];
        $_SESSION["role"] = $user["role"];
        session_regenerate_id(true);  // Prevent session fixation
        header("Location: /dashboard.php");
        exit;
    }
}

// Check auth on any page
function requireAuth() {
    if (empty($_SESSION["user_id"])) {
        header("Location: /login.php");
        exit;
    }
}

// Check role
function requireRole($role) {
    if (($_SESSION["role"] ?? "") !== $role) {
        http_response_code(403);
        die("Access denied.");
    }
}

// Logout
function logout() {
    session_destroy();
    header("Location: /login.php");
    exit;
}
?>
Always Use password_hash and password_verify

Never hash passwords with md5(), sha1(), or even plain hash("sha256", ...). PHP's password_hash() uses bcrypt with automatic salting. The password_verify() function checks the hash safely. This single function call prevents the most common authentication vulnerability.

PHP Variables Start with $ — The #1 Syntax Error

Every PHP variable must start with $. Forgetting it is the most common beginner mistake. name = "Alice" is a syntax error in PHP — it must be $name = "Alice". Also, PHP variables are case-sensitive: $name and $Name are different variables.

PHP echo vs print — They Look the Same But Aren't

Both output text, but with differences: echo is slightly faster and can output multiple values (echo $a, $b;). print returns 1 (so it can be used in expressions like $result = print "done"). For simple output, use echo. The main trap: beginners use print expecting it to work like JavaScript's console.log() — it won't.

PHP Arrays Start at 0 — But Can Be Associative

PHP has two array types: indexed (0-based) and associative (string keys). Both use the same $array[] syntax:

$fruits[0] = "apple";
$fruits[1] = "banana";

$user["name"] = "Alice";
$user["email"] = "alice@example.com";

Mixing them creates an "array with mixed keys" — valid but confusing. Choose one style per array. The trap: iterating a mixed array with a numeric for loop fails because the keys aren't sequential.

Database Access with PDO

PDO (PHP Data Objects) is a consistent interface for accessing databases in PHP. It supports MySQL, PostgreSQL, SQLite, and others with the same API. The most important feature: prepared statements that prevent SQL injection.

PDO connection and CRUD
<?php
// Connect
$dsn = "mysql:host=localhost;dbname=myapp;charset=utf8mb4";
$pdo = new PDO($dsn, "username", "password", [
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC
]);

// READ — parameterized query
$stmt = $pdo->prepare("SELECT * FROM posts WHERE id = ?");
$stmt->execute([$_GET["id"]]);
$post = $stmt->fetch();

// READ ALL
$posts = $pdo->query("SELECT * FROM posts ORDER BY created_at DESC")->fetchAll();

// CREATE
$stmt = $pdo->prepare("INSERT INTO posts (title, body) VALUES (?, ?)");
$stmt->execute([$_POST["title"], $_POST["body"]]);
$newId = $pdo->lastInsertId();

// UPDATE
$stmt = $pdo->prepare("UPDATE posts SET body = ? WHERE id = ?");
$stmt->execute([$_POST["body"], $_POST["id"]]);

// DELETE
$stmt = $pdo->prepare("DELETE FROM posts WHERE id = ?");
$stmt->execute([$_POST["id"]]);
?>

The ? placeholders in the SQL are replaced with the array values in execute(). The database receives the query structure and the data separately — the data is never interpreted as SQL. This is the only reliable way to prevent SQL injection.

Composer and Modern PHP Ecosystem

Composer is PHP's dependency manager (like npm for JavaScript). It handles autoloading, installs packages from Packagist, and manages version constraints.

Composer setup and usage
# Install Composer globally
php -r "copy('https://getcomposer.org/installer', 'composer-setup.php');"
php composer-setup.php --install-dir=/usr/local/bin --filename=composer

# Create a new project
composer init

# Install dependencies
composer require vlucas/phpdotenv
composer require --dev phpunit/phpunit

# Autoload your own classes (add to composer.json)
{
  "autoload": {
    "psr-4": {
      "App\\": "src/"
    }
  }
}
# Then run: composer dump-autoload

PSR-4 autoloading maps namespaces to directory structures. A class App\Controllers\UserController lives at src/Controllers/UserController.php. Use composer dump-autoload after updating the autoload config — no manual require or include needed.

Environment Configuration

Never hardcode configuration values. Use environment variables and a .env file with vlucas/phpdotenv:

.env and loading
# .env (never commit to Git)
DB_HOST=localhost
DB_NAME=myapp
DB_USER=app_user
DB_PASS=secure_password_here
APP_ENV=production

# config.php
require __DIR__ . '/vendor/autoload.php';
$dotenv = Dotenv\Dotenv::createImmutable(__DIR__);
$dotenv->load();

$dsn = "mysql:host={$_ENV['DB_HOST']};dbname={$_ENV['DB_NAME']};charset=utf8mb4";
$pdo = new PDO($dsn, $_ENV['DB_USER'], $_ENV['DB_PASS']);
Never Commit .env Files

Add .env to your .gitignore. Commit a .env.example file instead with placeholder values and documentation. Leaked credentials from committed .env files are the most common CI/CD security incident on GitHub.

Laravel: The Dominant PHP Framework

While you can build anything with vanilla PHP, Laravel is the standard for professional PHP development. It provides routing, ORM (Eloquent), templating (Blade), authentication, queues, testing, and more.

Laravel route and controller
// routes/web.php
Route::get('/posts', [PostController::class, 'index']);
Route::post('/posts', [PostController::class, 'store']);

// app/Http/Controllers/PostController.php
class PostController extends Controller
{
    public function index(): View
    {
        $posts = Post::latest()->paginate(10);
        return view('posts.index', ['posts' => $posts]);
    }

    public function store(Request $request): RedirectResponse
    {
        $validated = $request->validate([
            'title' => 'required|max:255',
            'body'  => 'required',
        ]);

        Post::create($validated);
        return redirect('/posts')->with('success', 'Post created!');
    }
}

Key Laravel concepts: Eloquent provides an ActiveRecord ORM (Post::find($id), Post::where('published', true)), Blade templates extend layouts and include components, Artisan CLI generates code (php artisan make:controller), and queues with Horizon handle async jobs.

Start with Vanilla, Scale with Laravel

For small projects (1-5 endpoints, simple database), vanilla PHP with PDO and a single-file router is faster to write and deploy. As complexity grows — multiple models, auth roles, email, queues, testing — Laravel's conventions save time and prevent architectural drift. The boundary is around 10+ database tables or 3+ developers.

Testing PHP Code

PHPUnit is the standard testing framework. Write tests for your business logic, HTTP endpoints, and database interactions.

PHPUnit test example
use PHPUnit\Framework\TestCase;

class UserTest extends TestCase
{
    public function test_user_can_be_created_with_valid_data(): void
    {
        $user = new User('Alice', 'alice@example.com');
        
        $this->assertSame('Alice', $user->name);
        $this->assertSame('alice@example.com', $user->email);
        $this->assertTrue($user->isValid());
    }

    public function test_invalid_email_throws_exception(): void
    {
        $this->expectException(\InvalidArgumentException::class);
        new User('Bob', 'not-an-email');
    }
}

Run tests with ./vendor/bin/phpunit or define a composer.json script: "scripts": { "test": "phpunit" }. Integrate with CI/CD — tests should run on every push and every pull request. Laravel projects include PHPUnit by default with php artisan test.

Test What Matters

Focus tests on business logic and critical paths: validation rules, pricing calculations, authentication flows, and API responses. Don't test PHP's core functions or simple getters. A good test suite catches regressions without becoming a maintenance burden — 50 well-written tests are better than 500 brittle ones.

Building a REST API

PHP can serve as a lightweight API backend. The modern approach: a single index.php entry point that reads the request URI and routes accordingly.

Simple PHP router
<?php
header("Content-Type: application/json");
header("Access-Control-Allow-Origin: https://mysite.com");
header("Access-Control-Allow-Methods: GET, POST");

$method = $_SERVER["REQUEST_METHOD"];
$path = parse_url($_SERVER["REQUEST_URI"], PHP_URL_PATH);

// Route
switch (true) {
    case $path === "/api/posts" && $method === "GET":
        $posts = $pdo->query("SELECT * FROM posts")->fetchAll();
        echo json_encode($posts);
        break;

    case $path === "/api/posts" && $method === "POST":
        $data = json_decode(file_get_contents("php://input"), true);
        $stmt = $pdo->prepare("INSERT INTO posts (title, body) VALUES (?, ?)");
        $stmt->execute([$data["title"], $data["body"]]);
        http_response_code(201);
        echo json_encode(["id" => $pdo->lastInsertId()]);
        break;

    default:
        http_response_code(404);
        echo json_encode(["error" => "Not found"]);
}
?>

For larger APIs, use a framework (Laravel, Slim, Symfony) that provides routing, middleware, validation, and error handling built-in. But for small projects, a single-file router like this is sufficient and keeps complexity low.

Junior Developer

How a Junior Might Think

  • Mixes PHP and HTML freely without separation of concerns
  • Uses mysql_* functions (deprecated and insecure)
  • Concatenates variables into SQL queries directly
  • Hashes passwords with md5()
  • Does not validate or sanitize form input server-side
Senior Developer

How a Senior Thinks

  • Uses MVC pattern or a framework (Laravel, Slim)
  • Always uses PDO with prepared statements
  • Validates and sanitizes all input server-side
  • Uses password_hash() / password_verify()
  • Separates business logic from presentation logic
