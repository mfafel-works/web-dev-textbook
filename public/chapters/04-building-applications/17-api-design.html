Chapter 17

API Design & REST

URL design, request/response patterns, error handling, pagination, versioning, CORS, rate limiting, and OpenAPI

An API (Application Programming Interface) is how two systems communicate over HTTP. A well-designed API is intuitive to use, consistent across endpoints, and handles errors gracefully. A poorly designed API creates confusion, bugs, and integration nightmares. This chapter covers REST API design principles that apply to any backend language.

REST Fundamentals

REST (Representational State Transfer) is an architectural style for APIs. It uses HTTP methods as verbs and URLs as nouns:

HTTP MethodActionExampleResult
GETRetrieveGET /api/usersReturns a list of users
POSTCreatePOST /api/usersCreates a new user
PUTReplacePUT /api/users/42Replaces user 42 entirely
PATCHPartial updatePATCH /api/users/42Updates only specified fields of user 42
DELETERemoveDELETE /api/users/42Deletes user 42

URL Conventions

  • Plural nouns: /api/users, not /api/user
  • Nested resources: /api/users/42/posts (posts belonging to user 42)
  • Actions as verbs only when necessary: POST /api/users/42/activate (non-CRUD action)
  • Version prefix: /api/v1/users — enables breaking changes without breaking existing clients
  • Consistent casing: snake_case or camelCase — pick one and use it everywhere
RESTful URL design — good vs bad
// GOOD — clear, consistent, RESTful
GET    /api/v1/users              // list users
POST   /api/v1/users              // create user
GET    /api/v1/users/42           // get user 42
PUT    /api/v1/users/42           // replace user 42
PATCH  /api/v1/users/42           // update user 42
DELETE /api/v1/users/42           // delete user 42
GET    /api/v1/users/42/posts     // user 42's posts

// BAD — inconsistent, confusing
GET    /api/getUsers              // verb in URL
POST   /api/createUser            // verb in URL
GET    /api/userInfo?uid=42       // inconsistent naming

Request and Response Formats

Standardize on JSON for both requests and responses. Set Content-Type: application/json and Accept: application/json headers.

Response Envelope

Wrap responses in a consistent structure that includes metadata:

Consistent response envelope
// Success response
{
  "data": { ... },           // the payload
  "meta": {                  // metadata (pagination, etc.)
    "page": 1,
    "totalPages": 5,
    "totalCount": 98
  }
}

// Success — list response
{
  "data": [ ... ],
  "meta": {
    "page": 1,
    "limit": 10,
    "total": 98,
    "hasMore": true
  }
}

// Error response
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Email is required",
    "details": [
      { "field": "email", "message": "must not be blank" }
    ],
    "requestId": "req_abc123"
  }
}

Error Handling

Use HTTP status codes correctly and consistently. Every endpoint should return appropriate error codes with descriptive messages.

ScenarioStatus CodeResponse
Validation failed400 Bad RequestList of which fields failed and why
Unauthenticated401 UnauthorizedMissing or invalid auth credentials
Forbidden403 ForbiddenAuthenticated but lacks permission
Not found404 Not FoundResource does not exist
Conflict409 ConflictDuplicate resource, stale version
Rate limited429 Too Many RequestsInclude Retry-After header
Server error500 Internal Server ErrorGeneric message, log the details internally

Never expose stack traces or internal details in error responses. Log them server-side and return a generic message. Include a requestId so you can correlate client reports with server logs.

Pagination

Any endpoint that returns a list must support pagination. Without it, a user with 100,000 records would crash the server and the client.

Cursor-based pagination (recommended)
// Request
GET /api/v1/posts?cursor=eyJpZCI6NDJ9&limit=20

// Response
{
  "data": [ ... ],
  "meta": {
    "nextCursor": "eyJpZCI6NjJ9",    // pass this for the next page
    "hasMore": true,
    "limit": 20
  }
}

// Cursor-based pagination is preferred over page/offset because:
// - Stable: new items inserted at the start don't shift page boundaries
// - Fast: WHERE id > cursor queries use indexes efficiently
// - The cursor is opaque (base64-encoded JSON or similar)
Offset-based pagination (simpler, use for small datasets)
// Request
GET /api/v1/users?page=2&limit=20

// Response
{
  "data": [ ... ],
  "meta": {
    "page": 2,
    "limit": 20,
    "total": 98,
    "totalPages": 5,
    "hasMore": true
  }
}

API Versioning

APIs change. Clients depend on them. Versioning lets you make breaking changes without breaking existing clients.

URL Prefix Versioning (Recommended)

/api/v1/users, /api/v2/users. The version is in the URL, making it explicit and cache-friendly. Maintain old versions for a deprecation period (6-12 months) with clear sunset headers.

Deprecation headers
// Response header when using a deprecated API version
Sunset: Sat, 01 Nov 2026 23:59:59 GMT
Deprecation: true

// This tells clients: "this version is going away, migrate now"
Prefer Additive Changes

A well-designed API rarely needs breaking changes. Adding a new field to a response does not break existing clients. Adding a new endpoint does not break anything. Only change behavior when you must — and when you do, version it. Most "version 2" APIs could have been additive changes on version 1.

CORS

CORS (Cross-Origin Resource Sharing) is a browser security mechanism. When your frontend at yoursite.com calls your API at api.yoursite.com, the browser sends a preflight OPTIONS request and checks the response headers.

CORS headers your server must send
Access-Control-Allow-Origin: https://yoursite.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400          // cache preflight for 24 hours

// For simple requests (GET, POST with Content-Type:
// application/x-www-form-urlencoded, text/plain, or multipart/form-data),
// no preflight is needed. All other requests trigger OPTIONS first.
CORS Is a Server-Side Configuration, Not a Client Fix

You cannot fix CORS from the frontend. No JavaScript library, no proxy can bypass CORS. The server must send the correct Access-Control-Allow-Origin header. During development, use * (allow any origin). In production, restrict to your specific domain.

fetch() Does Not Throw on HTTP Errors — 404 Is Not an Exception

The most common beginner bug: fetch() does not reject on HTTP errors. A 404 or 500 response is a successful network request — .then() runs, not .catch(). You must check response.ok or response.status and throw manually:

const response = await fetch('/api/users');
if (!response.ok) {
  throw new Error(`HTTP ${response.status}: ${response.statusText}`);
}
const data = await response.json();

Without this check, your code proceeds as if the data is valid, and await response.json() on a 404 error page returns garbage or throws.

Always Await response.json() — Never Assume the Body Exists

Even on a successful response, the body might be empty. Calling response.json() on an empty response throws "Unexpected end of JSON input." Always handle the empty case:

const response = await fetch('/api/data');
const text = await response.text();
if (!text) return null;  // empty response
const data = JSON.parse(text);

Or use a helper that checks for empty bodies automatically.

Rate Limiting

Rate limiting protects your API from abuse and ensures fair usage. Communicate limits through response headers:

Rate limit headers
// Every response includes your current rate limit status
X-RateLimit-Limit: 100         // requests allowed per window
X-RateLimit-Remaining: 42      // requests remaining in current window
X-RateLimit-Reset: 1680000000  // unix timestamp when the limit resets

// When exceeded — 429 Too Many Requests
Retry-After: 30                // seconds until you can retry

Common rate limiting strategies:

  • IP-based — limit by client IP. Simplest but unfair (office NAT shares one IP)
  • User-based — limit by authenticated user ID. Fairer, requires auth
  • Token bucket — each user gets a bucket of tokens that refills over time. Allows bursts
  • Sliding window — count requests in the last N minutes. More accurate than fixed window

OpenAPI Specification

OpenAPI (formerly Swagger) is the industry standard for documenting REST APIs. Write a YAML/JSON spec that describes every endpoint, and tools generate interactive documentation, client SDKs, and server stubs.

OpenAPI 3.0 example
openapi: 3.0.0
info:
  title: Blog API
  version: 1.0.0
paths:
  /posts:
    get:
      summary: Returns paginated posts
      parameters:
        - name: page
          in: query
          schema: { type: integer, default: 1 }
      responses:
        '200':
          description: A list of posts
          content:
            application/json:
              schema:
                type: object
                properties:
                  data:
                    type: array
                    items:
                      $ref: '#/components/schemas/Post'
                  meta:
                    type: object
                    properties:
                      total: { type: integer }
    post:
      summary: Creates a new post
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreatePost'
      responses:
        '201':
          description: Post created
          headers:
            Location:
              schema: { type: string }
              description: URL of the new post
Junior Developer

How a Junior Designs APIs

  • Inconsistent URL naming — verbs, mixed case, weird abbreviations
  • No error response format — each endpoint returns different error shapes
  • Does not version the API — breaks all clients with every change
  • Exposes stack traces in error responses
  • Ignores CORS until it blocks the frontend, then tries to fix it client-side
Senior Developer

How a Senior Designs APIs

  • Consistent RESTful URL design with version prefix
  • Standard error envelope with codes, messages, and request IDs
  • Adds pagination, rate limiting, and deprecation headers from day one
  • Uses OpenAPI spec to document and validate the contract
  • Designs additive APIs that rarely need breaking changes
