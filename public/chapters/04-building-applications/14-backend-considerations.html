Chapter 14

Backend Considerations

API design, databases, auth models, and understanding when and why you need a backend

The most important backend decision is whether you need one at all. Many projects start with a backend because "that is how the web works" — and many of those projects add months of complexity for features that could have been handled by static files, a form service, or a serverless function. This chapter helps you decide, then shows you how to build what you need.

What Is a Backend?

A backend is code that runs on a server, not in the browser. Its responsibilities include:

  • Data storage and retrieval — reading and writing to a database
  • Business logic — operations that must not be visible to or tampered with by the client
  • Authentication and authorization — verifying identity and controlling access
  • API serving — delivering data to frontend clients (your SPA, mobile app, third-party integrations)
  • Background jobs — email sending, image processing, data cleanup

The key insight: a frontend can do many of these things (store data in localStorage, run business logic in JavaScript, check user roles in the client). But anything done in the browser is visible to and controllable by the user. The backend exists because there are operations you must keep out of the user's control — validating purchases, processing payments, enforcing access rules.

The Backend Is a Trust Boundary

Think of the backend as the line between what the user can control and what they cannot. If a malicious user can manipulate a value to get something for free, that value belongs on the backend. If a user can read data they should not see, that check belongs on the backend. The frontend is for user experience. The backend is for truth.

API Design Principles

An API is a contract between your backend and its consumers (frontend, mobile app, other services). Good API design makes consumers productive. Bad API design creates confusion, bugs, and versioning nightmares.

RESTful Resource Naming

REST organizes APIs around resources (nouns), not actions (verbs). Actions are expressed via HTTP methods:

RESTful endpoint patterns
GET    /api/users          # List users
POST   /api/users          # Create a user
GET    /api/users/:id      # Get a specific user
PUT    /api/users/:id      # Replace a user
PATCH  /api/users/:id      # Update fields of a user
DELETE /api/users/:id      # Delete a user

GET    /api/users/:id/posts       # List user's posts (nested resource)
POST   /api/users/:id/posts       # Create a post for a user

Notice: no verbs in URLs. No /api/getUser, no /api/createPost. The HTTP method encodes the action, the URL encodes the resource.

Status Codes

Use the right HTTP status code for every response. This lets consumers write generic error handling:

CodeMeaningWhen
200OKSuccessful GET, PUT, PATCH
201CreatedSuccessful POST (new resource created)
204No ContentSuccessful DELETE (no body needed)
400Bad RequestInvalid input, missing fields, validation failure
401UnauthorizedMissing or invalid authentication
403ForbiddenAuthenticated but not allowed (different from 401)
404Not FoundResource does not exist
409ConflictDuplicate resource, conflicting state
429Too Many RequestsRate limit exceeded
500Internal Server ErrorUnexpected server failure (generic, don't overuse)

Versioning

APIs change. Consumers must be protected from breaking changes. Two common strategies:

  • URL versioning: /api/v1/users, /api/v2/users — simple, explicit, but clutters URLs
  • Header versioning: Accept: application/vnd.myapp.v1+json — cleaner URLs, harder to test manually

For small projects, URL versioning is fine. For public APIs, plan for versioning from day one — even if you only have v1. Adding it later means breaking every consumer.

Databases: SQL vs NoSQL

The database decision shapes your entire backend. Choose wrong and you will fight your data layer for the life of the project.

SQL (PostgreSQL, MySQL)NoSQL (MongoDB, Firestore)
Data modelTables with rows and columns. Schema enforced at write time.Documents (JSON-like). Schema flexible per document.
RelationshipsForeign keys, JOINs — first-class supportEmbedding or referencing — manual management
ACIDStrong ACID guarantees by defaultVaries by product. MongoDB added multi-doc ACID in v4.0.
ScalingVertical (bigger server) first. Read replicas. Sharding is complex.Horizontal scaling designed in from the start.
Best forRelational data, complex queries, data integrity criticalRapid prototyping, hierarchical data, massive scale
Default choice: PostgreSQL. Switch to MongoDB only if you have a specific reason (schema-less data, hierarchical documents, horizontal scaling needs from day one).

The vast majority of web applications have relational data (users have posts, posts have comments, orders have line items). SQL databases model this naturally. MongoDB forces you to denormalize (duplicate) or manually resolve relationships in application code. PostgreSQL's JSONB column type even gives you document-like flexibility when needed.

Authentication Models

Authentication answers "who are you?" Authorization answers "what are you allowed to do?" Both must happen on the backend — frontend checks are purely cosmetic.

Session-Based Auth

The traditional model: user logs in, server creates a session record (in memory, database, or Redis) and sends a session cookie. The browser automatically includes the cookie on subsequent requests. The server looks up the session to identify the user.

  • Pros: simple, server can invalidate sessions instantly (logout, ban), no sensitive data in the client
  • Cons: server must maintain session state (not truly stateless), scaling requires shared session storage (Redis)

Token-Based Auth (JWT)

User logs in, server issues a signed token (JSON Web Token) containing user ID and claims. The client stores the token (usually in localStorage or an httpOnly cookie) and sends it as an Authorization: Bearer header.

  • Pros: stateless (server does not need to store sessions), works well for mobile and third-party API access
  • Cons: tokens cannot be revoked server-side (they are valid until expiry), must keep expiry short (15-60 minutes) and use refresh tokens
JWT middleware pattern (Node.js)
function authenticate(req, res, next) {
  const header = req.headers.authorization;
  if (!header || !header.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing token' });
  }

  try {
    const token = header.split(' ')[1];
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded;
    next();
  } catch (err) {
    return res.status(401).json({ error: 'Invalid or expired token' });
  }
}
Never Store JWTs in localStorage for Sensitive Apps

localStorage is accessible to any JavaScript running on your domain. An XSS vulnerability can steal all tokens. For high-security applications, store the JWT in an httpOnly cookie (inaccessible to JS) and use CSRF protection. For most applications, localStorage is acceptable — but understand the tradeoff.

Server vs Serverless

The hosting model for your backend is as important as the code itself. The choice affects cost, scalability, maintenance burden, and development speed.

Traditional ServerServerless (Functions)
ManagementYou manage OS, runtime, scaling, security patchesPlatform manages everything. You write functions.
ScalingManual or auto-scaling groups. You pay for provisioned capacity.Auto-scales to zero. You pay per invocation and duration.
Cold startsNone — server is always runningFunction instances have latency on first invocation (100ms-1s)
StateServer can hold in-memory stateFunctions are stateless — use external storage (DB, Redis)
Max executionUnlimitedTime limit (AWS Lambda: 15 minutes, Cloudflare Workers: 30s CPU)
Best forSteady traffic, complex backends, long-running processesVariable/spiky traffic, event-driven tasks, APIs, prototypes

Cloudflare Workers, Vercel Edge Functions, and Netlify Functions are excellent serverless options that integrate with static frontends. They handle form submissions, API proxying, authentication, and database access — often at zero or minimal cost for small projects.

Junior Developer

How a Junior Might Think

  • Jumps into building a backend without asking if one is needed
  • Picks MongoDB because "it is faster" without understanding data relationships
  • Implements auth from scratch with basic hashing
  • Deploys to a VPS without CI/CD, manually SSH-ing to update code
  • Writes APIs without versioning or error standardization
Senior Developer

How a Senior Thinks

  • Evaluates whether a backend is needed at all (static + form service may suffice)
  • Defaults to PostgreSQL unless there is a specific reason for NoSQL
  • Uses established auth libraries (Passport, Laravel Breeze, NextAuth)
  • Chooses serverless for variable traffic, server for steady load
  • Designs API contract first, versions from v1, uses consistent error format

Web Servers: Apache and Nginx

When choosing a web server, the two dominant options are Apache and Nginx. Understanding what each offers helps you make informed decisions based on your project needs.

Apache HTTP Server

Apache has been the workhorse of the web for decades. It's known for its flexibility through configuration files and extensive module system.

  • .htaccess: Per-directory configuration, great for shared hosting, allows password protection, redirects, and custom error pages without server access
  • mod_rewrite: Powerful URL rewriting for clean URLs, redirects, and routing logic
  • Process model: Spawns processes/threads per request; simple but can be memory-heavy under load
  • Best for: Shared hosting environments, projects needing .htaccess flexibility, PHP applications on traditional hosting

Nginx

Nginx was built as a response to the C10K problem (handling 10,000+ concurrent connections). It excels at serving static content and as a reverse proxy.

  • Event-driven: Single thread handles many connections; memory-efficient under high load
  • Reverse proxy: Excellent at forwarding requests to backend services (PHP-FPM, Node, Python)
  • No .htaccess: Configuration is central; more performant but requires server access
  • Best for: High-traffic static sites, reverse proxy setups, microservices architecture
The "Why" Behind the Choice

For most modern JavaScript applications deployed on platforms like Vercel, Netlify, or Cloudflare Pages, you won't interact with Apache or Nginx directly — the platform handles it. However, understanding these options matters when: you need traditional PHP hosting, you're working with legacy systems, or you're configuring a VPS. The principle: choose based on your specific constraints, not habit.
