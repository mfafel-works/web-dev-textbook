Chapter 16

React

Components, state, hooks, routing, build toolchain, and component-based UI architecture

React is the most popular frontend library for building interactive UIs. It is not a framework — it is a library for rendering views. You add routing, data fetching, and state management separately. This modularity is both its strength (pick what you need) and its weakness (many choices to make). This chapter gives you the mental model to use React effectively.

Why React?

Before React, frontend code manipulated the DOM directly. When data changed, you found the relevant DOM element and updated it. This works for small apps but breaks at scale — keeping the DOM in sync with application state becomes a tangle of document.getElementById and manual updates.

React introduces declarative rendering: you describe what the UI should look like for a given state, and React handles the DOM updates automatically. This shifts mental effort from "how do I update the DOM?" to "what should the UI look like for this state?"

Imperative (old) vs Declarative (React)
// Imperative: manually update DOM
const el = document.getElementById('counter');
let count = 0;
button.addEventListener('click', () => {
  count++;
  el.textContent = count;  // manual sync
});

// Declarative: describe UI for each state
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
// React syncs the DOM automatically

The virtual DOM is the mechanism: React maintains a lightweight JavaScript representation of the DOM, computes the difference (diff) between renders, and applies only the minimal DOM updates needed. You never touch the DOM directly.

Build Toolchain: Vite, Babel, and the Dev/Prod Divide

React code does not run directly in the browser. JSX is not valid JavaScript, and modern ES features need transpilation for older browsers. Every React project requires a build step — this is non-negotiable. Understanding what happens between saving a file and seeing it in the browser is essential for debugging build errors and shipping performant code.

Why a Build Step?

Three things the build step handles:

  • JSX transformation — <div>Hello</div> becomes React.createElement('div', null, 'Hello')
  • Module bundling — dozens of import/export statements across many files are merged into a single file (or a few optimized files) the browser can load
  • Optimization — production builds are minified, tree-shaken, and cache-busted for performance

Vite

Vite is the current standard build tool for React (replacing Create React App, which is deprecated). It uses two entirely different strategies for development and production — understanding this split is key.

Development: Native ES Modules

When you run npm run dev, Vite starts a dev server that serves your source files as native ES modules — no bundling at all. The browser fetches each module as a separate HTTP request. This means startup is near-instant regardless of project size, and hot module replacement (HMR) updates only the changed module without a full page reload, preserving component state.

Production: Rollup

For production (npm run build), Vite uses Rollup to bundle everything into optimized static assets: minified JavaScript, inlined or extracted CSS, and hashed filenames for cache busting. The same code served as individual modules in dev is now a lean production bundle.

Babel

Babel is a JavaScript compiler that transforms modern JS and JSX into code older browsers can run. In a Vite project, Babel is configured automatically via the @vitejs/plugin-react plugin. Its main jobs:

  • JSX → React.createElement — the core transformation that makes JSX work
  • Modern JS → compatible JS — optional chaining (?.), nullish coalescing (??), arrow functions, etc. for older browsers
  • React Refresh — preserves component state during HMR edits

The Build Pipeline

Source files (.jsx, .js, .css) ↓ Vite Dev Server (dev) / Rollup (prod) ↓ Babel transforms JSX → JS ↓ PostCSS processes CSS (autoprefixer, etc.) ↓ esbuild minifies (prod only) ↓ Browser receives flat JS/CSS files
You Do Not Need to Configure This

Vite + the React plugin is zero-config. Babel, PostCSS, and Rollup are pre-configured and invisible — you write code and run commands. But understanding what each piece does is the difference between an engineer who is blocked by a build error and one who reads the stack trace and knows exactly which tool is complaining.

dist/ Is What You Deploy

Everything in dist/ after npm run build is a static file — HTML, CSS, JS, images. There is no Node.js server, no dynamic processing. The dist/ folder contains the entire deployable artifact. Inspect it before deploying: open dist/index.html and verify it references the correct asset paths. If you use a CDN or subdirectory deployment, you may need to set Vite's base config so asset paths are correct. Deploying the wrong build (or forgetting to build) is the most common production mistake.

Environment Variables: Build-Time vs Runtime

In Vite, environment variables prefixed with VITE_ are inlined at build time. When you run npm run build, Vite reads VITE_API_URL from your shell or .env file and replaces every occurrence of import.meta.env.VITE_API_URL with its literal value in the bundled JavaScript. This means: (1) you must rebuild to change them, (2) they are visible in the client-side bundle (never put secrets in VITE_ variables), and (3) what works in npm run dev may differ from production if your .env files are set up differently. Server-side env vars (Node.js process.env) are never exposed to the client.

JSX Requires Closing Tags — </> Is Not Optional

In JSX, every tag must close: <div>...</div>, <img /> (self-closing), <Br />. Unlike HTML, you cannot write <div></div> without the slash, or leave a <img> unclosed — JSX will throw a syntax error. The self-closing slash /> is required for void elements in JSX (unlike optional in HTML). The error message will say something like "Unexpected token" pointing at where you forgot to close.

className Instead of class — class Is a Reserved Word

HTML uses class, but JavaScript already uses class for ES6 classes. JSX uses className instead: <div className="container">. The same applies to for on labels — use htmlFor. Get used to these differences: JSX maps HTML attributes to their JS-compatible names to avoid conflicts with JavaScript keywords.

State Updates Are Async — The UI Doesn't Update Immediately

In React, calling setCount(count + 1) does not update count right away. It schedules an update — the component re-renders on the next paint. If you do setCount(count + 1); console.log(count), you'll see the old value, not the new one. If you need the new value immediately, use the functional form: setCount(c => c + 1) or use useEffect with a dependency to act after the update.

useEffect Runs Twice in Development — That's Normal

In React 18+ strict mode (which is on by default in dev), useEffect runs twice to help you find bugs. If your effect fetches data twice, that's a hint — you need cleanup or a check to prevent duplicate fetches. In production (non-strict mode), it runs once as expected. Don't be alarmed — this is a feature, not a bug. Add an abort controller or check if the component is mounted to handle this properly.

Components and JSX

A component is a function that returns UI. JSX is the syntax extension that lets you write HTML-like markup inside JavaScript.

Basic React component
function Greeting({ name, role }) {
  return (
    <div className="greeting">
      <h1>Hello, {name}!</h1>
      {role && <p>You are a {role}.</p>}
    </div>
  );
}

// Usage: <Greeting name="Alice" role="admin" />

Key JSX rules:

  • Return a single root element (use <></> fragment if no wrapper needed)
  • Use className instead of class (because class is a JS keyword)
  • Close all tags — even <br /> and <img />
  • JavaScript expressions go inside {} — any valid JS expression works
  • Use curly braces for dynamic attributes: src={imageUrl}

Component Composition

Components are composable — a component can render other components. This is how you build complex UIs from small, reusable pieces:

Composing components
function Page() {
  return (
    <div>
      <Header />
      <Main>
        <Sidebar />
        <Content>
          <Article />
        </Content>
      </Main>
      <Footer />
    </div>
  );
}

State and Props

These are the two core concepts in React. Props (properties) are inputs passed to a component. State is data that a component owns and can change. The distinction drives everything.

Props Flow Down

Data flows one way: parent passes props to child. A child cannot modify its props — they are read-only. If a child needs to communicate up, the parent passes a callback function as a prop.

Props and callbacks
function Parent() {
  const [items, setItems] = useState([]);

  const addItem = (item) => {
    setItems(prev => [...prev, item]);
  };

  return (
    <>
      <ItemList items={items} />
      <AddItemForm onAdd={addItem} />
    </>
  );
}

function AddItemForm({ onAdd }) {
  const [text, setText] = useState('');

  const handleSubmit = (e) => {
    e.preventDefault();
    onAdd(text);  // calls parent's addItem
    setText('');
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={text} onChange={e => setText(e.target.value)} />
      <button type="submit">Add</button>
    </form>
  );
}

Lifting State Up

When two components need to share state, move that state to their nearest common ancestor and pass it down as props. This is called "lifting state up" — it is the primary state management pattern in React. Do not reach for a state management library (Redux, Zustand) until lifting state up becomes genuinely painful (prop drilling through 5+ levels).

Hooks Deep Dive

Hooks are functions that let you "hook into" React features from function components. Before hooks (React < 16.8), state and lifecycle were only available in class components. Hooks simplified everything.

useState

Declares a state variable. Returns an array: current value and setter function.

useEffect

Runs side effects after render. The dependency array controls when it re-runs:

useEffect patterns
// Runs once on mount (empty deps)
useEffect(() => {
  fetch('/api/data').then(r => r.json()).then(setData);
}, []);

// Runs when `userId` changes
useEffect(() => {
  fetchUser(userId).then(setUser);
}, [userId]);

// Cleanup function (runs on unmount or before re-run)
useEffect(() => {
  const subscription = api.subscribe(id, handleUpdate);
  return () => subscription.unsubscribe();  // cleanup
}, [id]);

Custom Hooks

The most powerful pattern: extract reusable logic into your own hooks. A custom hook is a function that calls other hooks:

Custom hook for data fetching
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);

    fetch(url)
      .then(r => r.json())
      .then(d => { if (!cancelled) setData(d); })
      .catch(e => { if (!cancelled) setError(e); })
      .finally(() => { if (!cancelled) setLoading(false); });

    return () => { cancelled = true; };
  }, [url]);

  return { data, loading, error };
}

// Usage in any component:
function UserProfile({ userId }) {
  const { data, loading, error } = useFetch(`/api/users/${userId}`);
  if (loading) return <Spinner />;
  if (error) return <Error message={error} />;
  return <Profile user={data} />;
}
Rules of Hooks

Two rules, non-negotiable: (1) Only call hooks at the top level — not inside loops, conditions, or nested functions. (2) Only call hooks from React function components or custom hooks. These rules ensure that hooks are called in the same order on every render, which is how React maintains state between renders.

React Router

React is a view library — it does not include routing. React Router is the de facto standard for adding client-side navigation to React apps.

React Router v6 setup
import { BrowserRouter, Routes, Route, Link } from 'react-router-dom';

function App() {
  return (
    <BrowserRouter>
      <nav>
        <Link to="/">Home</Link>
        <Link to="/about">About</Link>
        <Link to="/contact">Contact</Link>
      </nav>

      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
        <Route path="/contact" element={<Contact />} />
        <Route path="*" element={<NotFound />} />
      </Routes>
    </BrowserRouter>
  );
}

<Link> renders an anchor tag that intercepts clicks and navigates without a full page reload. <Routes> matches the current URL to a route and renders the corresponding element. The * catch-all route handles 404s.

SPA Architecture with React

React naturally lends itself to SPAs: load once, navigate without full page reloads. But SPAs come with tradeoffs that must be managed.

Performance: Code Splitting

Without code splitting, the entire app bundles into one large JavaScript file. With React.lazy and Suspense, you load components only when needed:

Code splitting with React.lazy
import { lazy, Suspense } from 'react';

const Dashboard = lazy(() => import('./Dashboard'));
const Settings = lazy(() => import('./Settings'));

function App() {
  return (
    <Suspense fallback={<Spinner />}>
      <Routes>
        <Route path="/dashboard" element={<Dashboard />} />
        <Route path="/settings" element={<Settings />} />
      </Routes>
    </Suspense>
  );
}

SEO Challenge

SPAs render content in the browser. Search engine crawlers may not execute JavaScript, leading to empty pages being indexed. Solutions:

  • Server-side rendering (SSR) — React renders on the server first, sending HTML to the browser. Next.js is the standard SSR framework for React.
  • Static site generation (SSG) — pre-render pages at build time. Best for content-driven sites (blogs, marketing pages).
  • Hybrid — SSG for static pages, client-side rendering for authenticated/dynamic pages.

For most projects, the choice is not "React vs vanilla" but "React with what rendering strategy?" Use Create React App or Vite for SPAs (client-side only). Use Next.js or Remix when you need SSR or SSG.

Junior Developer

How a Junior Might Think

  • Puts all logic in one giant component
  • Overuses useState for derived/computed data
  • Forgets cleanup functions in useEffect (memory leaks)
  • Prop drilling through 10 levels without question
  • Reaches for Redux on day one for a todo app
  • Does not understand when re-renders happen
  • Does not know what Vite or Babel do — runs commands blindly
  • Pushes dev code to production without understanding the dev/prod difference
Senior Developer

How a Senior Thinks

  • Breaks UI into small components with clear responsibilities
  • Extracts custom hooks for reusable logic
  • Always cleans up subscriptions and intervals
  • Lifts state up to the appropriate level, uses composition to avoid drilling
  • Chooses state management based on actual complexity
  • Memoizes with useMemo/useCallback only when there is a measured problem
  • Understands the build toolchain — can read a build error and know which tool failed
  • Knows the dev/prod difference and tests production builds before deploying
