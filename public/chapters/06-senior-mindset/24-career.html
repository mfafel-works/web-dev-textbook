Chapter 24

Career & Communication

Code review, commit messages, asking questions, estimating, portfolios, impostor syndrome, and using AI effectively

Technical skill gets you in the door. Communication and professional habits determine how far you go. A brilliant engineer who cannot communicate is less valuable than a solid engineer who writes clear pull requests, asks for help at the right time, and makes their teammates better. This chapter covers the non-technical skills that define a senior engineer.

Code Review

Code review is the single highest-leverage practice for code quality and team growth. It catches bugs, spreads knowledge, and establishes standards.

As the Reviewer

  • Read the description first. Understand what the change is supposed to do before reading the code.
  • Review the tests. If there are no tests, start there. Tests document expected behavior.
  • Comment on the code, not the person. Say "This condition is inverted" not "You always get this wrong."
  • Distinguish blocking vs non-blocking. Use labels: "blocking" (must fix before merge), "should fix" (important but not blocking), "nit" (style preference, author can ignore).
  • Ask questions instead of making demands. "Why did you choose a loop here instead of Array.map?" invites discussion. "Use Array.map instead" shuts it down.
  • Review in small batches. A 200-line PR gets thorough review. A 2000-line PR gets a skim. Encourage small, focused pull requests.
  • Respond within 24 hours. Blocking someone's PR for days kills momentum. If you cannot review fully, leave a quick note: "I'll review this tomorrow."

As the Author

  • Write a good description. What does this change do? Why? How was it tested? Include screenshots for UI changes.
  • Keep PRs small. A PR should do one thing. If you find yourself describing multiple changes, split the PR.
  • Respond to comments. Acknowledge every comment. Fix blocking issues. Explain why you disagree with non-blocking suggestions.
  • Do not take feedback personally. Code review is about the code, not you. A critical review of your PR is not a critical review of your ability.
The Socratic Code Review

The most effective code review technique: ask questions instead of stating conclusions. "What happens if this API returns null?" is better than "You forgot to handle null." The question makes the author think and learn. The statement just makes them fix one instance.

Writing Good Commit Messages

A commit message documents why a change was made. Six months from now, someone (probably you) will read the git log to understand why something is the way it is. The commit message is your message to that future person.

Good commit message format
# Conventional Commits format:
type(scope): short summary (max 50 chars)

Optional body with details (wrap at 72 chars).
Explain WHAT and WHY, not HOW (the diff shows HOW).

- Bullet points for multiple reasons
- Reference issue numbers: Fixes #123

Types: feat, fix, refactor, test, docs, chore, style, perf
Bad commit messageGood commit message
fix stufffix(auth): prevent infinite redirect when token expires
update buttonfeat(ui): add loading state to submit button
changesrefactor(api): extract validation logic into middleware

Follow the Conventional Commits format. It is machine-readable (enables automated changelogs, version bumps) and human-readable (scan the log and instantly know what happened).

Asking Good Questions

The ability to ask a good question is one of the most underrated engineering skills. A good question gets answered quickly. A bad question wastes everyone's time.

The Anatomy of a Good Technical Question

1. What are you trying to do? (the goal, not the attempted solution) 2. What have you tried? (shows you did your homework) 3. What happened? (full error message, stack trace, unexpected behavior) 4. What did you expect? (be specific) 5. What is the environment? (browser, Node version, OS, relevant versions)
Bad vs good question
// BAD:
"React isn't working. Help!"

// GOOD:
"I'm trying to fetch data when a component mounts.
I'm using useEffect with fetch(), but the component
renders twice and makes two API calls.

Here's the code: [link]
DevTools console shows: [screenshot]
Using React 18 with StrictMode enabled.

Expected: one API call on mount.
Actual: two API calls.
I tried removing StrictMode which fixed the double-render
but I'm told StrictMode is intentional in dev. Is this
expected behavior or am I doing something wrong?"

Where to Ask

  • Your team's chat — quick questions, context-specific
  • Stack Overflow — well-scoped questions about specific tools or errors
  • GitHub Issues/Discussions — library-specific questions, bug reports, feature requests
  • AI assistants — exploratory questions, code generation, debugging help

Reading Documentation

The most important skill in software development is not writing code — it is reading documentation. Every tool, library, and framework you use is documented. Learning to read docs efficiently saves hours of trial and error.

How to Approach a New Library

  1. Read the README. The README tells you what the library does, how to install it, and a minimal example. 90% of questions are answered in the README.
  2. Skim the API reference. You do not need to memorize everything. Know what exists so you can find it later.
  3. Look at examples. Most libraries have an examples folder or a "recipes" section. Examples show real usage patterns.
  4. Search for common patterns. " + " is often faster than reading linearly.
Read the Source When the Docs Are Unclear

Documentation can be outdated, incomplete, or wrong. The source code is always the truth. When a function does not behave as documented, open the library's source and read the function. Modern editors (VS Code) make this easy — Cmd+click on any function to jump to its definition. The ability to read source code is a superpower.

Estimating Tasks

Estimating is hard because you are predicting the future. No one is good at it. The goal is not perfect estimates — it is useful estimates that trends converge over time.

Techniques

  • Break it down. A 2-week feature cannot be estimated accurately. Twenty 2-hour tasks can. Break the work into small pieces, estimate each, sum them up.
  • Use reference points. "I built something similar last week and it took 4 hours. This is slightly more complex, so 6 hours."
  • Add uncertainty. Give a range: 4-8 hours instead of 6 hours. The range communicates uncertainty honestly.
  • Multiply by a fudge factor. If you always estimate under, multiply your gut estimate by 1.5-2x. If you always overestimate, adjust down.
  • Track actual vs estimated. After completing a task, note how long it actually took. Over time, this data calibrates your estimation sense.
The Planning Fallacy

Every human systematically underestimates how long tasks will take, especially unfamiliar ones. This is called the planning fallacy. The fix: (1) break the work into very small pieces (2-4 hours each), (2) double the total, (3) add 50% for unexpected issues. If the result seems too large, your gut is wrong — trust the math from past experience.

Building a Portfolio

A portfolio is not a collection of tutorials you followed. It is evidence of problems you solved. Quality over quantity — one well-documented project is worth more than five half-finished clones.

What Makes a Good Portfolio Project

  • Real problem, real users. A tool your friends actually use. A website for a local business. An open-source contribution.
  • Technical depth. Shows you can handle state management, API integration, authentication, responsive design, and deployment.
  • Good README. What does it do? How do I run it? What decisions did you make? What would you do differently?
  • Live demo. A link to a working version. Deploy it. Free hosting (Cloudflare Pages, Netlify, Vercel) removes all excuses.
  • Clean code. Your GitHub repo is your resume. Use consistent formatting, meaningful names, and clear commit messages.
Contribute to Open Source

Open-source contributions are the most credible portfolio items. They show you can work with existing codebases, follow someone else's conventions, and collaborate asynchronously. Start with documentation fixes and small bug fixes. Every large project has a "good first issue" label.

Impostor Syndrome

Impostor syndrome — the feeling that you are a fraud and will be found out — is nearly universal in software development. It affects junior and senior engineers alike. The difference is not whether you feel it, but how you respond.

  • It never fully goes away. Every time you level up (new job, new language, new domain), the feeling returns. This is normal.
  • Compare yourself to your past self, not to others. You know more than you did six months ago. That is the only comparison that matters.
  • Your job is not to know everything. Your job is to figure things out. "I don't know, but I can find out" is the most powerful phrase in engineering.
  • Share your ignorance. When a senior engineer admits "I don't know how to do that yet," it creates psychological safety for everyone.

Using AI Tools Effectively

AI coding assistants (GitHub Copilot, Claude, ChatGPT) are now a standard part of the engineering toolkit. Used well, they multiply productivity. Used poorly, they generate code you do not understand and cannot maintain.

When to Use AI

  • Boilerplate generation — repetitive patterns, CRUD endpoints, form components
  • Exploration and learning — "Explain how this algorithm works" or "Show me examples of this pattern"
  • Debugging assistance — paste the error and relevant code, get hypotheses
  • Refactoring — "Convert this callback chain to async/await"
  • Writing tests — generate test cases for edge conditions you might miss

When NOT to Use AI

  • Code you do not understand. If you cannot explain what the generated code does, do not commit it.
  • Security-critical logic. Authentication, authorization, encryption, input sanitization — these require human review.
  • When you are learning fundamentals. Writing code yourself builds mental models. Using AI to skip the struggle skips the learning.
  • Without verification. AI generates plausible-looking code that may be subtly wrong. Test it. Review it. Do not trust it.
AI Is an Intern, Not a Senior Developer

AI generates code that looks correct but may be wrong, insecure, or inefficient. Treat it like a smart intern: review everything it produces, understand why it made those choices, and do not ship code you cannot explain. The goal is to augment your skills, not replace your judgment.

Junior Developer

How a Junior Might Think

  • Writes vague commit messages ("fix", "update", "changes")
  • Asks vague questions — does not include error messages or context
  • Does not read documentation — guesses until something works
  • Says "I don't know" with shame, then stays stuck
  • Does not estimate or consistently underestimates
  • Feels intense impostor syndrome and does not talk about it
  • Uses AI to generate code they do not fully understand
Senior Developer

How a Senior Thinks

  • Writes clear commit messages that explain the why
  • Asks precise questions with context, error messages, and reproduction steps
  • Reads documentation first, then source code when docs are unclear
  • Says "I don't know, but I'll find out" with confidence
  • Breaks work into small pieces and estimates based on past data
  • Acknowledges impostor feelings and uses them as motivation to learn
  • Uses AI for boilerplate and exploration, but reviews and understands every line
