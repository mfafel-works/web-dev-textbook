Chapter 1

How the Web Works

Browsers, servers, HTTP, DNS, and what happens when you type a URL and press Enter

What Is a Browser?

A browser is the program you use to visit websites — like Chrome, Safari, Firefox, or Edge. It's the app on your computer that fetches web pages from servers and displays them. That's it! When we say "open in your browser," we mean "open with Chrome/Safari/etc."

Client vs Server — Simply Explained

Think of it like a restaurant:

  • Client = You (the customer). You look at a menu, order food, and eat it.
  • Server = The kitchen. It receives your order, prepares the food, and sends it back to your table.

In web terms: your browser is the client. It asks the server for something (a web page), and the server sends it back. The client is on your computer. The server is on another computer somewhere else.

What Is a Server?

A "server" is just a computer that's always on and always connected to the internet, running special software that waits for requests and responds with data. That's it — it's not magic, it's just a computer. When you deploy a website, you're putting your files on one of these computers.

What Is a URL?

A URL (Uniform Resource Locator) is an address for something on the internet. Here's what the parts mean:

https://example.com:8080/path/to/page
  • https:// — The protocol (how to communicate). https means secure, encrypted.
  • example.com — The domain name (the address of the server). This maps to an IP address via DNS.
  • :8080 — The port (which "door" on the server). If omitted, defaults to 80 for HTTP, 443 for HTTPS.
  • /path/to/page — The path (what specific resource you want on that server).
What Is localhost?

localhost means "this computer" — it's a special word that always refers to your own machine. When you run a local server and visit http://localhost:8080, you're connecting to a server running on your own computer, not reaching out to the internet. 127.0.0.1 is the numeric IP address for localhost — they mean the same thing.

What Is a Port?

A port is like an apartment number. A single computer can run many "services" (websites, databases, email servers). Each one has a unique port number so requests go to the right place. When you see localhost:8080, the :8080 is the port number telling your computer "use the service running on door 8080." Common ports: 80 (HTTP), 443 (HTTPS), 3000 (dev servers), 5173 (Vite), 8080 (Python server).

The Client-Server Model

The web is built on a simple relationship: one computer asks for something, another computer provides it. The asker is called the client (usually a browser). The provider is called the server (a computer somewhere on the internet running web server software like Nginx or Apache).

When you open a browser and navigate to a website, your browser acts as the client. It sends a request across the network to a server. The server processes that request and sends back a response — typically an HTML page, an image, JSON data, or some other resource. This back-and-forth is the fundamental unit of the web.

Client-server flow (simplified)
Your Browser (Client)
       ↓  GET /index.html
       ↓
  [Internet]
       ↓
Web Server (Server)
       ↓  sends index.html content
       ↓
Your Browser (Client)
       ↓  renders the page

A single server can serve thousands or millions of clients. To manage this, servers use ports (port 80 for HTTP, port 443 for HTTPS), virtual hosts to serve multiple domains from one machine, and load balancers to distribute traffic across many servers.

HTTP and HTTPS

HTTP (Hypertext Transfer Protocol) is the language that clients and servers use to communicate. It is a text-based protocol — you can read HTTP messages directly. Every HTTP message has a method, a path, headers, and optionally a body.

An HTTP request looks like this
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
An HTTP response looks like this
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1234

<!DOCTYPE html>
<html>...</html>

HTTP Methods

There are four common methods you will use constantly:

MethodPurposeExample
GETRetrieve dataLoading a page, fetching JSON
POSTSend new dataSubmitting a form, creating a record
PUTReplace existing dataUpdating a user profile
DELETERemove dataDeleting a post

HTTPS — The Secure Version

HTTPS is HTTP encrypted with TLS (Transport Layer Security). The encryption ensures three things: privacy (no one can read the data in transit), integrity (no one can modify the data), and authentication (you are talking to the real server, not an impersonator). Modern browsers mark HTTP sites as "Not Secure" and may block features like geolocation and service workers. There is no excuse for shipping an HTTP site in 2026.

Why HTTPS Matters

Beyond security, some browser APIs (like Service Workers and the Geolocation API) require HTTPS. Search engines rank HTTPS sites higher. And if you ever need to accept payments or passwords, HTTPS is legally required (PCI-DSS).

DNS Resolution

Computers communicate using IP addresses — numbers like 93.184.216.34. Humans use domain names like example.com. The Domain Name System (DNS) is the phonebook that translates one to the other.

When you type example.com into your browser, the following happens in milliseconds:

  1. Browser cache check — your browser checks if it already knows the IP for this domain
  2. OS cache check — if not found, your operating system's DNS cache is checked
  3. Recursive resolver — your request goes to your ISP's DNS server (or Cloudflare's 1.1.1.1, Google's 8.8.8.8)
  4. Root server — the resolver asks one of 13 root servers where to find the .com TLD server
  5. TLD server — the .com server points to the authoritative nameserver for example.com
  6. Authoritative nameserver — this returns the actual IP address (93.184.216.34)
  7. Response — the IP is passed back up the chain and cached at each level

DNS results are cached at every level (your browser, your OS, your router, your ISP) to speed up subsequent lookups. This is why DNS changes can take hours to propagate — old cached entries must expire first. The time-to-live (TTL) value on a DNS record controls how long it is cached.

The Request-Response Cycle

Now let's trace the complete journey from the moment you press Enter to the moment the page renders:

  1. URL parsing — browser identifies the protocol (https), domain (example.com), path (/page), and query parameters
  2. DNS lookup — browser resolves the domain to an IP address (see DNS section above)
  3. TCP connection — browser opens a TCP connection to the server on port 443 (HTTPS). This is a three-way handshake (SYN, SYN-ACK, ACK)
  4. TLS handshake — if HTTPS, the browser and server negotiate encryption keys and verify the server's certificate
  5. HTTP request — browser sends an HTTP GET request for /page
  6. Server processing — the server receives the request, processes it (running backend code, querying a database, etc.), and prepares a response
  7. HTTP response — server sends back HTML, CSS, JavaScript, images, or JSON with a status code
  8. Browser rendering — browser parses the HTML, fetches additional resources (CSS, JS, images), constructs the DOM, applies styles, executes scripts, and paints the page
Why This Matters

Every performance optimization you will ever learn targets one of these steps. CDNs cache at step 7. Minification reduces step 8. Keep-alive connections reuse step 3. Knowing this cycle gives you a mental model for debugging: "Is the problem at DNS? The server? The browser rendering?"

Status Codes and Headers

HTTP status codes are three-digit numbers that tell you what happened with your request. They are grouped into five categories:

CodeMeaningWhen You'll See It
200OKRequest succeeded. Page loaded normally.
301Moved PermanentlyThe URL has changed permanently. Browser redirects automatically.
302Found (temporary redirect)URL temporarily moved. Used for things like post-login redirects.
304Not ModifiedBrowser cache is fresh — use the cached version (saves bandwidth).
400Bad RequestServer couldn't understand the request (malformed syntax).
401UnauthorizedYou need to log in first.
403ForbiddenYou're logged in but don't have permission.
404Not FoundThe resource doesn't exist. Classic "page not found".
429Too Many RequestsYou hit a rate limit — slow down.
500Internal Server ErrorSomething broke on the server. Not your problem, but you need to fix it.
503Service UnavailableServer is overloaded or down for maintenance.

Headers are key-value pairs sent with every HTTP request and response. They control caching (Cache-Control), content type (Content-Type), authentication (Authorization), security (Content-Security-Policy), and much more. You rarely need to set them manually in early projects, but you will see them constantly in browser DevTools.

Why Caching Matters

Browsers cache responses to avoid re-downloading the same files. The server controls caching via the Cache-Control header. A cached page loads nearly instantly on repeat visits. Understanding caching is essential — it is the single biggest performance lever available to every developer, and misconfiguring it causes the most common "it worked before" bugs.

Gotcha: SPA Routing and 404 on Refresh

SPA Routing: 404 on Refresh

Single-page applications use client-side routing. When a user refreshes at yoursite.com/about, the browser sends a real HTTP request for /about. If the server has no file at that path, it returns a 404. Fix: configure your server to serve index.html for all routes. On Cloudflare Pages, this is "SPA mode" — a one-click setting. The alternative is hash routing (yoursite.com/#/about), which needs no server config but produces uglier URLs.

Nginx SPA fallback config
location / {
  try_files $uri $uri/ /index.html;  # try real file, then fall back to index.html
}

If you forget this, every deep link to your SPA will break on refresh. This is the most common deployment bug for first-time SPA developers.

Junior Developer

How a Junior Might Think

  • Knows that typing a URL "just works" but can't explain the steps
  • May confuse HTTP methods or use them incorrectly
  • Doesn't check the Network tab when debugging
  • Thinks DNS is just "some setting in cPanel"
  • Does not understand status codes beyond 200 and 404
Senior Developer

How a Senior Thinks

  • Can trace the full request cycle from URL entry to page render
  • Understands when to use each HTTP method and why
  • Regularly uses the Network tab to diagnose performance and errors
  • Configures DNS records directly and understands propagation
  • Uses status codes and headers to debug server issues without guessing
