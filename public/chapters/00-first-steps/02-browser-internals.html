Chapter 2

How the Browser Works

Rendering pipeline, JavaScript engine, event loop, and the runtime environment

The browser is the most important piece of software in a web developer's life — and the least understood. When you write code, you are writing instructions for the browser to execute. Understanding how the browser reads, interprets, and displays your work is the difference between guessing and knowing. This chapter pulls back the curtain.

The Browser's Job

A browser has four main responsibilities:

  1. Networking — fetch resources (HTML, CSS, JS, images) over HTTP
  2. Parsing — convert raw bytes into structured data the browser can work with
  3. Rendering — turn structured data into pixels on screen
  4. Execution — run JavaScript, handle events, manage state

Every browser (Chrome, Firefox, Safari, Edge) does these four things. They do them differently under the hood, but the concepts are the same. Once you understand the concepts, browser-specific quirks become predictable.

The Rendering Pipeline

When the browser receives an HTML page, it does not display it all at once. It goes through a sequence of steps, each building on the last:

Step 1: HTML Parsing → DOM Tree

The browser reads raw HTML bytes and converts them into tokens (tags, attributes, text), then builds a tree structure called the DOM (Document Object Model). Each HTML element becomes a node in the tree. The DOM is an in-memory representation of the page structure that JavaScript can access and modify.

HTML → DOM tree mentally
<html>
  <body>
    <h1>Hello</h1>
    <p>World</p>
  </body>
</html>

// Becomes:
document
 └── html
      └── body
           ├── h1 ("Hello")
           └── p  ("World")

Step 2: CSS Parsing → CSSOM Tree

CSS bytes become the CSSOM (CSS Object Model) — a tree of styles that mirrors the DOM. The browser calculates which styles apply to which elements, resolving specificity, inheritance, and cascade. This is why the cascade (the C in CSS) is named that — styles cascade down the tree.

Step 3: Render Tree

The DOM and CSSOM are combined into the render tree. This contains only the elements that will actually be displayed — display: none elements are excluded. visibility: hidden elements are included (they take up space but are invisible).

Step 4: Layout

The browser calculates the exact position and size of every element in the render tree. This is where the box model comes to life — margins, padding, borders, and content areas are all computed. Layout is the most expensive step in terms of performance. Changing a element's width or height triggers a reflow, which recalculates positions for potentially all elements on the page.

Step 5: Paint

The browser fills in pixels: text colors, background images, borders, shadows. Modern browsers break the page into layers (like Photoshop layers) and paint each one separately. This is why transform and opacity animations are cheap — they only affect the composite step, not layout or paint.

Step 6: Compositing

The painted layers are assembled into the final image you see on screen. Only compositing-related properties (transform, opacity) can be animated without triggering layout or paint.

Layout Thrashing — The Hidden Performance Killer

Every time you read a layout property like offsetHeight or getBoundingClientRect() after changing a style, the browser must force a synchronous layout to give you the correct value. This is called layout thrashing. Batch your reads and writes: read all measurements first, then make all changes. Or use requestAnimationFrame to coordinate.

The JavaScript Engine

JavaScript code does not run directly on your CPU — it runs in an engine embedded in the browser. Each browser has its own engine:

How an Engine Executes Code

The simplified flow:

Source Code (.js file) ↓ Parser → builds an Abstract Syntax Tree (AST) ↓ Interpreter → executes the AST (slow but immediate) ↓ Compiler (JIT) → observes hot code paths, compiles to optimized machine code ↓ Optimized machine code replaces interpreted code at runtime

Modern engines use Just-In-Time (JIT) compilation. They start interpreting code immediately so there is no startup delay, then observe which code runs most frequently (hot paths) and compile those into highly optimized machine code. This is why JavaScript can be both fast to start and fast to run.

The Cost of Dynamic Types

JavaScript is dynamically typed — a variable can hold a string, then a number, then an object. The engine must check the type every time it accesses a variable. If you change a variable's type inside a hot loop, the engine deoptimizes that code path and falls back to the interpreter. This is why TypeScript and consistent types actually make your code faster — the engine can optimize more aggressively when types don't change.

The Event Loop

JavaScript runs on a single thread — it can do only one thing at a time. But the browser is multi-threaded. The event loop is the mechanism that coordinates between JavaScript, the browser's rendering engine, and asynchronous operations (timers, network requests, user input).

The Three Queues

Call Stack (one thing at a time — where your code runs) ↓ Microtask Queue (Promise resolutions, MutationObserver — higher priority) ↓ Task Queue (setTimeout, setInterval, events, fetch callbacks)

Call Stack

When a function calls another function, the stack grows. When a function returns, the stack shrinks. If the stack overflows (too many nested calls, or infinite recursion), you get a "Maximum call stack size exceeded" error. The stack is where synchronous code executes.

Microtask Queue

Promise resolutions (.then(), .catch(), .finally()) and MutationObserver callbacks go here. After every stack frame, the browser empties the ENTIRE microtask queue before moving on. This is why promise callbacks run before timer callbacks — they are higher priority.

Task Queue

setTimeout, setInterval, user events (click, keydown), and fetch response handling go here. After emptying microtasks, the browser takes ONE task from this queue, executes it, and checks microtasks again. This is why setTimeout(0) does not run immediately — it still has to wait for the stack and microtask queue to clear.

Event loop execution order — predict the output
console.log('1');  // sync

setTimeout(() => {
  console.log('2');  // task queue
}, 0);

Promise.resolve().then(() => {
  console.log('3');  // microtask queue
});

console.log('4');  // sync

// Output: 1, 4, 3, 2
// Sync runs first, then microtasks, then tasks.
Blocking the Event Loop

A long-running synchronous operation (an infinite loop, a massive calculation, a synchronous network request) blocks the entire event loop. The page freezes — no clicks register, no animations play, no scroll events fire. This is why you never use synchronous XMLHttpRequest and why heavy computations should be offloaded to Web Workers or broken into chunks with setTimeout.

Storage APIs

Browsers provide several ways to store data on the client side. Each has different characteristics for capacity, persistence, and scope:

APICapacityPersistenceScopeUse Case
localStorage~5-10 MBUntil manually clearedPer originUser preferences, theme choice, non-sensitive cached data
sessionStorage~5-10 MBUntil tab closesPer tabForm data in progress, temporary state
cookies~4 KB per cookieControlled by Expires / Max-AgeSent with every request to that domainSession tokens, authentication, tracking (avoid for storage)
IndexedDBUsually unlimited (browser-dependent)Until manually clearedPer originLarge structured data, offline apps, file storage
Cache APIUnlimited (disk space permitting)Managed by Service WorkerPer originHTTP response caching, offline-first PWAs

localStorage vs sessionStorage

Storage basics
// localStorage — survives browser restarts
localStorage.setItem('theme', 'dark');
const theme = localStorage.getItem('theme');
localStorage.removeItem('theme');
localStorage.clear();

// sessionStorage — survives page refreshes, NOT tab close
sessionStorage.setItem('draft', 'unsaved content');
const draft = sessionStorage.getItem('draft');

When to Use What

  • User preferences (theme, language, layout) → localStorage
  • Temporary form state → sessionStorage
  • Authentication tokens → cookies (with HttpOnly and Secure flags — set by the server, not client JS)
  • Large datasets (product catalog, messages) → IndexedDB
  • Offline page caching → Cache API via Service Worker
localStorage Is Synchronous — It Blocks the Main Thread

Reading from or writing to localStorage blocks the event loop. For small values (a theme setting), this is negligible. For large values, use IndexedDB which is async and does not block rendering. Never store sensitive data (passwords, tokens) in localStorage — it is accessible to any JavaScript running on the same origin, including third-party scripts.

Feature Detection, Not Browser Detection

Never check which browser the user is running. Instead, check whether the API you need exists:

Feature detection pattern
// ❌ Bad — browser detection
if (navigator.userAgent.includes('Chrome')) {
  useGeolocation();
}

// ✅ Good — feature detection
if ('geolocation' in navigator) {
  navigator.geolocation.getCurrentPosition(success, error);
} else {
  fallbackToManualLocation();
}

Browser detection breaks constantly — user agents can be spoofed, new browser versions change behavior, and feature availability varies even within the same browser. Feature detection is future-proof.

Browser Compatibility

Not all browsers support all features at the same time. Every time you use a modern API, you need to check whether the browsers your users are on support it.

How to Check Compatibility

  • Can I Use (caniuse.com) — the standard reference. Search any feature and see support tables across all major browsers and versions.
  • MDN Browser Compatibility Tables — every MDN page has a browser compatibility section at the bottom showing exactly which versions support the feature.
  • Babel / Core.js — for JavaScript features, Babel can transpile modern syntax to older equivalents. For APIs (like fetch), you need polyfills.

Polyfills vs Transpilation

  • Transpilation — converts new syntax to old syntax. Babel turns const into var, arrow functions into regular functions, optional chaining into ternary checks. This happens at build time.
  • Polyfill — adds missing functionality at runtime. If the browser lacks Array.prototype.includes, a polyfill adds it. Tools like core-js and polyfill.io provide these.

For modern development, use Babel for syntax and specify browser targets in your package.json or .browserslistrc file. Tools like Vite and Next.js handle this automatically.

Junior Developer

How a Junior Might Think

  • Treats the browser as a black box — "it just works"
  • Does not understand why animations sometimes stutter (layout thrashing)
  • Uses localStorage for everything, including sensitive data
  • Does not know the difference between microtasks and tasks
  • Checks browser version instead of feature detection
Senior Developer

How a Senior Thinks

  • Understands the rendering pipeline and knows which properties trigger layout vs paint vs composite
  • Chooses the right storage API based on data size, persistence needs, and security
  • Understands the event loop and can predict async execution order
  • Uses feature detection for cross-browser compatibility
  • Knows that blocking the event loop freezes the page — keeps heavy work async
