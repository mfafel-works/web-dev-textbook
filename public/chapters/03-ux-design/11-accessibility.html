Accessibility

WCAG POUR, screen readers, focus management, ARIA, keyboard navigation, and legal requirements

Accessibility (a11y) is not a feature. It is a property of your code — a page is either accessible or it is not. Approximately 15% of the global population has some form of disability. If your site is not accessible, you are excluding one in seven people. Beyond the ethical argument: inaccessible sites face lawsuits (in many jurisdictions), rank lower in search engines, and provide a worse experience for all users (captions help people in noisy environments, good contrast helps people in bright sunlight, keyboard navigation helps power users).

WCAG POUR Principles

The Web Content Accessibility Guidelines define four principles. Every accessibility requirement falls under one of these:

POUR Principles: ├── Perceivable — Users must be able to perceive the content │ ├── Text alternatives for non-text content (img alt, video captions) │ ├── Captions and alternatives for multimedia │ └── Content adaptable without losing meaning ├── Operable — Users must be able to operate the interface │ ├── Keyboard-only operation │ ├── Enough time to read and use content │ ├── No seizure-inducing flashing (more than 3 flashes per second) │ └── Easy navigation (skip links, headings, clear focus) ├── Understandable — Users must be able to understand the content │ ├── Readable language │ ├── Predictable behavior (consistent navigation) │ └── Input assistance (clear labels, helpful errors) └── Robust — Content must work with current and future tools ├── Valid HTML ├── Proper ARIA usage when needed └── Works with assistive technologies (screen readers, switch devices)
WCAG Compliance Levels

WCAG has three levels: A (minimum), AA (standard — most legal requirements target this), and AAA (highest). AA compliance is the practical target for most web projects. It requires: 4.5:1 color contrast for text, captions for prerecorded video, keyboard navigation for all functionality, and clear labels on all inputs. AAA is rarely achievable site-wide (e.g., 7:1 contrast for all text is very restrictive).

Keyboard Navigation

Keyboard navigation is the foundation of accessibility for motor disability. Many users cannot use a mouse — they rely on Tab, arrow keys, Enter, and Space to navigate.

Tab Order

The natural tab order follows DOM order. Interactive elements (links, buttons, form controls) are focusable by default. Use tabindex carefully:

  • tabindex="0" — makes an element focusable in natural DOM order (use for custom interactive components)
  • tabindex="-1" — makes an element programmatically focusable but not reachable by Tab (use for off-screen panels that should receive focus when opened)
  • tabindex="5" (or any positive value) — never use. Positive tabindex overrides natural order and creates confusing tab flows.
Managing focus for a modal
function openModal(modal) {
  modal.style.display = 'block';
  // Trap focus inside the modal
  modal.querySelector('[data-autofocus]').focus();
  // Prevent tabbing to elements behind the modal
  document.body.style.overflow = 'hidden';
}

function closeModal(modal) {
  modal.style.display = 'none';
  // Return focus to the element that opened the modal
  document.activeElement.blur();
  lastFocusedElement.focus();
}

Focus Trapping

When a modal, dialog, or flyout menu opens, focus must be trapped inside it — tabbing past the last element should cycle back to the first. This is called focus trapping or roving tabindex. Libraries like focus-trap handle this, or you can implement it manually by intercepting Tab key events and cycling focus within the container.

Never Remove Focus Indicators

outline: none or outline: 0 without a replacement makes your site unusable for keyboard users. If you remove the default focus ring, you must provide an alternative style — a visible box-shadow, a different background color, or a thicker border. The focus indicator must have a contrast ratio of at least 3:1 against the unfocused state.

Screen Reader Testing

You cannot test accessibility by looking at the screen. You must test with a screen reader.

What to Listen For

  • Does the page announce its structure (banner, navigation, main, content info landmarks)?
  • Are images announced with useful alt text or skipped with alt=""?
  • Can you navigate by headings (h1 → h2 → h3) and does the hierarchy make sense?
  • Are dynamic content updates announced (aria-live regions)?
  • Does the screen reader announce form field labels, required states, and errors?
Quick Screen Reader Test

With VoiceOver on Mac: turn it on (Cmd+F5), then use Ctrl+Option+Left/Right to navigate by element and Ctrl+Option+U to open the Rotor for quick navigation by headings or links. If you can navigate your entire page using only these keys and understand the content, your basic structure is accessible. Most issues become obvious within 30 seconds of screen reader testing.

ARIA and Live Regions

ARIA (Accessible Rich Internet Applications) attributes supplement HTML semantics. The first rule of ARIA: do not use it if a native HTML element works.

ARIA when needed, not by default
<!-- Good: native button, zero ARIA needed -->
<button>Submit</button>

<!-- Bad: div pretending to be a button, lots of ARIA -->
<div role="button" tabindex="0" aria-pressed="false"
     onclick="submit()" onkeydown="if(event.key==='Enter')submit()">
  Submit
</div>

aria-live Regions

aria-live tells screen readers to announce dynamic content changes without the user needing to focus the element:

  • aria-live="polite" — announce when the user is idle (use for status updates, search results)
  • aria-live="assertive" — announce immediately (use for critical errors, but sparingly — it interrupts the user)
Live region for form validation
<div aria-live="polite" aria-atomic="true" role="alert">
  <!-- error messages injected here by JavaScript -->
  <p>Please enter a valid email address.</p>
</div>

Accessible Forms

Forms are the most common point of failure for accessibility. A form that cannot be filled out by a screen reader user is a form that excludes people.

Label Every Input

Every <input>, <select>, and <textarea> must have a <label>. The for attribute connects the label to the input's id:

Properly labeled form field
<label for="email">Email address</label>
<input type="email" id="email" name="email" required>

Group Related Fields

Use <fieldset> and <legend> to group related inputs (radio buttons, checkboxes, address fields):

Grouped radio buttons
<fieldset>
  <legend>Shipping method</legend>
  <label><input type="radio" name="shipping" value="standard"> Standard (5-7 days)</label>
  <label><input type="radio" name="shipping" value="express"> Express (2-3 days)</label>
</fieldset>

Error Association

When validation fails, associate the error message with the input using aria-describedby:

Accessible error association
<label for="email">Email</label>
<input type="email" id="email" name="email"
       aria-describedby="email-error" aria-invalid="true">
<p id="email-error" role="alert">Please enter a valid email address.</p>

Color and Contrast

Color contrast is the most common accessibility failure on the web. WCAG AA requires:

  • 4.5:1 contrast ratio for normal text (below 18px or not bold)
  • 3:1 for large text (18px+ bold or 24px+ regular)
  • 3:1 for UI components and graphical objects (icons, chart lines)
Safe and unsafe color combinations
/* GOOD: #333 on #fff — 10.7:1 contrast ratio */
body { color: #333; background: #fff; }

/* BAD: #999 on #fff — 2.8:1 contrast ratio (fails AA) */
body { color: #999; background: #fff; }

/* GOOD: #fff on #2563eb — 5.8:1 contrast ratio */
.button { color: #fff; background: #2563eb; }

/* BAD: #fff on #a0c4ff — 2.6:1 contrast ratio (fails AA) */
.button { color: #fff; background: #a0c4ff; }
Never Rely on Color Alone

Approximately 8% of men have red-green color blindness. Never convey information using only color: links should be underlined (not just colored), required fields should have an asterisk (not just a red label), charts should use patterns (not just colored bars), and error states should include text (not just a red border).

Automated Testing Finds ~50% of Issues

Tools like axe and Lighthouse catch the low-hanging fruit: missing alt text, insufficient contrast, missing labels. They cannot catch: whether alt text is actually meaningful, whether a screen reader user can complete a workflow, or whether focus management is logical. Automated testing is a necessary first pass — manual screen reader testing is what catches the rest.

Legal Landscape

Accessibility is increasingly a legal requirement, not just a best practice:

  • ADA (US) — Title III requires places of public accommodation to be accessible. Courts have consistently ruled that websites are covered.
  • Section 508 (US) — Government websites and federally-funded projects must meet WCAG AA.
  • EAA / EN 301 549 (EU) — European Accessibility Act requires public sector websites and apps to meet WCAG AA.
  • AODA (Ontario, Canada) — Requires WCAG AA compliance for public organizations.
  • Equal Act (proposed US) — If passed, would explicitly codify website accessibility under the ADA.

The number of website accessibility lawsuits in the US has been increasing year over year (over 4,000 in 2024). In many cases, the plaintiff's law firm sends a demand letter first — fixing the issues before litigation is almost always cheaper than fighting the lawsuit. The best defense: maintain WCAG AA compliance as a standard practice, not a reaction to a lawsuit.

Junior Developer

How a Junior Might Think

  • Thinks accessibility is "just about alt text"
  • Uses outline: none without a replacement
  • Only tests with automated tools and considers it done
  • Does not know how to open or use a screen reader
  • Adds ARIA attributes without understanding them
  • Uses color alone to convey state (red for error, green for success)
Senior Developer

How a Senior Thinks

  • Tests with a real screen reader as part of every feature's definition of done
  • Provides custom focus styles instead of removing the default ring
  • Knows that automated tools only catch ~50% of issues — manual testing is required
  • Uses native HTML elements before reaching for ARIA
  • Designs with color-blind users in mind (patterns, text labels, underlines)
  • Monitors the legal landscape and maintains WCAG AA as standard practice
