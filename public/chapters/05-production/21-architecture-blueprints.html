Chapter 21

Architecture Blueprints

The 4-layer model, architecture patterns, and decision trees for choosing the right system design

Architecture is about making decisions before you write code. The right architecture makes your system easy to change, test, and scale. The wrong one makes every feature feel like fighting against your own code. This chapter gives you the mental models to choose well.

The 4-Layer Model

Every web application — no matter how simple or complex — can be understood as four layers:

User Interface → Application Logic → Data Layer → Infrastructure Layer
  • UI Layer — what the user interacts with: HTML, CSS, JS, React components, static pages
  • Application Logic — routing, state management, business rules, workflows, API endpoints
  • Data Layer — where content and state live: CMS, databases, JSON config files, APIs
  • Infrastructure Layer — hosting, CDN, security, email, DNS, server config

The critical rule: each layer only communicates with the layer directly below it. The UI never touches the database. The application logic never generates HTML directly. Infrastructure never contains business logic. This is the foundation of every good system design.

What changes as you scale is not the model — it's how many layers you introduce. A static site might collapse all four layers into a single HTML file. A large application might split each layer into multiple services. But the mental model stays the same.

Project Type Decision Tree

Before choosing a stack, define what you're building:

Content-driven site

Portfolio, marketing site, documentation, blog. Best architecture: Static site or SPA + config JSON, optionally with a headless CMS.

Interactive app

Dashboards, user accounts, workflows, CRUD systems. Best architecture: SPA + backend API with database and auth system.

Hybrid system

CMS + app features, admin panel + public UI, content + interactions. Best architecture: Headless CMS + SPA + API layer.

Architecture Blueprints

Blueprint 1: Lightweight SPA (Config-Driven)

index.html + JS + config.json → dynamic UI renderer

All content lives in a single JSON file. The frontend fetches it at runtime and renders pages dynamically. When to use: small sites, fast prototyping, no backend needed. This is the architecture built in Chapter 09.

Blueprint 2: Headless CMS Architecture

CMS → API → SPA frontend → UI render layer

Content is managed in a CMS (Strapi, WordPress, Sanity) and accessed via API. The frontend is decoupled — it just fetches content and renders it. When to use: when non-developers need to edit content, client-managed sites, scaling content complexity.

Blueprint 3: Full Stack App

Frontend SPA → Backend API → Database → Auth system

The full stack. Backend handles business logic, authentication, database operations. Frontend handles presentation. When to use: user accounts, permissions, dashboards, transactions, any application logic that can't live in the browser.

Data Flow Patterns

Static flow (no backend)

config.json → fetch → render UI

Simplest system. No persistence, no auth. Content is pre-defined or loaded from a static file. This is the right choice for the majority of websites.

API-driven flow

UI → API request → backend → DB → response → UI update

Dynamic data. Requires server logic. Scalable. This is the standard pattern for any application with user-generated content or real-time data.

Event-driven flow (advanced)

User action → event → handler → state update → UI re-render

React-like systems. Highly modular. The UI doesn't know about other components — it just emits events and listens for state changes. This scales to complex applications without tight coupling.

Backend Decision Map

The question "do I need a backend?" is one of the most important decisions you'll make. Here's how to decide:

NO backend if:

  • Content is static (pre-built or loaded from a JSON file)
  • No user accounts or authentication
  • No data needs to be saved between sessions
  • Forms can use a third-party service (Resend, Formspree, MailerLite)

YES backend if:

  • Forms need custom processing or data storage
  • Users log in and have permissions
  • Content changes dynamically based on user state
  • You need to process payments, send emails, or run scheduled tasks

Lightweight backend options

  • Email/form handling: Resend, MailerLite, Formspree — no server needed
  • Serverless functions: Cloudflare Workers, Vercel Edge Functions, Lambda — API routes only, no full server management
  • BaaS (Backend as a Service): Supabase, Firebase — database, auth, and storage without managing servers

The trend in modern web development is toward no-backend or minimal-backend architectures. Add a backend only when you have a specific need that can't be solved by a third-party service or edge function.

CMS Architecture Pattern (Core Insight)

Most content systems reduce to this pattern:

Content Source → Schema → Renderer → UI

The key abstraction rule: the CMS defines structure, not appearance. The CMS specifies what content exists (title, body, hero image) but not how it looks (font size, color, layout). That is the frontend's job.

CMS data — structure only
{
  "type": "hero",
  "data": {
    "title": "Hello",
    "subtitle": "World"
  }
}

The frontend receives this structure and decides the presentation. This separation means you can redesign your entire UI without touching the CMS, and you can swap CMS providers without touching the frontend. This is not a technical constraint — it's a design principle that prevents expensive coupling.

SPA Architecture Pattern

Router → State Manager → Renderer → DOM

A Single Page Application has three responsibilities:

  • Router: URL → page mapping. Determines what to render based on the current URL.
  • State Manager: current app memory. Holds the data that the UI depends on.
  • Renderer: converts state into DOM elements. Takes data and produces HTML.

Notice this matches the architecture built in Chapter 09. Whether you use React or vanilla JS, the mental model is the same: route input, state management, DOM output.

Hybrid Combinations (Most Real-World Projects)

Most projects are not pure architecture types — they're hybrids:

  • React + CMS — CMS → API → React SPA → UI. Common for blogs with admin features, marketing sites with interactive UI, client-managed content with a dynamic frontend.
  • Vanilla + CMS — CMS → JSON API → Vanilla JS renderer. Lightweight hybrid for small marketing sites, performance-first projects.
  • Full Stack + CMS — CMS → Backend API → Database → React Admin + Public UI. Enterprise SaaS platforms, marketplaces, complex content + user systems.

Practical Build Order

Regardless of which architecture you choose, build in this order:

  1. Define system type — static, app, or hybrid (use the decision tree above)
  2. Define data source — JSON, CMS, or database (choose the simplest that works)
  3. Build frontend first — routing, rendering system, UI components
  4. Add data layer — API or CMS integration, fetch and render
  5. Add infrastructure last — hosting, email, bot protection, CDN

This order ensures you always have a working (if simple) system at each step. You can launch after step 3 for a static site. You launch after step 4 for a CMS site. Infrastructure is always last because it depends on knowing what you're deploying.

Key Engineering Principle

The Most Important Rule

UI design, data structure, and system architecture are separate concerns. If they are mixed early, your system becomes fragile, scaling becomes expensive, and the CMS becomes messy. If they are separated, systems remain replaceable, the backend can change without rewriting the UI, and the frontend stays stable regardless of what happens behind it.

Think in layers, not frameworks. Framework choice is just how many layers you introduce — not what the system fundamentally is.

Junior Developer

How a Junior Thinks About Architecture

  • Starts coding without any architectural plan
  • Picks a framework because it's popular, not because it fits the problem
  • Mixes presentation, logic, and data access in a single file
  • Doesn't consider how the system will evolve
  • Uses a full backend when a JSON file would suffice
Senior Developer

How a Senior Thinks About Architecture

  • Sketches the architecture before writing any code
  • Evaluates architecture patterns against project constraints
  • Enforces layer separation with clear interface boundaries
  • Designs for future scaling and team growth
  • Chooses the minimal viable architecture first, adds complexity only when measured data demands it
