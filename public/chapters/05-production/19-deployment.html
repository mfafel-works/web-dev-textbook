Chapter 19

Deployment

From static sites to full-stack apps — Cloudflare Pages, VPS, Docker, AWS, and decision framework

Deployment is how your code becomes a live website. The right platform depends on your project type, traffic expectations, team size, and budget. This chapter covers the full spectrum — from zero-cost static hosting to production-grade infrastructure — with a decision framework to choose the right option.

Choosing the Right Platform

Static site or SPA? ├── Yes → Git-based host (Cloudflare Pages, Vercel, Netlify) └── No → Server-side app? ├── Small traffic, PHP/MySQL? → Shared hosting or cheap VPS ├── Need reproducible environments? → Docker on VPS ├── Growing traffic, need scaling? → PaaS or cloud └── Enterprise / compliance? → AWS, GCP, Azure

The cheapest option is almost always the right starting point. You can migrate later if you grow. Most projects never need to.

The Migration Path

Start with Cloudflare Pages (free) for any static or SPA project. When you need a backend, add a serverless function or a cheap VPS. As traffic grows, containerize with Docker. Only reach for AWS/GCP/Azure when you need managed services (RDS, ElastiCache) or have a team that demands them. Each step up the ladder adds complexity — don't climb unless you have to.

Cloudflare Pages (Recommended for Static Sites)

Cloudflare Pages is a Git-integrated static site host that builds on every push and deploys to Cloudflare's global edge network. It is free, fast, and handles HTTPS, CDN, and DDoS protection automatically.

Setup

  1. Push your code to GitHub, GitLab, or Bitbucket
  2. In Cloudflare, go to Pages → "Connect to Git" → authorize and select your repo
  3. Set build command (empty for plain HTML) and output directory
  4. Deploy — your site is live at your-project.pages.dev

Subsequent pushes trigger automatic redeploys. Pull requests get preview deployments for testing before merging.

Custom Domains and HTTPS

  1. In Cloudflare Pages → Custom Domains → "Set up a custom domain"
  2. Update nameservers at your domain registrar to Cloudflare's
  3. Cloudflare provisions and auto-renews a free SSL/TLS certificate — no manual certificate management

CDN and Caching

Cloudflare's CDN caches assets at 330+ data centers. Cache behavior is controlled by a _headers file:

_headers — Cache rules
/assets/*
  Cache-Control: public, max-age=31536000, immutable
/
  Cache-Control: public, max-age=0, must-revalidate
/*.html
  Cache-Control: public, max-age=0, must-revalidate

Security Headers via _headers

_headers — Security
/*
  X-Content-Type-Options: nosniff
  X-Frame-Options: DENY
  Strict-Transport-Security: max-age=31536000; includeSubDomains
  Referrer-Policy: strict-origin-when-cross-origin

Turnstile (Bot Protection)

Cloudflare Turnstile is a privacy-focused CAPTCHA alternative — users pass invisibly or with one click. Get your site key and secret key from the Cloudflare dashboard, add the widget to your form, and verify the token server-side via /siteverify.

Turnstile supports three modes: non-interactive (invisible check, passes most visitors), managed (shows a challenge only if the visitor is suspicious), and invisible (fully invisible, calls your callback). Start with non-interactive — it balances security and user experience without adding friction.

Workers (Edge Functions)

Workers run JavaScript at Cloudflare's edge — serverless functions that execute on every request with no cold starts. Use them for API proxying, A/B testing, JWT validation, or geolocation-based content. Free tier: 100,000 requests/day.

Basic Worker — A/B test
export default {
  async fetch(request) {
    const url = new URL(request.url);
    const cookie = request.headers.get('Cookie');

    if (cookie && cookie.includes('variant=b')) {
      url.pathname = '/b' + url.pathname;
    }
    return fetch(url);
  }
}

VPS (DigitalOcean, Linode, Hetzner)

A VPS gives you full control — root access, any software, any configuration. You manage the OS, web server, firewall, database, and deployment pipeline.

  1. Create a droplet (Ubuntu, cheapest plan, enable SSH key auth)
  2. SSH in: ssh root@your-server-ip
  3. Install Nginx: apt install nginx
  4. Set up firewall: allow ports 22, 80, 443 only
  5. Deploy code via rsync, scp, or Git hooks
  6. Configure HTTPS with Certbot: certbot --nginx (free Let's Encrypt)

Choose VPS when you need full control or a non-standard stack. The tradeoff: you are now the sysadmin — updates, backups, and monitoring are your responsibility.

Docker and Containerization

Docker packages your app and its dependencies into a container that runs identically everywhere — eliminating "it works on my machine."

Dockerfile for a static site
FROM nginx:alpine
COPY public/ /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
docker-compose.yml — app + database
version: '3'
services:
  app:
    build: .
    ports:
      - "80:80"
    depends_on:
      - db
  db:
    image: postgres:15
    environment:
      POSTGRES_DB: myapp
      POSTGRES_PASSWORD: securepass
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:

Docker shines in teams — every developer and every deployment target runs the same container. No version mismatches, no OS differences.

Multi-Stage Builds

Multi-stage builds keep production images small by separating build dependencies from runtime dependencies:

Multi-stage Dockerfile for a Node.js app
# Stage 1: Install ALL dependencies and build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production image — only what's needed
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
EXPOSE 3000
CMD ["node", "dist/server.js"]

The production image contains only the compiled output and runtime dependencies — no build tools, no source code. Typical reduction: 1.2GB → 150MB.

Docker Best Practices

Use specific image tags (node:20-alpine not node:latest) for reproducible builds. Add a HEALTHCHECK instruction so orchestrators know when your container is ready. Use .dockerignore to exclude node_modules and build artifacts. Never store secrets in images — use environment variables at runtime.

Traditional Hosting (cPanel, FTP)

Shared hosting is still common for PHP/MySQL projects. You get a control panel, one-click CMS installs, and email — but limited performance and no SSH access.

  • Pros: Cheap ($3-15/mo), no sysadmin work
  • Cons: Shared resources, no Docker, limited Node.js/Python
  • Best for: Small PHP/WordPress sites, client projects where you won't maintain the server

Use SFTP (not plain FTP) for encrypted transfers, and keep a local Git repo even if you cannot run Git on the server.

CI/CD Pipelines

Continuous Integration and Continuous Deployment automate the steps between pushing code and having it live. Every deploy should go through the same pipeline — no manual steps, no "it worked on my machine."

GitHub Actions — Deploy to Cloudflare Pages
name: Deploy
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test
      - run: npm run build
      - uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CF_API_TOKEN }}
          command: pages deploy dist/ --project-name=my-app

Key pipeline principles:

  • Fail fast — run linting and tests before building. If tests fail, the pipeline stops immediately.
  • Artifact immutability — build once, promote the same artifact through environments (dev → staging → prod). Never rebuild for production.
  • Environment parity — use the same Docker image in development, CI, and production. Differences between environments are the #1 source of deployment bugs.
  • Preview deployments — each PR gets a unique URL for testing before merging. Cloudflare Pages and Vercel do this automatically.

Nginx as a Reverse Proxy

When running a VPS, Nginx sits in front of your application server, handling SSL termination, static file serving, and request routing.

Nginx reverse proxy config
server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Proxy API requests to the Node.js app
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # Serve static files directly
    location / {
        root /var/www/myapp/dist;
        try_files $uri $uri/ /index.html;  # SPA fallback
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}

Nginx handles static file serving (much faster than Node.js/Python), buffers slow clients, and provides a security layer before requests reach your application. The SPA fallback (try_files ... /index.html) is critical — without it, refreshing any non-root route returns a 404.

Always Redirect HTTP to HTTPS

The return 301 block at the top redirects all HTTP traffic to HTTPS. Never serve your app over plain HTTP — every request (including passwords, API keys, session cookies) travels in cleartext. Let's Encrypt provides free certificates that auto-renew via Certbot.

Zero-Downtime Deployment

Zero-downtime deployment ensures users never see an error page during a deploy. Two common strategies:

StrategyHow It WorksComplexityBest For
Blue-GreenRun two identical environments. Deploy to the inactive one, then swap the load balancer.MediumProduction apps where downtime costs money
CanaryRoute a small % of traffic to the new version, gradually increase as confidence grows.HighHigh-traffic apps where bugs affect many users
RollingUpdate servers one at a time while the rest serve traffic.LowContainerized apps on Kubernetes / Nomad
Symlink SwapBuild new version to a timestamped directory, swap a symlink, reload Nginx.LowVPS with Nginx (simple and effective)
Symlink swap deployment script
#!/bin/bash
# deploy.sh — zero-downtime via symlink swap
TIMESTAMP=$(date +%Y%m%d-%H%M%S)
DEPLOY_DIR="/var/www/myapp/releases/$TIMESTAMP"
CURRENT_LINK="/var/www/myapp/current"

# Build the new release
mkdir -p "$DEPLOY_DIR"
git clone git@github.com:user/myapp.git "$DEPLOY_DIR"
cd "$DEPLOY_DIR"
npm ci && npm run build

# Swap the symlink and reload
ln -sfn "$DEPLOY_DIR" "$CURRENT_LINK"
nginx -s reload

# Remove old releases (keep last 3)
ls -t /var/www/myapp/releases/ | tail -n +4 | \
  xargs -I {} rm -rf "/var/www/myapp/releases/{}"

The symlink swap is atomic — either users see the old version or the new one, never a half-deployed state. If the new version fails, swap the symlink back instantly.

Feature Flags and Rollbacks

Deployment is not just about getting code to production — it is about managing risk when things go wrong.

  • Feature flags — Ship code behind a toggle. Turn it on for internal users first, then 1% of traffic, then ramp up. If something breaks, flip the toggle off without redeploying.
  • Rollbacks — Every deployment platform supports rollback to a previous version. On Cloudflare Pages, click "Rollback" in the deployment dashboard. On a VPS, keep the previous build directory and swap the symlink back.
  • Incremental deploy — Deploy to a staging or canary environment first. Run smoke tests. Then promote to production. This catches environment-specific issues before they affect users.
Always Have a Rollback Plan

Every deployment is a risk. Before deploying, know exactly how to undo it. If your platform supports one-click rollback (Cloudflare Pages, Vercel), great. If not, script it: keep the last three builds, symlink the active one, and keep the symlink in version control. A deploy without a rollback plan is not a deploy — it is a gamble.
