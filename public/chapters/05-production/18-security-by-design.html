Chapter 18

Security by Design

XSS, CSRF, SQL injection, HTTPS, CSP, rate limiting — security isn't an afterthought

Security is not a feature you bolt on at the end. Every line of code you write either adds or removes attack surface. This chapter gives you the mental model to think about security from the start: assume every input is malicious, every connection is intercepted, and every user is an attacker until proven otherwise.

The Security Mindset

Before we talk about specific attacks, establish the principles that guide every security decision:

  • Defense in depth — never rely on a single security layer. If your firewall fails, your input validation should still catch the attack. If your validation fails, your parameterized queries should still prevent injection. Layers overlap so no single failure is catastrophic.
  • Least privilege — every component, user, and process should have only the permissions it needs. A database user that only reads public articles should not have DELETE access. An API key that only sends emails should not have access to user data.
  • Never trust user input — treat every byte from the network as hostile. This includes URL parameters, headers, cookies, file uploads, JSON bodies, and even seemingly safe data like User-Agent strings.
  • Fail securely — when something goes wrong, the system should default to a locked state, not an open one. An unhandled error should deny access, not grant it.
  • Security is not privacy — security prevents unauthorized access. Privacy is about how you handle data you are authorized to collect. You need both, but they require different thinking.
Why This Matters

The adversarial mindset is not pessimism — it is realism. Every public web server is scanned by automated bots within hours of going live. Within days, it will be probed for SQL injection, directory traversal, XSS vectors, and known CVE exploits. Security is not paranoia. It is seeing the environment as it actually is.

Input Validation

Every feature that accepts input is an attack surface. The question is not "is this input safe?" but "how is this input dangerous, and how do I make it safe?"

Validation vs Sanitization

These are different operations with different goals:

ConceptWhat it doesExample
ValidationRejects input that does not match expected formatAn email field must contain @ and a domain. If it does not, reject the request with an error.
SanitizationRemoves or escapes dangerous parts of inputStripping <script> tags from a rich text field before storing it

Use validation when you can define exact expected formats. Use sanitization when you must accept flexible input. Validation is always preferred because it is stricter and has no edge cases. Sanitization is a fallback for when you genuinely need to accept free-form content.

Whitelist vs Blacklist

Whitelist (allowlist): define what is allowed, reject everything else. Blacklist (blocklist): define what is forbidden, allow everything else. Always prefer whitelisting for security. Blacklists can never be complete — attackers will always find something you did not think to block.

Good: "Only allow alphanumeric characters, @, ., and -" → Bad: "Block <, >, ', \x22, --, and other known bad patterns"

File Upload Security

File uploads are one of the most exploited surfaces. An attacker can upload a PHP file disguised as an image and execute it on your server if you are not careful.

  • Validate the file extension against a whitelist (not by checking Content-Type header — that is trivially spoofed)
  • Validate the file content by reading magic bytes, not by trusting the extension or MIME type
  • Store uploaded files outside the web root or use a CDN with restricted execution
  • Rename files on upload to prevent path traversal and name collision attacks
  • Scan files for malware if your application is high-risk
PHP — safe file upload validation
<?php
$allowed_extensions = ['jpg', 'png', 'gif', 'webp'];
$allowed_mime = ['image/jpeg', 'image/png', 'image/gif', 'image/webp'];

$ext = strtolower(pathinfo($_FILES['file']['name'], PATHINFO_EXTENSION));
$mime = mime_content_type($_FILES['file']['tmp_name']);

if (!in_array($ext, $allowed_extensions)) {
    die('Invalid file type.');
}
if (!in_array($mime, $allowed_mime)) {
    die('File content does not match allowed types.');
}

// Rename and move
$safe_name = bin2hex(random_bytes(16)) . '.' . $ext;
move_uploaded_file($_FILES['file']['tmp_name'], 'uploads/' . $safe_name);

Notice: the extension is checked, the actual content is verified via mime_content_type(), and the stored name is randomized. Even if an attacker uploads a file containing PHP code, it will never execute because it is stored with a .jpg extension and served as static content.

Never Trust Frontend Validation

Client-side validation is for user convenience, not security. Every validation check you perform in the browser can be bypassed by any moderately skilled attacker — by disabling JavaScript, editing the DOM, or sending raw HTTP requests with curl. Always re-validate everything on the server side. Treat the frontend as untrusted by default.

Cross-Site Scripting (XSS)

XSS is the most common web vulnerability. It occurs when an attacker injects executable code (usually JavaScript) into a page that other users view. There are three types, and you need to understand all of them because each requires a different prevention strategy.

Stored XSS (Persistent)

The attacker submits malicious code that gets stored on the server (in a database, comment field, profile bio, etc.) and served to every visitor. This is the most dangerous type because it affects many users without direct attacker interaction.

Vulnerable: Stored XSS in a comment system
<!-- User submits: <script>fetch('/steal-cookies')</script> -->
<div class="comment">
  <?= $_POST['comment'] ?>  <!-- Outputs raw JS into page -->
</div>

Reflected XSS (Non-persistent)

The injected code is part of the request (typically a URL parameter) and is reflected back in the response immediately. The attacker must trick the victim into clicking a crafted link. While less damaging than stored XSS, it is extremely common in search pages and error handlers.

Vulnerable: Reflected XSS in a search page
<!-- URL: /search?q=<script>alert('xss')</script> -->
<p>Search results for: <?= $_GET['q'] ?></p>

DOM-based XSS

The vulnerability exists entirely in client-side JavaScript. The server response is safe, but the JS reads attacker-controlled data from the URL hash or document.referrer and writes it into the DOM unsafely.

Vulnerable: DOM-based XSS
// URL: /page#<img onerror="alert(1)" src=x>
const hash = location.hash.slice(1);
document.getElementById('output').innerHTML = hash; // DANGEROUS
Safe alternative
const hash = location.hash.slice(1);
document.getElementById('output').textContent = hash; // SAFE — no HTML parsing

XSS Prevention Summary

ContextDangerous API / PatternSafe Alternative
HTML bodyinnerHTML, outerHTML, document.write()textContent, createTextNode()
HTML attributeString concatenation in setAttribute() or template literalssetAttribute() with validated values, dataset API
URL contextjavascript: URLs in a.href or locationValidate protocol is https? before assigning
CSS contextstyle attribute with user inputUse CSS classes, never raw style strings from user data
Server-side templateUnescaped output (<?= $var ?> in PHP without escaping)Context-aware escaping: htmlspecialchars(), template engine auto-escaping
Why This Works

The most effective XSS prevention is Content Security Policy (covered below) combined with never using innerHTML with user-controlled data. CSP acts as a safety net: even if you accidentally introduce an XSS bug, the browser blocks script execution if the policy is strict enough. This is defense in depth at the browser level.

CSRF Protection

Cross-Site Request Forgery (CSRF) tricks an authenticated user into performing actions they did not intend. The attacker crafts a malicious page that, when visited by a logged-in user, submits a request to your application using the user's existing session cookie.

How CSRF Works

The key insight: the browser automatically attaches cookies to requests regardless of where the request originated. The server cannot distinguish between a legitimate request from the real site and a forged request from an attacker's page — unless we add an anti-CSRF mechanism.

Anti-CSRF Token (Synchronizer Token Pattern)

The server generates a random token, associates it with the user's session, and includes it in every form. When the form is submitted, the server verifies the token matches. The attacker's site cannot read this token because of the same-origin policy.

PHP — CSRF token generation and validation
<?php
session_start();

// Generate token if not exists
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}

// In form:
// <input type="hidden" name="csrf_token" value="<?= $_SESSION['csrf_token'] ?>">

// On submission:
if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'] ?? '')) {
    die('CSRF token validation failed.');
}

Use hash_equals() (or timingSafeEqual in Node) for comparison — it prevents timing attacks by taking the same amount of time regardless of how many characters match.

SameSite Cookies

Modern browsers support the SameSite cookie attribute, which is the simplest CSRF defense:

  • SameSite=Strict — cookie is never sent for cross-site requests. Most secure, but breaks legitimate cross-site links (e.g., a link from an email to your site)
  • SameSite=Lax — cookie is sent for top-level GET navigations (links from other sites work), but not for POST requests from other origins. This is the modern default and covers most CSRF scenarios.
  • SameSite=None — cookie is sent for all requests. Requires Secure flag (HTTPS only). Only use when you explicitly need cross-site cookie access.

Best practice: Use SameSite=Lax as your default and add CSRF tokens for sensitive forms as a second layer.

Origin and Referer Header Checks

A simpler but less reliable defense: verify the Origin or Referer header matches your domain. This works because those headers are not set by cross-site form submissions from <form> elements in the same way. However, these headers can be missing or spoofed in some scenarios, so they are best used as a secondary check.

SQL Injection Prevention

SQL injection is the easiest vulnerability to prevent and one of the most devastating when missed. An attacker can read, modify, or delete your entire database through a single unescaped query string.

The Vulnerability

When you concatenate user input directly into a SQL query, you give the attacker control over the query structure:

Vulnerable: String concatenation
<?php
$id = $_GET['id'];
$sql = "SELECT * FROM users WHERE id = $id"; // Attacker sets id=1;DROP TABLE users;
$result = mysqli_query($conn, $sql);

If id=1;DROP TABLE users;--, the query becomes: SELECT * FROM users WHERE id = 1; DROP TABLE users; --. The -- comments out the rest of the query.

Parameterized Queries (Prepared Statements)

The only reliable prevention. The query structure is sent to the database first, and user input is bound as data — never as part of the SQL syntax.

Safe: Parameterized query (PHP PDO)
<?php
$stmt = $pdo->prepare('SELECT * FROM users WHERE id = :id');
$stmt->execute(['id' => $_GET['id']]);
$user = $stmt->fetch();
Safe: Parameterized query (Node.js with mysql2)
const [rows] = await connection.execute(
  'SELECT * FROM users WHERE id = ?',
  [req.query.id]
);

The database receives the query structure and the data separately. Even if the user passes 1; DROP TABLE users, it is treated as a string value for id, not as SQL syntax.

Escaping Is Not Enough

Functions like mysqli_real_escape_string() escape certain characters, but they can still be bypassed depending on the database charset and query context. The only safe approach is parameterized queries. Escaping is a band-aid. Prepared statements are the cure.

ORM Protection

Modern ORMs (Eloquent, Prisma, Doctrine, Sequelize) use parameterized queries internally, but you can still bypass them with raw query methods. Always use the ORM's query builder instead of raw SQL when possible:

Safe: ORM query (Prisma)
const user = await prisma.user.findUnique({
  where: { id: req.params.id }  // Prisma parameterizes this internally
});

Authentication Security

Authentication is the most security-critical feature you will build. A single mistake here compromises every user's account.

Password Hashing

Never store passwords in plain text. Never use MD5, SHA-1, or even plain SHA-256 for passwords. These are hash functions designed for speed — and speed is the enemy of password storage because it lets attackers try billions of combinations per second.

Use a purpose-built password hashing algorithm:

AlgorithmUse CasePHPNode.js
bcryptDefault choice for most appspassword_hash()bcrypt package
Argon2idStrongest option (PHP 7.2+, Node 16+)PASSWORD_ARGON2IDargon2 package
PHP — password hashing (automatic salt, variable cost)
<?php
// Hashing
$hash = password_hash($password, PASSWORD_BCRYPT, ['cost' => 12]);

// Verification
if (password_verify($input_password, $hash)) {
    // Login successful
}

The password_hash() function handles salting automatically — never manually generate or store salts. The cost factor (12) controls how many computation rounds are used. Increase it over time as hardware gets faster.

Session Management

  • Use server-side sessions, not client-side (JWT is fine, but JWTs stored in localStorage cannot be invalidated server-side — you lose control)
  • Regenerate the session ID after login to prevent session fixation
  • Set session timeout and absolute expiry (e.g., 30 minutes inactivity, 24 hours absolute)
  • Use secure, httpOnly, SameSite cookies for session tokens
  • Store session data server-side (Redis, database, or filesystem), not in the cookie itself
PHP — secure session configuration
<?php
session_set_cookie_params([
    'lifetime' => 0, // Until browser closes
    'path' => '/',
    'domain' => 'yourdomain.com',
    'secure' => true,     // HTTPS only
    'httponly' => true,   // Inaccessible to JavaScript
    'samesite' => 'Lax'
]);
session_start();

// Regenerate after login
session_regenerate_id(true);

HTTPS and Security Headers

HTTPS is non-negotiable in 2026. Every site — including personal blogs and prototypes — should use HTTPS. Browsers now mark HTTP sites as "Not Secure" and some APIs (geolocation, camera, notifications) require HTTPS to work at all.

TLS Configuration

With Cloudflare, Let's Encrypt, and most hosting platforms, TLS is free and automatic. If you are configuring it manually:

  • Use TLS 1.2 and 1.3 only (disable TLS 1.0, 1.1, and SSL 2.0/3.0)
  • Use modern cipher suites (ECDHE + AES-GCM or ChaCha20)
  • Set HSTS (HTTP Strict Transport Security) to tell browsers to always use HTTPS
  • Use a tool like SSL Labs to verify your configuration

Security Headers Reference

These six headers provide massive security improvements for almost zero effort:

HeaderValueWhat It Prevents
Content-Security-Policydefault-src 'self'XSS and data injection — restricts which resources can be loaded and executed
Strict-Transport-Securitymax-age=31536000; includeSubDomainsSSL stripping and downgrade attacks — forces HTTPS for one year
X-Content-Type-OptionsnosniffMIME-type sniffing — prevents browser from interpreting files as a different type than declared
X-Frame-OptionsDENYClickjacking — prevents your site from being embedded in iframes on other domains
Referrer-Policystrict-origin-when-cross-originInformation leakage — controls what referrer data is sent with requests
Permissions-Policycamera=(), microphone=(), geolocation=()Feature restriction — disables browser features your site does not need
Complete security headers configuration
# Cloudflare Pages _headers file
/*
  Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' https:; connect-src 'self' https:; font-src 'self'; object-src 'none'; base-uri 'self'; form-action 'self'
  Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
  X-Content-Type-Options: nosniff
  X-Frame-Options: DENY
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: camera=(), microphone=(), geolocation=()

The CSP shown here is a reasonable starting point: scripts and styles only from your own domain, images from your domain or any HTTPS source, connections only to your domain and HTTPS APIs, and inline styles allowed (needed for many frameworks). Tighten it further by removing 'unsafe-inline' if your CSS is all in external files.

CSP Reporting

CSP supports a report-uri or report-to directive that sends violation reports to an endpoint. Deploy with Content-Security-Policy-Report-Only first to catch violations without blocking them, then switch to enforcement once you have tuned the policy.

API Security

APIs are the backbone of modern web applications. If your frontend communicates with a backend API, that API needs its own security layer.

Rate Limiting

Rate limiting prevents a single client from overwhelming your server, whether through a DoS attack, brute force login attempt, or just a buggy client sending infinite retries.

  • IP-based — simplest but can block legit users behind a shared IP (office, college, VPN)
  • User-based — keyed to the authenticated user (more accurate but requires auth)
  • Endpoint-based — different limits for different endpoints (login: 5/min, read: 100/min, write: 20/min)
  • Global — total request cap per client across all endpoints
PHP — simple IP-based rate limiting with Redis
<?php
$ip = $_SERVER['REMOTE_ADDR'];
$key = "rate_limit:$ip:" . date('YmdH'); // Per hour
$current = $redis->incr($key);
if ($current > 1000) {
    http_response_code(429);
    die('Rate limit exceeded.');
}

Return HTTP 429 (Too Many Requests) with a Retry-After header so the client knows when to retry.

CORS (Cross-Origin Resource Sharing)

CORS is a browser mechanism — it does not actually block requests at the HTTP level. A properly configured server restricts which origins can access its resources:

PHP — strict CORS headers
<?php
$allowed_origins = ['https://mysite.com', 'https://www.mysite.com'];
$origin = $_SERVER['HTTP_ORIGIN'] ?? '';

if (in_array($origin, $allowed_origins)) {
    header("Access-Control-Allow-Origin: $origin");
    header("Access-Control-Allow-Methods: GET, POST");
    header("Access-Control-Allow-Headers: Content-Type, Authorization");
    header("Access-Control-Max-Age: 86400");
}

if ($_SERVER['REQUEST_METHOD'] === 'OPTIONS') {
    http_response_code(204);
    exit;
}

Never use Access-Control-Allow-Origin: * for authenticated endpoints. Only whitelist specific origins.

API Keys

  • Treat API keys as passwords — store them hashed (if you do not need to retrieve the raw value) or encrypted
  • Never expose API keys in client-side code. Use a backend proxy if the frontend needs to call a third-party API
  • Rotate keys periodically and allow key revocation
  • Scope keys to specific permissions (read-only, write, admin) rather than using a single master key
  • Monitor key usage for anomalies

Bot Protection with Cloudflare Turnstile

CAPTCHAs have evolved from distorted text (annoying) to image selection (less annoying) to invisible challenges. Cloudflare Turnstile is the current best option: it runs a non-interactive browser check (JS challenge, canvas fingerprinting, behavior analysis) and typically requires zero user interaction.

Turnstile Integration

Frontend: Turnstile widget (SPA-compatible)
<!-- Load the API -->
<script src="https://challenges.cloudflare.com/turnstile/v0/api.js" async defer></script>

<!-- Widget container -->
<div id="turnstile-widget"></div>
JavaScript: Render widget and capture token
window.onloadTurnstileCallback = function () {
  turnstile.render('#turnstile-widget', {
    sitekey: 'YOUR_SITE_KEY',
    callback: function(token) {
      // Token is valid — include in form submission
      document.getElementById('cf-token').value = token;
    }
  });
};
PHP: Server-side verification
<?php
$turnstile_secret = 'YOUR_SECRET_KEY';
$token = $_POST['cf-turnstile-response'] ?? '';

$ch = curl_init('https://challenges.cloudflare.com/turnstile/v0/siteverify');
curl_setopt($ch, CURLOPT_POST, true);
curl_setopt($ch, CURLOPT_POSTFIELDS, http_build_query([
    'secret' => $turnstile_secret,
    'response' => $token,
    'remoteip' => $_SERVER['REMOTE_ADDR']
]));
curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
$response = json_decode(curl_exec($ch), true);
curl_close($ch);

if (!$response['success']) {
    die('Bot verification failed.');
}
SPA Token Lifecycle

In an SPA, the Turnstile token is generated once when the page loads. If your token expires before the user submits a long form, you need to call turnstile.reset() and regenerate it. Alternatively, call turnstile.getResponse() just before submission to get a fresh token. Always verify the token server-side — never trust client-side validation alone.

Security Checklist

Before deploying any application, go through this checklist:

CategoryCheck
HTTPSEnforce HTTPS at the server/edge level. Redirect all HTTP traffic. Set HSTS.
HeadersCSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy configured
AuthenticationPasswords hashed with bcrypt or Argon2id. Sessions use secure httpOnly SameSite cookies. Session IDs regenerated after login.
InputAll input validated server-side. Parameterized queries for all database operations. File uploads validated by extension AND content.
OutputAll user-supplied data contextually escaped on output (HTML, JS, URL, CSS contexts). CSP in enforcement mode.
APIRate limiting active. CORS restricted to specific origins. API keys scoped and rotatable.
Bot protectionTurnstile or equivalent on forms. Rate limiting on login and signup endpoints.
DependenciesAll third-party packages audited and updated. No known CVEs. SRI hashes on external scripts.
BackupAutomated database backups. Off-site backup storage. Tested restore process.
LoggingSuspicious activity logged. Login attempts (success and failure) logged. No sensitive data in logs.
Security Is a Process, Not a Product

This checklist is a starting point, not a final destination. Security posture evolves as new vulnerabilities are discovered, new tools are built, and your application grows. Revisit your security configuration every quarter. Subscribe to security advisories for your dependencies. And never assume you are "done" with security.

Junior Developer

How a Junior Thinks

  • Adds security at the end, if there is time
  • Relies on frontend validation as a security measure
  • Uses string concatenation in SQL because "it works on my machine"
  • Ignores security headers because the site "looks fine"
  • Stores passwords with SHA-256 because "it's secure, right?"
  • Thinks HTTPS is only for e-commerce and banking
  • Deploys without checking once for security misconfigurations
Senior Developer

How a Senior Thinks

  • Bakes security into every architectural decision from day one
  • Treats the frontend as completely untrusted
  • Uses parameterized queries or ORM as the only option — no exceptions
  • Configures CSP, HSTS, and other headers before the first deploy
  • Uses bcrypt or Argon2id and understands what cost factor means
  • Enforces HTTPS at the infrastructure level, not the application level
  • Runs through a security checklist before every production release
