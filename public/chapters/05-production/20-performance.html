Chapter 20

Performance

Core Web Vitals, measuring, optimizing images, code splitting, caching, and budgets

Performance is a feature. Users expect pages to load in under 2 seconds — every additional second of load time reduces conversions by 7% on desktop and up to 20% on mobile. Google uses performance as a ranking factor. And performance is accessibility: slow sites disproportionately affect users on slow connections, older devices, and in developing markets. This chapter covers what to measure, what to optimize, and how to stay fast.

Core Web Vitals

Google's Core Web Vitals are the three metrics that matter most for user experience. They are also search ranking factors:

MetricWhat It MeasuresGoodPoor
LCP (Largest Contentful Paint)Loading — when the largest visible element (hero image, heading) finishes rendering≤ 2.5s> 4.0s
FID (First Input Delay)Interactivity — time from user first interaction to browser being able to respond≤ 100ms> 300ms
CLS (Cumulative Layout Shift)Visual stability — how much the page jumps around while loading≤ 0.1> 0.25

LCP is typically improved by optimizing images, eliminating render-blocking resources, and using a CDN. FID is improved by code splitting, deferring non-critical JavaScript, and avoiding long tasks. CLS is improved by setting explicit dimensions on images and ads, using font-display: swap, and avoiding injecting content above existing content.

Always Set Image Dimensions

The single most common CLS cause: images without width and height attributes. The browser reserves zero space until the image loads, then the page jumps. Always set <img width="800" height="600"> — the browser uses these to calculate aspect ratio and reserve the correct space even before the image downloads. Combined with CSS max-width: 100%; height: auto, this prevents layout shift.

Measure on Mobile First

Most performance issues are worse on mobile: slower CPUs, limited memory, smaller screens, flaky connections. Lighthouse defaults to mobile emulation for a reason. Optimize for mobile first, then verify desktop. A fast mobile experience is almost always fast on desktop. The reverse is not true.

Image Optimization

Images account for 60-70% of a typical page's weight. Optimizing images is the single highest-impact performance improvement you can make.

Format Decision Tree

Is the image photographic or has complex gradients? ├── YES → Does it need transparency? │ ├── YES → WebP (lossy, supports transparency) │ └── NO → WebP (lossy) or AVIF (even smaller, less supported) └── NO (flat colors, logos, icons) → SVG if vector source exists → WebP (lossless) or PNG if no vector source

Techniques

TechniqueImpactHow
Resize to max display sizeHighDo not serve a 4000px image for a 600px container
Modern formats (WebP, AVIF)High20-30% smaller than JPEG, 30-50% smaller than PNG
Responsive images (srcset)HighServe different sizes for different viewports
Lazy loadingMediumloading="lazy" on below-the-fold images
CompressionMediumJPEG quality 80%, PNGQuant for PNGs
CDN with image transformationHighCloudflare Images, Imgix, Cloudinary — transform on the fly
Responsive image with WebP fallback
<picture>
  <source srcset="photo.avif" type="image/avif">
  <source srcset="photo.webp" type="image/webp">
  <img src="photo.jpg" width="800" height="600"
       alt="Description"
       loading="lazy">
</picture>

Image CDNs

Image CDNs (Cloudinary, Imgix, Cloudflare Images) automatically transform images on the fly — resize, format conversion, compression, cropping — based on URL parameters. This eliminates the need to generate multiple image sizes at build time:

Cloudinary URL transformations
// Original: https://res.cloudinary.com/demo/image/upload/sample.jpg

// Resize to 400px wide, auto-quality, WebP format
https://res.cloudinary.com/demo/image/upload/w_400,q_auto,f_webp/sample.jpg

// Responsive via srcset
<img srcset="
  https://.../w_400/sample.jpg 400w,
  https://.../w_800/sample.jpg 800w,
  https://.../w_1200/sample.jpg 1200w"
  sizes="(max-width: 600px) 100vw, 800px"
  src="https://.../w_800/sample.jpg"
  width="800" height="600" alt="">

Image CDNs also handle CDN delivery, caching, and format negotiation. The browser requests the best format automatically. For projects with many images, an image CDN is worth the cost — it replaces hours of manual optimization with URL parameters.

HTTP/2, HTTP/3, and Modern Protocols

HTTP versions affect how browsers download resources. Each version improves performance in different ways:

VersionKey FeaturePerformance Impact
HTTP/1.1One request per connectionBrowsers open 6 parallel connections per domain. Bundling CSS/JS into fewer files was necessary to reduce request overhead.
HTTP/2Multiplexing — multiple requests over one connectionOne TCP connection handles many concurrent streams. No head-of-line blocking. Enables server push (rarely used). Makes file bundling less critical.
HTTP/3 (QUIC)UDP-based, no head-of-line blocking at transport layerFaster connection establishment (0-RTT), better performance on lossy networks (mobile). No TCP head-of-line blocking.

All modern hosting platforms (Cloudflare, Vercel, Netlify, AWS CloudFront) support HTTP/2 and HTTP/3 automatically — no configuration needed. The practical effect: you can serve more individual files without performance penalty, reducing the need for aggressive CSS/JS bundling. However, still do minimize and compress resources — smaller files are always faster regardless of protocol.

Real User Monitoring (RUM)

Synthetic testing (Lighthouse, PageSpeed Insights) tells you how fast a page can be. Real User Monitoring tells you how fast it actually is for your users, on their devices, networks, and locations.

Collecting Web Vitals in the browser
import { onLCP, onFID, onCLS } from 'web-vitals';

function sendToAnalytics(metric) {
  const body = {
    name: metric.name,
    value: metric.value,
    rating: metric.rating,      // 'good' | 'needs-improvement' | 'poor'
    delta: metric.delta,
    id: metric.id,
    url: location.href,
    userAgent: navigator.userAgent,
  };

  // Send to your analytics endpoint
  navigator.sendBeacon('/analytics', JSON.stringify(body));
}

onLCP(sendToAnalytics);
onFID(sendToAnalytics);
onCLS(sendToAnalytics);

Send the data to Google Analytics 4 (which tracks Web Vitals automatically), Datadog RUM, Sentry Performance, or your own backend. Watch the 75th percentile (p75) — if 75% of your users have a good experience, most are happy. Ignore outliers (the 99th percentile often reflects edge cases like loading on a 2G network during a thunderstorm).

Synthetic vs RUM: Both Are Necessary

Synthetic tests catch issues before users see them. RUM tells you about issues users actually encounter. Use Lighthouse in CI to prevent regressions. Use RUM in production to discover problems your synthetic tests miss (slow 3G networks, older devices, regional CDN issues). They complement each other — you need both.

JavaScript and CSS Optimization

Code Splitting

Instead of sending one giant JavaScript bundle, split it into chunks that are loaded on demand. Modern bundlers (Vite, Webpack) support code splitting natively:

Dynamic import for code splitting
// Instead of:
import { HeavyComponent } from './HeavyComponent';

// Do this:
const HeavyComponent = () => import('./HeavyComponent');
// Webpack/Vite creates a separate chunk, loaded only when this component renders

Tree Shaking

Tree shaking removes unused code from the production bundle. It happens automatically when you import specific exports:

Tree shaking — import what you use
// BAD — imports the entire library (60KB)
import _ from 'lodash';
_.debounce(fn, 300);

// GOOD — imports only the function (4KB)
import debounce from 'lodash/debounce';
// or with ES modules:
import { debounce } from 'lodash-es';

Verify tree shaking with a bundle analyzer. Some packages (especially older ones) are not tree-shakeable — avoid them or use a modern alternative.

Critical CSS

Critical CSS is the minimal CSS needed to render the above-the-fold content. By inlining it in <head>, the page can render without waiting for the full CSS file to download:

Critical CSS pattern
<head>
  <style>
    /* Critical CSS — inline, blocks rendering but tiny */
    header { background: white; padding: 1rem; }
    .hero  { font-size: 2rem; margin: 2rem 0; }
  </style>
  <link rel="stylesheet" href="full.css" media="print" onload="this.media='all'">
</head>

Tools like critical (Node package) automate extracting critical CSS. Use this technique for pages where time-to-first-paint matters most (landing pages, marketing sites). For logged-in dashboards, the complexity may not be worth it.

Resource Hints

Tell the browser about important resources early:

Resource hints in <head>
<!-- Preload critical resources -->
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>

<!-- Preconnect to third-party origins -->
<link rel="preconnect" href="https://api.example.com">

<!-- Prefetch resources for the next page -->
<link rel="prefetch" href="/next-page" as="document">

Use preload sparingly — only for critical above-the-fold resources. Overuse competes with the browser's own resource prioritization.

Caching Strategy

Caching is the most impactful performance lever and the most common source of "it worked before" bugs. The browser caches responses based on the Cache-Control header sent by the server.

Resource TypeCache StrategyCache-Control Header
HTML pagesNo cache or short TTLno-cache (revalidate with server)
JavaScript bundles with hashImmutable, long cachepublic, max-age=31536000, immutable
CSS with hashImmutable, long cachepublic, max-age=31536000, immutable
Images (non-versioned)Long cache with revalidationpublic, max-age=2592000 (30 days)
FontsLong cachepublic, max-age=31536000, immutable
API responsesShort or no cacheno-cache or private, max-age=60

When filenames include a content hash (main.a1b2c3.js), you can cache forever — a new hash means a new file. When the hash changes, the browser treats it as a different URL and downloads the new version. This is called cache busting and is the reason modern build tools add hashes to output filenames.

Performance Budgets

A performance budget is a set of constraints you agree not to exceed. Without a budget, performance degrades gradually — a few KB here, a new library there — until the site is slow and no one knows which change caused it.

Example performance budget
{
  "budgets": [
    {
      "resourceType": "total",
      "budget": 300000,        // 300KB total JavaScript
      "resourceType": "total",
      "budget": 500000,        // 500KB total images
      "resourceType": "total",
      "budget": 100000         // 100KB total CSS
    },
    {
      "resourceType": "script",
      "budget": 200000,        // 200KB per script
      "resourceType": "image",
      "budget": 100000         // 100KB per image
    }
  ]
}

Tools like bundlesize, Lighthouse CI, and webpack-bundle-analyzer can enforce budgets in CI. When a PR exceeds the budget, the build fails. This is the only reliable way to prevent performance regression over time.

Start With a Generous Budget

Do not set a 100KB budget on day one if your site is already 500KB. Measure the current state, add 10-20% buffer, and set that as the initial budget. Then tighten it over time. A budget that is impossible to meet will be ignored. A budget that requires minor effort will be respected.

Junior Developer

How a Junior Might Think

  • Does not measure performance — "it loads fine on my machine"
  • Adds libraries without considering bundle impact
  • Uses giant PNGs exported at 3x resolution from Photoshop
  • Does not know what Core Web Vitals are
  • Forgets to set image dimensions — pages jump around during loading
  • Has no caching strategy — does not understand Cache-Control
Senior Developer

How a Senior Thinks

  • Measures with Lighthouse and real-user monitoring before optimizing
  • Checks bundle size impact before adding any dependency
  • Uses responsive images with srcset and modern formats (WebP, AVIF)
  • Monitors LCP, FID, and CLS as part of the development workflow
  • Sets explicit width/height on all images — zero layout shift
  • Enforces a performance budget to prevent regression
