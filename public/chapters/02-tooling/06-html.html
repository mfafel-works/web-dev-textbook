Chapter 6

HTML

Semantic structure, forms, accessibility, and building the content layer of the web

HTML is not a programming language. It is a markup language — a way of structuring content so that browsers, search engines, and assistive technologies all understand what matters. CSS handles appearance. JavaScript handles behavior. HTML handles meaning. Get the meaning right and everything else follows.

HTML Fundamentals

Before we build a full document, there are two HTML concepts that trip up every beginner: void elements (tags that don't need a closing tag) and HTML entities (how to type special characters).

Void Elements (Self-Closing Tags)

Most HTML tags come in pairs: <p>...</p>, <div>...</div>, <h1>...</h1>. But some elements have no content — they are a single tag that does not need a closing counterpart. These are called void elements:

TagPurposeExample
<meta>Metadata about the page<meta charset="UTF-8">
<link>Link to external resources (CSS, favicon)<link rel="stylesheet" href="style.css">
<img>Image<img src="photo.jpg" alt="Description">
<br>Line breakLine one<br>Line two
<hr>Thematic break (horizontal rule)<hr>
<input>Form input field<input type="email">
<source>Media source (for video/audio)<source src="video.mp4" type="video/mp4">

Writing <img></img> is valid but pointless — the closing tag does nothing. Writing <img /> (with a self-closing slash) is also valid but unnecessary in HTML (it is required in JSX/React). In HTML, just write <img ...>.

HTML Entities (Special Characters)

Some characters have special meaning in HTML. The angle brackets < and > are used for tags. The ampersand & starts an entity. To display these characters literally, you use named entities — codes that begin with & and end with ;:

CharacterEntityWhy You Need It
<&lt;Without it, the browser thinks you are starting a tag
>&gt;Without it, the browser thinks you are ending a tag
&&amp;Without it, the browser thinks you are starting an entity
"&quot;When inside a double-quoted attribute value
'&apos;When inside a single-quoted attribute value
 &nbsp;Non-breaking space — prevents line break between words
Entities in action
<p>Use &lt;h1&gt; for the main heading.</p>
<p>The &amp; sign means "and".</p>
<p>Copyright &copy; 2026</p>
Renders as
Use <h1> for the main heading.
The & sign means "and".
Copyright © 2026

The most common mistake: writing &amp; in an HTML file and seeing it display as "&amp;" instead of "&". This happens because the browser decodes entities. When you write &amp; in your HTML, the browser decodes it as &. If you want to show the literal text &amp;, you need &amp;amp;. This is confusing at first — if your entity is not rendering correctly, check whether the browser is decoding it twice.

Void + Entity = Two Things Beginners Don't Know They Don't Know

The #1 support question in every beginner HTML forum: "Why is my image not showing?" — often because they wrote <img></img> with content inside, or forgot the src attribute. And the #2 question: "Why is my whole page in italics?" — because they used a bare & in text and the browser interpreted <i> as an italic tag. These two concepts alone prevent hours of confusion.

Document Structure

Every HTML document follows the same skeleton. Learn this skeleton once and you have the foundation for every page you will ever build:

Minimal valid HTML document
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Page Title</title>
</head>
<body>
  <!-- visible content goes here -->
</body>
</html>

Line-by-Line Breakdown

LinePurpose
<!DOCTYPE html>Tells the browser to render in standards mode. Without this, browsers fall back to "quirks mode" which emulates ancient, broken rendering. Always start with this.
<html lang="en">The root element. The lang attribute is critical for screen readers (pronunciation), search engines (language targeting), and browser translation features.
<meta charset="UTF-8">Declares the character encoding. UTF-8 supports every character in every language. Without this, some browsers will guess the encoding and display garbled text.
<meta name="viewport" ...>Controls how the page scales on mobile devices. This line is why your page does not render as a zoomed-out desktop version on phones. Always include it.
<title>Page Title</title>The only required head element. It appears in the browser tab, search results, and social share previews. Every page needs a unique, descriptive title.
Why the Skeleton Matters

Every HTML page ever written shares this exact structure. It is the first thing a browser looks for and the first thing that breaks when missing. A missing <!DOCTYPE> causes layout bugs that are nearly impossible to diagnose because the browser silently switches rendering modes. Memorize this skeleton.

Semantic Elements

Before HTML5, pages were built with <div> elements and CSS classes. A page looked like: <div class="header">, <div class="nav">, <div class="main">. This worked visually but communicated nothing to browsers, screen readers, or search engines. HTML5 introduced semantic elements that encode meaning directly:

ElementMeaningReplaces
<header>Introductory content, logo, site title<div class="header">
<nav>Navigation links<div class="nav">
<main>Primary page content (one per page)<div class="main">
<section>Thematic group of content<div class="section">
<article>Self-contained content (blog post, comment)<div class="post">
<aside>Tangentially related content (sidebar)<div class="sidebar">
<footer>Closing content, copyright, links<div class="footer">

The practical benefit: a screen reader can jump directly to <nav> or <main>. Search engines give more weight to content inside <main>. The page communicates its structure even without CSS. And your code is shorter and clearer.

Semantic vs non-semantic layout
<!-- Non-semantic: structure only visible to humans -->
<div class="header"><h1>My Site</h1></div>
<div class="nav"><a href="/">Home</a></div>
<div class="main"><p>Content...</p></div>
<div class="footer"><p>Copyright 2026</p></div>

<!-- Semantic: structure visible to everyone -->
<header><h1>My Site</h1></header>
<nav><a href="/">Home</a></nav>
<main><p>Content...</p></main>
<footer><p>Copyright 2026</p></footer>
Semantic Is Not Visual

Semantic elements do not change the visual appearance. <nav> does not make a nav bar. <button> does not style a button. They communicate meaning, not appearance. You still need CSS for layout and styling. The gain is in how the page is understood — by browsers, search engines, and assistive technology — not in how it looks.

Why Can't I Put a <div> Inside a <span>?

<div> is a block-level element — it takes up its own line. <span> is an inline element — it flows with text. Putting a block inside an inline is invalid HTML. The browser will "fix" it by closing the <span> before the <div>, then reopening after — which breaks your layout.

The rule: block elements can go inside block or inline elements. Inline elements can only go inside other inline elements. Use <div> for layout sections, <span> for styling text within paragraphs.

Duplicate IDs Break JavaScript and CSS

An id must be unique on a page. Using the same ID twice is invalid HTML and causes unpredictable behavior: document.getElementById('header') returns only the first element, CSS #header styles only the first, and links to #header scroll to the first match.

If you need multiple elements with the same behavior, use class instead — classes can repeat freely. Only use IDs when you need a single, unique reference (like a form label's for attribute).

My Form Does Nothing When I Click Submit

The most common beginner form mistake: no action attribute. Without it, the form submits to the current URL (which usually just reloads the page with no data). Always include:

<form action="/your-endpoint" method="POST">

Also common: the submit button is outside the <form> tag — then clicking it does nothing useful. Keep your <button> inside the <form>.

Forms and Inputs

Forms are the primary way users send data to a server. A form without proper structure confuses users, breaks screen readers, and sends malformed data to the backend.

Form Structure

Accessible form pattern
<form action="/submit" method="POST">
  <fieldset>
    <legend>Contact Information</legend>

    <label for="name">Full Name</label>
    <input type="text" id="name" name="name" required>

    <label for="email">Email Address</label>
    <input type="email" id="email" name="email" required>

    <label for="message">Message</label>
    <textarea id="message" name="message" rows="4"></textarea>

    <button type="submit">Send</button>
  </fieldset>
</form>

Input Types (HTML5)

HTML5 introduced typed input fields that trigger specialized keyboards on mobile and built-in browser validation:

Using the correct input type is free UX improvement. An type="email" field on iPhone shows the @ key on the keyboard. A type="number" field shows a numeric keypad. These are small wins that compound across every user, every day.

Validation: Built-in vs Custom

HTML5 provides built-in validation through attributes like required, minlength, pattern, and type. The browser displays native error messages when validation fails. This is a good start, but:

  • Native error styles vary by browser and cannot be customized consistently
  • Native messages are generic ("Please fill out this field")
  • Screen reader support for native validation is inconsistent

For production forms, use the Constraint Validation API for custom feedback:

Custom validation with Constraint Validation API
const email = document.getElementById('email');
email.addEventListener('blur', () => {
  if (email.validity.typeMismatch) {
    showError(email, 'Please enter a valid email address');
  } else if (email.validity.valueMissing) {
    showError(email, 'Email is required');
  } else {
    clearError(email);
  }
});

GET vs POST

method="GET" appends form data to the URL as query parameters. Use for search forms and data retrieval (bookmarkable, shareable URLs). method="POST" sends data in the request body. Use for mutations: creating, updating, deleting data, or sending sensitive information like passwords.

Accessibility Foundations

Accessibility (a11y) is not a feature — it is a property of your code. A page is either accessible or it is not, and the difference is almost always in the HTML.

WCAG at a Glance

The Web Content Accessibility Guidelines define four principles — summarized as POUR:

  • Perceivable — information must be presentable to users in ways they can sense (text alternatives for images, captions for video, color not the only differentiator)
  • Operable — interface must be usable by everyone (keyboard-only navigation, enough time to read content, no seizure-inducing flashing)
  • Understandable — content and interface must be predictable (clear labels, consistent navigation, helpful error messages)
  • Robust — content must work across current and future user agents (valid HTML, proper ARIA usage)

Practical Checklist

CheckWhy
Every <img> has meaningful alt textScreen readers describe images via alt text. Decorative images should use alt="" (empty) to be ignored.
Every <input> has a matching <label>Labels make inputs clickable (clicking the label focuses the input) and identify the field to screen readers.
Heading hierarchy is sequential (h1 → h2 → h3)Screen reader users navigate by heading levels. Skipping from h1 to h4 breaks the navigation.
Color contrast meets WCAG AA (4.5:1 for text)Low contrast text is invisible to users with visual impairments. Use tools like WebAIM Contrast Checker.
Focus indicators are visibleKeyboard users need to see where they are. Never use outline: none without providing an alternative focus style.

ARIA: When HTML Is Not Enough

ARIA (Accessible Rich Internet Applications) attributes supplement HTML semantics when native elements do not exist. The first rule of ARIA: do not use it if a native HTML element works. A <button> is inherently focusable and clickable. A <div role="button"> is not — you must add tabindex, keyboard handlers, and ARIA state attributes manually.

ARIA: only when needed
<!-- Good: native button, zero ARIA needed -->
<button>Submit</button>

<!-- Bad: div pretending to be a button, lots of ARIA needed -->
<div role="button" tabindex="0" aria-pressed="false"
     onclick="submit()" onkeydown="if(event.key==='Enter')submit()">
  Submit
</div>

SEO Basics

Search engine optimization starts with HTML. Before Google indexes your content, it parses your markup. Clean, semantic HTML is the single best technical SEO investment you can make.

The Meta Tag Trio

Essential meta tags
<meta name="description" content="A clear, 150-160 character description of this page">
<meta name="robots" content="index, follow">
<meta property="og:title" content="Page Title — Site Name">
<meta property="og:description" content="Description shown in social shares">
<meta property="og:image" content="https://yoursite.com/preview.png">
  • description — Appears in search results below the title. Write for humans, not search engines. A good description increases click-through rate, which is a ranking signal.
  • robots — Controls whether search engines index and follow links on this page. Use noindex for admin pages, thank-you pages, or duplicate content.
  • Open Graph (og:) — Controls how the page appears when shared on social media (Facebook, Twitter, Discord, Slack). Without these, social platforms guess the preview image and description — often badly.

Heading Hierarchy

Search engines use headings to understand document structure. Think of it as an outline:

h1: Page title (one per page)
├── h2: Major section
│ ├── h3: Sub-section
│ └── h3: Sub-section
└── h2: Major section
└── h3: Sub-section

Never skip levels (h1 → h4). Never use h1 for styling — there should be exactly one per page. Heading hierarchy is the most commonly broken SEO rule, and also the easiest to fix.

There Is No <h7> — Headings Only Go H1 to H6

Every beginner tries to use <h7> or <h8> at some point. They do not exist. HTML defines only six heading levels: <h1> through <h6>. If you write <h7>, the browser treats it as plain text, not a heading. If you need more than six levels, use <h4> with CSS classes to create sub-sections, or restructure your content.

Structured Data with JSON-LD

JSON-LD (JavaScript Object Notation for Linked Data) is a format for embedding structured data in a <script> tag. It tells search engines explicit facts about your content: "this is a recipe with these ingredients," "this is an event on this date," "this is an article by this author."

JSON-LD example for an article
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "How the Web Works",
  "author": { "@type": "Person", "name": "Author Name" },
  "datePublished": "2026-01-15",
  "description": "A beginner-friendly guide to how the internet works."
}
</script>

JSON-LD does not affect ranking directly, but it enables rich search results (star ratings, recipe cards, event listings) that dramatically improve click-through rate. Use Google's Rich Results Test to verify your structured data.

SEO Is Not Magic

There is no secret trick to ranking first on Google. The fundamentals are: write useful content, structure it with semantic HTML, use descriptive titles and headings, ensure your site loads fast and works on mobile, and get other reputable sites to link to you. That is 90% of SEO. Everything else is marginal gains.

Junior Developer

How a Junior Might Think

  • Uses <div> for everything and only adds classes for styling
  • Writes forms without <label> elements or proper validation
  • Ignores alt text or uses it incorrectly
  • Doesn't think about heading hierarchy or document outline
Senior Developer

How a Senior Thinks

  • Chooses semantic elements based on content meaning, not appearance
  • Builds forms with proper labels, fieldset grouping, and accessible validation
  • Writes meaningful alt text and considers screen reader flow
  • Plans heading structure before writing content
