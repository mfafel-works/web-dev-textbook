Chapter 7

CSS

Layout, styling, responsive design, and the decision-making process behind every property

CSS is the language of visual decisions. Every property is a tradeoff: readability vs brevity, flexibility vs control, consistency vs uniqueness. This chapter teaches you not just what each property does, but when to reach for it and why.

CSS Core Concepts

Three concepts underpin every line of CSS you will ever write. Understanding them first will save you hours of confusion.

CSS Units

Every measurement in CSS needs a unit. The unit tells the browser what kind of measurement you mean. They fall into two categories:

UnitTypeRelative ToBest For
pxAbsoluteOne physical pixel on screenBorders, shadows, anything that should not scale
remRelativeThe root element's font-size (<html>, default 16px)Font sizes and spacing — scales when user changes browser font size
emRelativeThe current element's font-sizePadding or margin that should scale with the element's text size. Be careful — em compounds (a 2em child inside a 1.5em parent is 3x the root).
%RelativeThe parent element's value for that propertyWidths (50% = half the parent), line-height (150% of font-size)
vwViewport1% of the viewport widthFull-width elements, large hero text that scales with screen
vhViewport1% of the viewport heightFull-screen sections ("hero" that fills the screen)
chRelativeThe width of the "0" character in the current fontLimiting line width for readability (max-width: 65ch)
frGrid-specificFraction of available space in a CSS Grid containerGrid columns: 1fr 2fr = second column gets twice the space of the first

The rule of thumb: use rem for font sizes and spacing (it respects user accessibility preferences), use px for borders and fine details, use % or vw/vh for layout widths. Avoid em for font sizes unless you understand compounding.

Specificity and the Cascade

When two CSS rules conflict, the browser must decide which one wins. It uses a three-tier priority system called the cascade:

1. Origin & importance (user's !important > author's !important > author normal > browser default)
2. Specificity (inline style > ID > class/attribute/pseudo-class > element/pseudo-element)
3. Source order (if specificity is equal, the last rule wins)

Specificity is calculated as a four-part score: (inline, IDs, classes, elements). Each selector type adds to its column:

SelectorExampleInlineIDsClassesElements
Elementp0001
Class.highlight0010
ID#header0100
Inlinestyle="color: red"1000
Combined#sidebar .widget p0111

One ID beats any number of classes. One class beats any number of elements. Inline styles beat everything except !important.

The !important Trap

!important overrides the entire cascade, forcing a property to win regardless of specificity:

Why !important is dangerous
.button { color: blue !important; }
#submit { color: red; }  /* Loses — !important wins */

The problem: if you use !important to fix one style conflict, the next developer cannot override it without adding another !important. This creates an arms race. Never use !important in application code. The only acceptable uses are in utility classes (like a Tailwind helper) or user stylesheets. Fix specificity by making your selector more specific instead.

CSS Specificity Wars

The specificity calculation seems simple on paper, but real-world selectors create surprises. #header .nav a (0,1,1,1) beats .nav .item a (0,0,2,1) because one ID outweighs any number of classes. A class on a deeply nested element loses to an ID on a distant parent. The worst trap: .class:not(.something) — the :not() pseudo-class contributes the highest specificity of its argument, so :not(#id) has an ID's worth of specificity even though it negates it. If two selectors target the same element and you cannot figure out why one wins, paste them into a specificity calculator — the answer is never intuition.

Inheritance

Some CSS properties are automatically passed from a parent to its children. If you set color: blue on a <div>, all the text inside it (including nested <p>, <span>, and <a> elements) also becomes blue — unless something overrides it.

Inherits automaticallyDoes NOT inherit
color, font-family, font-size, font-weight, line-height, text-align, visibilitymargin, padding, border, width, height, background, display, position, overflow

You can force inheritance with the inherit keyword: border: inherit makes an element use its parent's border. You can prevent it with initial (resets to browser default) or unset (acts as inherit for inheritable properties, initial for others).

Inheritance in practice
/* Set on parent — children inherit */
.card { color: #333; font-family: Georgia, serif; line-height: 1.6; }

/* Child overrides only what it needs */
.card h2 { font-size: 1.5rem; }     /* inherits color, font-family, line-height */
.card p  { font-size: 1rem; }        /* inherits color, font-family, line-height */
.card a  { color: blue; }            /* overrides color only */
Why These Three Concepts Matter

Units, specificity, and inheritance are the most common source of "CSS doesn't do what I expect" bugs. Why is my font 24px when I set it to 16px? — somewhere an em is compounding. Why is my button still blue when I wrote color: red? — a more specific selector (or !important) is winning. Why does my div have margin when I didn't set one? — margins are not inherited, but margin-collapse between siblings can look like it. Knowing these three rules eliminates whole categories of bugs.

The Box Model

Every element on a web page is a rectangle. The box model defines how that rectangle's size is calculated. It consists of four layers, from inside out:

LayerPropertyWhat it controls
Contentwidth / heightThe actual content area (text, image, child elements)
PaddingpaddingSpace between content and border. Transparent. Used for breathing room.
BorderborderVisual edge of the element. Can be styled (solid, dashed), colored, and rounded.
MarginmarginSpace outside the border. Creates gaps between elements.

content-box vs border-box

This is the single most impactful CSS decision you will make. The box-sizing property changes how width and height are calculated:

box-sizing comparison
/* content-box (default) — width excludes padding and border */
.box { width: 200px; padding: 20px; border: 2px solid; }
/* Total width: 200 + 20 + 20 + 2 + 2 = 244px */

/* border-box — width INCLUDES padding and border */
.box { box-sizing: border-box; width: 200px; padding: 20px; border: 2px solid; }
/* Total width: 200px (padding and border squeezed inside) */

Always use box-sizing: border-box on every element. It makes sizing predictable — a 50% width is always 50% of the parent, regardless of padding or border. Apply it globally:

Global reset (industry standard)
*, *::before, *::after {
  box-sizing: border-box;
}

Margin Collapsing

Vertical margins collapse: when two block elements are stacked, the larger margin wins and the smaller one is ignored. This is not a bug — it prevents double-spacing between elements. Horizontal margins never collapse.

Element A has margin-bottom: 30px. Element B has margin-top: 20px. The gap between them is 30px (the larger wins), not 50px.

Margin collapsing catches every developer at least once. The fix is usually simple: use padding on the parent instead of margin on the child, or use flexbox/gap which does not collapse.

Z-Index and Stacking Context

Z-index controls which element appears on top — but only within the same stacking context. A new stacking context is created by:

  • position: relative/absolute/fixed/sticky with z-index set
  • opacity less than 1
  • transform, filter, or clip-path
  • isolation: isolate

The most common z-index bug: setting z-index: 9999 on a modal, but the modal is inside a parent with transform and z-index: 1, creating a new stacking context. The modal's 9999 only applies within that parent — it is still behind elements in the root context. The fix: move the modal to the root of the DOM or use isolation: isolate deliberately.

Flexbox Deep Dive

Flexbox solves one-dimensional layout: distributing items along a row or column. It replaces floats, inline-block hacks, and table-based layouts for almost every case.

The Two Axes

main axis = flex-direction (row by default) → controls horizontal distribution
cross axis = perpendicular to main → controls alignment
Essential flex properties
.container {
  display: flex;
  justify-content: center;       /* main axis: start, center, end, space-between, space-around, space-evenly */
  align-items: center;           /* cross axis: stretch, center, start, end, baseline */
  gap: 1rem;                     /* spacing between items (modern, no collapsing) */
  flex-wrap: wrap;               /* allow items to wrap to next line */
}

.item {
  flex: 1;                       /* shorthand: flex-grow: 1, flex-shrink: 1, flex-basis: 0 */
  /* flex: 0 1 auto; */          /* default: don't grow, shrink if needed, basis is content width */
  /* flex: 0 0 200px; */         /* fixed width: don't grow, don't shrink, width 200px */
}

Common Flexbox Patterns

  • Centering: display: flex; justify-content: center; align-items: center; — solves the hardest CSS problem forever
  • Equal-height columns: Flex items stretch to the height of the tallest sibling by default (no hacks needed)
  • Sticky footer: Set flex: 1 on the main content area, footer stays at bottom
  • Auto margins: margin-left: auto on a flex item pushes it to the right — useful for nav bars with a logo on the left and links on the right
Holy grail: sticky footer with flexbox
body {
  display: flex;
  flex-direction: column;
  min-height: 100vh;
}
main { flex: 1; }  /* pushes footer down */
footer { /* stays at bottom */ }
Flexbox vs Grid Decision

Use Flexbox when content flows in one direction and you want it to distribute naturally. Use Grid when you need explicit rows AND columns control. The rough rule: flexbox is for components and small-scale layouts. Grid is for page-level layout. But they work together — a Grid cell can contain a Flexbox row, and vice versa.

CSS Grid

CSS Grid is a two-dimensional layout system. Where Flexbox excels at distributing items along one axis, Grid excels at placing items in rows AND columns simultaneously.

Basic Grid Setup

Simple page shell with Grid
.layout {
  display: grid;
  grid-template-columns: 250px 1fr;   /* sidebar 250px, main content fills rest */
  grid-template-rows: auto 1fr auto;   /* header auto-height, main fills, footer auto-height */
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  min-height: 100vh;
  gap: 1rem;
}

header { grid-area: header; }
aside  { grid-area: sidebar; }
main   { grid-area: main; }
footer { grid-area: footer; }

The fr Unit

fr stands for "fraction" of available space. grid-template-columns: 1fr 2fr means the second column gets twice as much space as the first. Unlike percentages, fr works with gap and does not overflow — it distributes leftover space after fixed columns are calculated.

auto-fit vs auto-fill

These are used with repeat() for responsive grids without media queries:

Auto-filling card grid
.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
  gap: 1.5rem;
}

auto-fit collapses empty tracks (good for sparse content). auto-fill preserves track sizes even if empty (good for consistent sizing). minmax(280px, 1fr) says "each column is at least 280px, but stretch equally if there is room." This single line creates a fully responsive grid with no breakpoints.

Don't Grid Everything

Grid is powerful but can be overkill. A simple nav bar with three links is easier with Flexbox. A button row with "Save" and "Cancel" is Flexbox territory. Reserve Grid for when you need both row and column control — page layouts, card grids, dashboards, forms with complex alignment.

"How Do I Center a Div?" — The #1 Beginner Question

Centering is the most asked CSS question. The answer depends on what you're centering:

  • Inline text inside a block: text-align: center on the parent
  • A block inside a block horizontally: margin: 0 auto (requires width!)
  • A block inside a block vertically + horizontally: Flexbox — display: flex; justify-content: center; align-items: center;
  • An element of unknown size: Flexbox or Grid — place-items: center

The old way: absolute positioning + transforms. The modern way: Flexbox or Grid. Do not use text-align: center on a block element — it does nothing.

Why Does Adding Padding Make My Element Wider?

By default, CSS adds padding on top of the width. If you write width: 100%; padding: 20px;, the element becomes 100% + 40px wide and breaks out of its container. This trips up every beginner.

The fix: box-sizing: border-box; — this makes padding count toward the width instead of adding to it. Add this to your CSS reset:

* { box-sizing: border-box; }

Every modern project starts with this line. Without it, building responsive layouts is painful.

display: none vs visibility: hidden — Both Hide, But Differently

Two ways to hide elements, but they behave very differently:

  • display: none — Removes the element entirely. It takes no space. The page layout acts as if the element doesn't exist.
  • visibility: hidden — Hides the element visually, but it still takes up its original space. The layout acts like an invisible ghost.

Common bug: you hide something with visibility: hidden and wonder why there's a blank gap in your layout. Use display: none unless you specifically need the space preserved.

Responsive Design Strategy

Responsive design means the same HTML looks good on every screen. The strategy is not "add breakpoints until it works" — it is a systematic approach to adaptation.

Mobile-First vs Desktop-First

Mobile-first writes the base styles for small screens, then adds complexity with min-width media queries as the viewport grows. Desktop-first writes for large screens, then overrides with max-width queries for smaller ones.

Mobile-first → base styles are simple (single column), queries add multi-column layouts. Less code, fewer overrides, inherently accessible. Prefer this.
Mobile-first media queries
/* Base: single column, works on any screen */
.content { display: block; }

/* Tablet: two columns */
@media (min-width: 768px) {
  .content { display: grid; grid-template-columns: 1fr 1fr; }
}

/* Desktop: three columns */
@media (min-width: 1024px) {
  .content { grid-template-columns: 1fr 1fr 1fr; }
}

Fluid Typography with clamp()

Instead of setting a fixed font size, use clamp() to define a minimum, preferred, and maximum value:

Fluid typography
h1 { font-size: clamp(1.5rem, 4vw, 3rem); }
/* Minimum: 1.5rem, preferred: 4% of viewport width, maximum: 3rem */

This eliminates the need for breakpoint-specific font sizes. The heading grows smoothly with the viewport between the bounds.

Responsive Images

Serve different image sizes based on viewport width using srcset:

Responsive image with srcset
<img src="photo-800.jpg"
     srcset="photo-400.jpg 400w, photo-800.jpg 800w, photo-1200.jpg 1200w"
     sizes="(max-width: 600px) 100vw, (max-width: 1000px) 50vw, 800px"
     alt="Description">

The browser selects the best image size based on viewport width and device pixel ratio. This saves bandwidth and improves load time on mobile without sacrificing quality on desktop.

Breakpoints Should Be Content-Based

Do not use device-specific breakpoints (iPhone, iPad, etc.). Use breakpoints where your content breaks: "at 700px the sidebar gets too narrow" or "at 900px the cards look too stretched." This makes your design future-proof — new device sizes will not break it.

Custom Properties and Theming

CSS custom properties (variables) bring the most important programming concept to CSS: don't repeat yourself. Define a value once, reference it everywhere.

Declaration and Usage

Custom properties — design tokens
:root {
  --color-primary: #2563eb;
  --color-text: #1a1a2e;
  --color-bg: #ffffff;
  --spacing-md: 1rem;
  --spacing-lg: 2rem;
  --font-body: Georgia, serif;
  --radius-sm: 4px;
}

.button {
  background: var(--color-primary);
  padding: var(--spacing-md) var(--spacing-lg);
  border-radius: var(--radius-sm);
}

Now change --color-primary in one place and every button updates. This is the foundation of a design system.

Dark/Light Theme Toggle

Custom properties make runtime theme switching trivial. Define both themes, then toggle with a class or attribute:

Theme system (used in this textbook)
:root {
  --color-bg: #ffffff;
  --color-text: #1a1a2e;
  --color-code-bg: #f4f4f4;
}

[data-theme="dark"] {
  --color-bg: #1a1a2e;
  --color-text: #e4e4e7;
  --color-code-bg: #2a2a3e;
}

body {
  background: var(--color-bg);
  color: var(--color-text);
  transition: background 0.3s, color 0.3s;
}
JavaScript toggle
const toggle = document.querySelector('[data-theme-toggle]');
toggle.addEventListener('click', () => {
  const theme = document.documentElement.getAttribute('data-theme');
  const next = theme === 'dark' ? 'light' : 'dark';
  document.documentElement.setAttribute('data-theme', next);
  localStorage.setItem('theme', next);
});

Fallback Values

Always provide a fallback when using var(), especially if the property might not be defined:

Fallback syntax
.box {
  background: var(--color-maybe-missing, #ccc);  /* uses #ccc if variable is not set */
}

Animations and Transitions

Animation guides attention. A button that changes color instantly feels jarring. A button that transitions smoothly feels polished. The difference is transition.

Transitions

Transitions interpolate between two states — a hover, a class toggle, a property change:

Button hover transition
.button {
  background: blue;
  color: white;
  transition: background 0.2s ease, transform 0.15s ease;
}

.button:hover {
  background: darkblue;
  transform: translateY(-1px);
}

Transitions fire automatically when a property changes. Define only the properties you want to animate and keep durations short (150-300ms for UI elements).

Keyframe Animations

For multi-step animations, use @keyframes:

Fade-in animation
@keyframes fadeIn {
  from { opacity: 0; transform: translateY(10px); }
  to   { opacity: 1; transform: translateY(0); }
}

.hero-title {
  animation: fadeIn 0.6s ease-out forwards;
}

Performance Rule

Animating certain properties causes the browser to recalculate layout on every frame — expensive and janky. Stick to these two properties for smooth 60fps animations:

Animate → transform (translate, scale, rotate) and opacity only.
Avoid animating → width, height, top, left, margin, padding, font-size.

The reason: transform and opacity are handled by the GPU compositor thread. Layout properties trigger reflow on the CPU thread, which is slower and causes visible stuttering.

Accessibility: Reduced Motion

Some users experience nausea or dizziness from motion effects. Always respect the prefers-reduced-motion media query:

Respecting user preferences
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
Junior Developer

  • Uses !important to override styles without understanding specificity
  • Struggles with vertical centering and reaches for hacks
  • Doesn't plan for responsive design — writes desktop-first and scrambles to fix breakpoints
Senior Developer

How a Senior Thinks

  • Chooses layout strategy (flex, grid, or block) based on content and constraints
  • Understands specificity and avoids !important entirely
  • Knows multiple ways to center and picks the robust one
  • Designs mobile-first, uses fluid sizing, and tests early on small screens
