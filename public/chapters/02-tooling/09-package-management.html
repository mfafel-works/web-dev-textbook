Chapter 9

Package Management & npm

Dependencies, lockfiles, semver, scripts, and surviving node_modules

Modern JavaScript development depends on open-source packages — small reusable modules that solve specific problems. A typical project uses hundreds of these packages. npm (Node Package Manager) is the tool that installs, updates, and manages them. Understanding how npm works is essential: it is the source of more confusing errors and hidden disk usage than almost any other tool.

Where Does npm Come From?

npm comes with Node.js. When you install Node.js (from nodejs.org), you get both Node.js (the JavaScript runtime) and npm (the package manager) automatically. There's no separate install. If you have npm, you have Node. If you have Node, you have npm.

To check if you have them: run node --version and npm --version in your terminal. If you get version numbers, you're ready to go.

What Is a Package Manager?

A package manager is a tool that automates installing, updating, and removing third-party code. Without it, you would download libraries manually, save them to a lib/ folder, and manage versions yourself. With npm, one command installs everything you need and records what you installed so it can be reproduced on another machine.

There are several package managers for JavaScript:

ToolUsed ByKey Difference
npmMost projects (ships with Node.js)The default. Largest registry.
yarnLegacy preferenceFaster than npm in 2016. npm caught up in v7+.
pnpmDisk-space-conscious teamsUses hard links — stores packages globally, saves disk space.

This chapter uses npm, but the concepts apply to all three.

The package.json File

Every Node.js project has a package.json file at its root. This is the manifest — it describes your project and lists its dependencies.

Anatomy of package.json
{
  "name": "my-app",                    // project name (must be unique on npm if publishing)
  "version": "1.0.0",                  // your project's current version (semver)
  "description": "A contact form app", // human-readable summary
  "main": "index.js",                  // entry point file
  "scripts": {                         // commands you can run with npm run
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview",
    "test": "vitest"
  },
  "dependencies": {                    // packages your app needs to RUN
    "react": "^18.2.0",
    "react-dom": "^18.2.0"
  },
  "devDependencies": {                 // packages your app needs to DEVELOP
    "vite": "^5.0.0",
    "vitest": "^1.0.0"
  }
}

dependencies vs devDependencies

This distinction matters:

  • dependencies — packages required at runtime. React, Vue, Express, axios. When someone installs your package, these are installed too.
  • devDependencies — packages needed only during development. Testing frameworks, bundlers (Vite, Webpack), linters, type definitions. These are NOT installed when a user installs your package.

Install a dependency: npm install react (adds to dependencies)
Install a dev dependency: npm install -D vitest (adds to devDependencies)

node_modules — The Elephant in the Room

When you run npm install, npm downloads every package listed in package.json and all of their dependencies into a folder called node_modules. A typical React project with a few dependencies pulls in 500-1500 packages. The folder can be 200-500 MB.

Why So Many Packages?

Modern packages are tiny and modular. A package that formats dates depends on a package that parses time zones, which depends on a package that handles locale data. Each of those may have their own dependencies. This is called the dependency tree. It is normal and expected.

Never Commit node_modules

The node_modules folder must NEVER be committed to Git. It is huge, platform-specific, and entirely reproducible from package.json and the lockfile. Always add node_modules/ to your .gitignore. If you accidentally commit it, remove it with git rm -r --cached node_modules.

Why Is node_modules So Slow to Install?

npm resolves the dependency tree (which version of each package satisfies all dependencies), downloads each package, extracts it to the correct location, and runs any install scripts. With hundreds of packages, this takes time. npm ci (see below) is faster because it skips the resolution step.

The Lockfile (package-lock.json)

The package-lock.json file is automatically generated by npm. It records the exact version of every package and every sub-dependency that was installed. This is the most important file you never think about.

Why You Need It

Your package.json might say "react": "^18.2.0" — which means "any React 18.x.x version." Today that installs 18.2.0. Tomorrow it might install 18.3.0, which could have a subtle breaking change. The lockfile pins every package to its exact installed version so that npm install on another machine (or next week) produces identical output.

Lockfile entry (simplified)
{
  "react": {
    "version": "18.2.0",           // exact version installed
    "resolved": "https://...",     // where it was downloaded from
    "integrity": "sha512-...",     // checksum to verify file integrity
    "dependencies": {
      "loose-envify": "^1.1.0"   // React's own dependencies
    }
  }
}
Always Commit package-lock.json

Without the lockfile, different developers (and your CI server) may install different versions of dependencies, causing "works on my machine" bugs. Always commit package-lock.json to Git. Never add it to .gitignore.

Semantic Versioning

Every npm package follows semantic versioning (semver): a three-part number MAJOR.MINOR.PATCH.

PartChange meansExample
MAJORBreaking change — your code may break2.0.0 → 3.0.0 (API changed)
MINORNew feature, backward compatible2.1.0 → 2.2.0 (new function added)
PATCHBug fix, backward compatible2.1.0 → 2.1.1 (bug fixed)

Version Ranges

In package.json, you specify allowed version ranges:

NotationMeaningExample
^1.2.3Compatible with 1.x.x (update to minor/patch)Allows 1.2.3 through 1.999.999
~1.2.3Approximately — patch onlyAllows 1.2.3 through 1.2.999
1.2.3Exact version onlyOnly 1.2.3
*Any version (avoid)Could install 5.0.0 with breaking changes
The ^ Range Is Not Safe

The ^ prefix (default for npm install) allows minor version bumps. In theory, minor versions are backward compatible. In practice, packages occasionally introduce breaking changes in minor versions — accidentally or by poor semver discipline. This is the most common cause of "it broke after npm install." The lockfile protects you from this, but only if it is committed.

npm install vs npm ci

Two commands that look similar but behave very differently:

npm installnpm ci
Use whenLocal development, adding/removing packagesCI/CD pipelines, fresh clones, deployment
LockfileMay update itFollows it exactly — errors if lockfile is missing
node_modulesAdds to existing folderDeletes and reinstalls from scratch
SpeedSlower (resolves dependency tree)Faster (skips resolution, uses lockfile)
DeterministicNo — may install different versionsYes — identical install every time
When to use each
# Local development — add a package
npm install react-router-dom

# CI server — reproduce exactly what's committed
npm ci

# First time cloning a project
npm ci        # faster than npm install, respects lockfile

# Updating all packages to latest within semver range
npm update
Always Use npm ci in CI

In your CI pipeline (GitHub Actions, GitLab CI, etc.), always use npm ci instead of npm install. It is faster, it respects the lockfile (so builds are reproducible), and it fails if the lockfile is out of sync with package.json — catching mistakes early.

npm Scripts

The "scripts" field in package.json lets you define commands that run with npm run <name>. This is how projects standardize their workflows:

Common npm scripts
{
  "scripts": {
    "dev": "vite",                    // start dev server
    "build": "vite build",            // production build
    "preview": "vite preview",        // preview production build
    "test": "vitest",                 // run tests
    "lint": "eslint src/",            // check code quality
    "format": "prettier --write src/", // auto-format code
    "typecheck": "tsc --noEmit"       // check TypeScript types
  }
}

# Run them:
npm run dev
npm run build
npm test          # "test" is a special alias — npm test works without "run"
npm start         # "start" is also a special alias

Pre and Post Hooks

npm automatically runs scripts with pre and post prefixes. If you define prebuild, it runs before build. If you define postbuild, it runs after:

Script lifecycle hooks
{
  "scripts": {
    "prebuild": "rm -rf dist",         // clean output folder before building
    "build": "vite build",
    "postbuild": "cp _headers dist/"   // copy deployment config after building
  }
}

// npm run build runs: prebuild → build → postbuild

npx — Run Without Installing

npx (bundled with npm) lets you run a package without permanently installing it. It downloads the package to a temporary cache, runs it, and discards it:

npx in action
# Create a new React project without installing create-react-app globally
npx create-react-app my-app

# Run a one-off command
npx serve .                    # start a static file server
npx jsonlint config.json       # validate JSON without installing jsonlint
npx vitest                     # run tests without installing vitest globally

Use npx for one-off commands, scaffolding tools, and rarely-used packages. If you use a tool daily (like Vite or ESLint), install it locally with npm install -D so it is available to your scripts.

Junior Developer

How a Junior Might Think

  • Does not understand node_modules — deletes and reinstalls when something breaks
  • Ignores package-lock.json — may even add it to .gitignore
  • Uses npm install everywhere, including CI
  • Does not understand semantic versioning — sees ^1.2.3 as a mystery
  • Installs packages globally with -g when they should be local
  • Does not use or write npm scripts — runs tools by typing full paths
Senior Developer

How a Senior Thinks

  • Understands that node_modules is reproducible — deletes it freely and reinstalls with npm ci
  • Always commits package-lock.json and uses npm ci in CI
  • Uses exact version pins for critical dependencies, ^ for trusted ones
  • Keeps dependencies minimal — audits them regularly with npm audit
  • Writes npm scripts for every repeated task — standardizes the team workflow
  • Knows the difference between npm install and npm ci and uses each correctly
