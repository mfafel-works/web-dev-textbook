Variables and Types

JavaScript has dynamic types — a variable can hold a string, then a number, then an object. This flexibility is powerful but requires discipline. Understanding types is the foundation of avoiding bugs.

Primitive Types

TypeExampleNotes
string"hello", 'world', `template`Text. Three quoting options (see Strings section).
number42, 3.14, NaN, InfinityAll numbers are floating-point (no integer type). NaN is the result of invalid math.
booleantrue, falseUsed in conditions, comparisons, and toggles.
nullnullExplicitly "no value." Must be assigned.
undefinedlet x; → x is undefinedDefault value of unassigned variables. Different from null.
bigint9007199254740991nFor integers larger than Number.MAX_SAFE_INTEGER. Rarely needed.

let, const, and var

KeywordScopeCan reassign?Hoisted?Use?
constBlockNoYes (but not initialized)Default choice. Use for everything that does not need reassignment.
letBlockYesYes (but not initialized)Use when you need to reassign (loop counters, accumulating values).
varFunctionYesYes (initialized as undefined)Never use. Function scoping causes bugs that block scoping prevents.
const vs let — when to use each
const name = 'Alice';        // never reassigned — use const
const items = [1, 2, 3];     // array content CAN change, reference stays the same — const is fine
items.push(4);               // allowed

let count = 0;               // will be reassigned — use let
for (let i = 0; i < 10; i++) { // loop counter — use let
  count += i;
}

Truthy and Falsy

In conditions, JavaScript coerces values to boolean. Only six values are falsy — everything else is truthy:

Falsy: false, 0, "" (empty string), null, undefined, NaN
Truthy: everything else including "false" (non-empty string), [] (empty array), {} (empty object)

This is why if (value) works for checking if a string is non-empty, but does NOT work for checking if a number is 0 (0 is falsy but may be a valid value).

== vs ===

== compares values after type coercion. === compares values AND types without coercion (strict equality). Always use === unless you explicitly need coercion (rare).

== vs === surprises
0 == ''      // true  — because '' coerces to 0
0 === ''     // false — different types
false == '0' // true  — because both coerce to 0
false === '0' // false — different types
null == undefined  // true  — special case
null === undefined // false — different types

Functions and Scope

Functions in JavaScript are first-class citizens: they can be assigned to variables, passed as arguments, returned from other functions, and stored in data structures. This is the key to many JS patterns.

Function Forms

Three ways to write a function
// Declaration — hoisted, can be called before definition
function add(a, b) { return a + b; }

// Expression — not hoisted, assigned to a variable
const subtract = function(a, b) { return a - b; };

// Arrow — concise, inherits 'this' from parent scope
const multiply = (a, b) => a + b;  // implicit return for single-expression functions

// Arrow with body block
const divide = (a, b) => {
  if (b === 0) throw new Error('Cannot divide by zero');
  return a / b;
};

The this Keyword

this is the single most confusing concept in JavaScript. The rule: this refers to how a function is called, not where it is defined.

  • Regular function: this is the object the method is called on (obj.method() → this is obj)
  • Standalone call: this is undefined in strict mode, window in sloppy mode
  • Arrow function: this is inherited from the enclosing scope (lexical this) — it does not have its own
  • Event handler: this is the element the handler is attached to
this — the arrow function advantage
button.addEventListener('click', function() {
  console.log(this);  // the button element
});

button.addEventListener('click', () => {
  console.log(this);  // the surrounding scope (e.g., window or class instance)
});

Closures

A closure is a function that remembers the variables from its defining scope even after that scope has exited. This is how data privacy works in JavaScript:

Closure — counter with private state
function createCounter() {
  let count = 0;  // private — cannot be accessed from outside
  return function() {
    count++;
    return count;
  };
}

const counter = createCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
// count is NOT accessible here — it is closed over
Why Closures Matter

Closures are everywhere in JavaScript: event handlers, callbacks, React hooks, debounce functions, module patterns. Understanding closures means understanding how JavaScript preserves state across time — a function defined now that runs later still has access to the variables that existed when it was created.

Scope Rules

Scope determines where variables are visible. Three levels:

  • Global — declared outside any function. Visible everywhere. Avoid — pollutes the namespace.
  • Function — declared with var inside a function. Visible anywhere in that function.
  • Block — declared with let or const inside {}. Visible only within that block.
Block scope vs function scope
if (true) {
  var x = 1;   // function-scoped — leaks out
  let y = 2;   // block-scoped — stays inside
}
console.log(x); // 1
console.log(y); // ReferenceError: y is not defined

This is why var causes bugs: a variable declared inside an if block is visible outside it. let and const fix this.

The DOM and Events

The Document Object Model is the bridge between JavaScript and HTML. You select elements, listen to what users do, and respond. The single most important skill in DOM work is choosing the right event for the right intent.

Selecting Elements

Before you can attach an event, you need the element:

Query selectors
const btn = document.querySelector('.submit-btn');   // first match
const allBtns = document.querySelectorAll('.btn');    // NodeList
const el = document.getElementById('header');         // fast, but less flexible
const els = document.getElementsByClassName('card');  // live collection, rare need

Prefer querySelector and querySelectorAll for almost everything. They accept any CSS selector and return static (non-live) collections, avoiding subtle bugs that live collections cause.

Three Ways to Attach Events

There are three registration methods. Only one is correct in production:

MethodExampleProblem
HTML attribute<button onclick="handle()">Mixes behavior into markup. Limited to one handler. Global scope pollution.
DOM propertybtn.onclick = handleOnly one handler at a time — assigning a second overwrites the first.
addEventListenerbtn.addEventListener('click', handle)Multiple handlers, capture/bubble control, removeEventListener works. Use this.
addEventListener — the right way
const btn = document.querySelector('.submit-btn');

// Attach multiple handlers (both will fire)
btn.addEventListener('click', handleClick);
btn.addEventListener('click', trackAnalytics);

// Remove a specific handler later
btn.removeEventListener('click', handleClick);

// Options: run once, use capture phase
btn.addEventListener('click', handleOnce, { once: true });

Event Type Decision Tree

The most common mistake junior developers make: using click for everything. The right event makes your app feel right. Run through this tree every time you attach a listener:

User interaction? ├── Click/tap? │ ├── Full action (button, link, toggle)? → click │ ├── Press started (drag, hold)? → mousedown │ └── Press released (drop)? → mouseup ├── Keyboard? │ ├── Any key press (game, shortcut)? → keydown │ ├── Key released (character input)? → keyup │ └── Typed a character (deprecated, avoid)? → keypress (don't use) ├── Form field? │ ├── Every keystroke/paste (live preview)? → input │ ├── Value committed (user left field)? → change │ ├── Field lost focus (validate on exit)? → blur │ └── Field gained focus (show help)? → focus ├── Form submission? │ └── Always on the <form>, not the button → submit ├── Page loaded? │ ├── HTML parsed, DOM ready? → DOMContentLoaded │ └── All resources loaded (images, CSS)? → load ├── Scroll / resize? │ └── Performance-sensitive — always debounce → scroll, resize └── Mobile touch? ├── Touch began → touchstart ├── Touch moved (swipe)? → touchmove └── Touch ended → touchend

Key Distinctions People Get Wrong

blur vs change

blur fires when an element loses focus — regardless of whether the value changed. change fires only when the value changed AND focus is lost. Use blur for inline validation (check every field exit). Use change when you only care about actual edits.

blur vs change
// Validate on every field exit — catches empty fields
emailInput.addEventListener('blur', () => {
  if (!emailInput.value) showError('Email is required');
});

// Only react when user actually changed something
emailInput.addEventListener('change', () => {
  saveDraft(emailInput.value);
});

input vs change

input fires on every keystroke, paste, or input event — great for live previews, character counts, and search-as-you-type. change fires only when the user commits (blur + value changed). Warning: input without debouncing will fire dozens of events per second.

input with debouncing
let debounceTimer;
searchInput.addEventListener('input', () => {
  clearTimeout(debounceTimer);
  debounceTimer = setTimeout(() => {
    fetchSearchResults(searchInput.value);  // expensive call
  }, 300);  // wait 300ms after user stops typing
});

click vs submit

Always listen for submit on the <form> element, not click on the submit button. The submit event also fires when the user presses Enter in any input field. A button click won't catch that.

Form submission — correct pattern
const form = document.querySelector('#contact-form');

form.addEventListener('submit', (e) => {
  e.preventDefault();  // stop page reload

  if (!validate(form)) return;

  submitBtn.disabled = true;  // prevent double-submit
  submitFormData(form);
});

mousedown vs click

click fires after a full press+release cycle. mousedown fires immediately on press. Use mousedown for drag initiation, drawing, or any interaction where response time matters more than completion.

Event Delegation

Instead of attaching a listener to every matching element, attach one to a parent and use event bubbling to catch events from children. This works for dynamically added elements and is more performant.

Event delegation pattern
// Instead of:
document.querySelectorAll('.list-item').forEach(el => {
  el.addEventListener('click', handleItem);
});

// Do this:
document.querySelector('.item-list').addEventListener('click', (e) => {
  const item = e.target.closest('.list-item');
  if (item) handleItem(item);
});

Bubbling vs Capturing

Events flow in three phases: capture (window → target), target, bubble (target → window). By default, listeners fire during the bubble phase. Pass { capture: true } to intercept during capture. Know this exists — you'll need it for things like stopping event propagation from specific layers.

Why Event Choice Matters

The right event makes your app feel right. Using change for a live search makes it feel sluggish (only updates when the user leaves the field). Using input without debouncing fires 50+ requests per second. Using click on a submit button misses Enter-key submissions. A senior engineer doesn't just know event names — they match event strategy to user intent.

Asynchronous JavaScript

JavaScript runs on a single thread — it can do only one thing at a time. But the browser (Node.js, etc.) is multi-threaded. Asynchronous patterns let you perform slow operations (network requests, file reads, timers) without blocking the entire program.

The Event Loop

Understanding the event loop is the key to understanding all async JavaScript:

Call Stack (one thing at a time) → Web APIs (timers, fetch, DOM) → Task Queue (callbacks, events) → Microtask Queue (Promises, MutationObserver)
  1. Execute code on the call stack (synchronous)
  2. When an async operation completes, its callback goes to the task queue
  3. Promise resolutions go to the microtask queue (higher priority)
  4. When the call stack is empty, process ALL microtasks first
  5. Then process one task from the task queue
  6. Repeat

This is why setTimeout(callback, 0) does not run immediately — it still has to wait for the call stack and microtask queue to empty. This is also why promise resolutions always run before timer callbacks.

Callbacks → Promises → async/await

The evolution of async patterns in JavaScript:

Three eras of async code
// 1. Callbacks (old — avoid)
getUser(id, (err, user) => {
  if (err) handleError(err);
  else getPosts(user.id, (err, posts) => {
    if (err) handleError(err);
    else render(posts);  // callback hell
  });
});

// 2. Promises (better — chainable)
getUser(id)
  .then(user => getPosts(user.id))
  .then(posts => render(posts))
  .catch(err => handleError(err));

// 3. async/await (best — reads like sync code)
async function loadUserPosts(id) {
  try {
    const user = await getUser(id);
    const posts = await getPosts(user.id);
    render(posts);
  } catch (err) {
    handleError(err);
  }
}

Promise Fundamentals

A Promise is an object representing a value that may not be available yet. It has three states: pending, fulfilled, or rejected.

Creating and consuming promises
// Creating a promise
const wait = (ms) => new Promise(resolve => setTimeout(resolve, ms));

await wait(1000); // pauses for 1 second

// Promise.all — run multiple promises in parallel
const [user, posts] = await Promise.all([
  fetchUser(id),
  fetchPosts(id)
]);

Promise.all is critical for performance — it runs all promises concurrently. The await version with sequential awaits runs them one at a time. If the operations are independent, use Promise.all.

Unhandled Rejections Crash Node.js

An unhandled promise rejection will crash Node.js (since version 15) and log warnings in browsers. Every async function should have a try/catch. Every .then() chain should have a .catch(). The most common cause: forgetting await — if you call fetchData() without await and it rejects, the rejection is unhandled because no try/catch wraps the async call. Node will print UnhandledPromiseRejection and terminate. Add a global handler with process.on('unhandledRejection', ...) as a safety net, but fix the root cause.

Why Does "1" == 1 Return True? The == vs === Trap

JavaScript has two equality operators that behave very differently:

  • == (loose equality): converts types before comparing. "1" == 1 is true. "" == 0 is true. null == undefined is true.
  • === (strict equality): no type conversion. "1" === 1 is false.

The loose == does type coercion — it was a mistake in JavaScript's original design. Always use === unless you have a specific reason not to. It prevents subtle bugs like comparing user input "42" to the number 42 and getting a false positive.

I Declared x with let Twice and It Broke — But Assigning Without let Works?

Re-declaring a variable with let in the same scope throws an error:

let x = 1;
let x = 2;  // SyntaxError: Identifier 'x' has already been declared

But reassigning without let works fine:

let x = 1;
x = 2;  // This is allowed

Why? let x = ... declares AND assigns. x = ... only assigns. Using let twice in the same scope is a syntax error — it's trying to create two variables with the same name. Using var instead would allow re-declaration (which is why var is dangerous and you should never use it).

Cannot Read Property of undefined — The #1 Runtime Error

This error stops every JavaScript beginner:

const user = { name: "Alice" };
console.log(user.email);        // undefined (property doesn't exist)
console.log(user.email.city);   // TypeError: Cannot read property 'city' of undefined

You can't access a property of undefined or null. Even if user.email is undefined, trying to go deeper crashes. Fix with optional chaining:

console.log(user.email?.city);   // undefined (safe, no error)

Always check if an object exists before accessing its properties, or use ?. (optional chaining) to safely traverse uncertain paths.

Fetch and APIs

The fetch API is the modern, browser-native way to make HTTP requests. It returns a Promise and works seamlessly with async/await.

Basic GET Request

Fetching and displaying data
async function loadUsers() {
  try {
    const response = await fetch('/api/users');

    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }

    const users = await response.json();
    renderUserList(users);
  } catch (err) {
    showError('Failed to load users. Please try again.');
    console.error(err);  // always log for debugging
  }
}

Two critical details: response.ok is false for any non-2xx status (including 404, 500). And response.json() is itself async — it returns a Promise because the body arrives as a stream.

POST Request with JSON Body

Submitting data with fetch
async function submitForm(data) {
  try {
    const response = await fetch('/api/contact', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(data)
    });

    if (!response.ok) {
      const error = await response.json();
      throw new Error(error.message || 'Submission failed');
    }

    return await response.json();
  } catch (err) {
    // Network errors (offline, DNS failure) and HTTP errors both land here
    throw err;
  }
}

Aborting Requests

When a user navigates away or a search field changes, you should cancel in-flight requests to avoid race conditions and wasted bandwidth:

AbortController pattern
let currentRequest;

async function search(query) {
  // Cancel previous request
  if (currentRequest) currentRequest.abort();

  const controller = new AbortController();
  currentRequest = controller;

  try {
    const response = await fetch(`/api/search?q=${query}`, {
      signal: controller.signal
    });
    const results = await response.json();
    displayResults(results);
  } catch (err) {
    if (err.name === 'AbortError') return; // expected — ignore
    showError('Search failed');
  }
}
CORS Explained Simply

CORS (Cross-Origin Resource Sharing) is a browser security mechanism. When your frontend at yoursite.com tries to fetch from api.othersite.com, the browser sends a preflight OPTIONS request and checks the response headers. The server must explicitly allow your origin with Access-Control-Allow-Origin: https://yoursite.com. This is not a backend-to-backend restriction — only browsers enforce it.

Strings: Quotes vs Template Literals

JavaScript gives you three ways to write strings. Understanding when to use each is a deceptively important skill — it affects readability, maintainability, and how naturally your code communicates intent.

The Three Options

Each syntax is valid for creating a string. The difference is what they enable:

SyntaxNameBest Use Case
'...'Single quotesSimple strings by convention. Most JS style guides (Airbnb, Standard, Prettier) prefer these.
"..."Double quotesJSON (required). HTML attributes. Some language ecosystems (PHP, Java) favor these — consistency across a stack can matter.
`...`Template literalsInterpolation, multi-line strings, and tagged templates. The modern choice for dynamic strings.

Decision Tree

When you need to write a string, run through these questions:

Need variable/expression interpolation? ├── YES → Template literals `Hello, ${name}!` └── NO ├── Multi-line string? │ ├── YES → Template literals │ └── NO │ ├── Writing JSON? → Double quotes " (JSON spec) │ ├── Contains single quote '? → Double quotes " (no escaping) │ ├── Contains double quote "? → Single quotes ' (no escaping) │ └── None of the above? → Single quotes ' (convention)

Interpolation: + vs Template Literals

This is where template literals earn their keep. Compare these two patterns that produce the same output:

Concatenation (old style)
const name = 'Alice';
const role = 'developer';
const message = 'Hello, ' + name + '! You are a ' + role + '.';
// Result: "Hello, Alice! You are a developer."
Template literal (modern)
const name = 'Alice';
const role = 'developer';
const message = `Hello, ${name}! You are a ${role}.`;
// Result: "Hello, Alice! You are a developer."

The template literal is shorter, more readable, and makes the structure of the string visible at a glance. The + version forces you to mentally parse quoted fragments and operators. This is not about performance — there is no meaningful difference. It is about human readability.

Multi-line Strings

Multi-line comparison
// With concatenation:
const address = '123 Main St.\n' +
  'Suite 100\n' +
  'New York, NY 10001';

// With template literals:
const address = `123 Main St.
Suite 100
New York, NY 10001`;

The template literal preserves actual line breaks in the source code. No \n, no +. But be aware: those line breaks become literal newlines in the string, including indentation.

Escaping: Choosing the Right Quote

A practical skill: pick the quote that minimizes escaping.

Escaping scenarios
// Contains apostrophe — use double quotes to avoid backslash:
const sentence = "It's a beautiful day";     // clean
const sentence = 'It\'s a beautiful day';     // noisy

// Contains double quote — use single quotes:
const html = '<button class="btn">Click</button>';  // clean
const html = "<button class=\"btn\">Click</button>";  // noisy

// Contains both — template literal saves you:
const msg = `He said "It's done" and left.`;  // clean with backticks
Why This Works

The goal is not to have a favorite quote type. The goal is to recognize the context you're in and choose the syntax that produces the cleanest, least ambiguous result. A senior engineer doesn't "prefer" one quote — they assess the string contents and pick accordingly.

Tagged Templates (Advanced)

Template literals have a less common but powerful feature: you can pass them through a "tag" function that processes the string and interpolated values. Libraries like styled-components in React use this pattern. This is an intermediate-to-advanced concept — understand that it exists, return to it when you need it.

Modern ES6+ Features

ES6 (ES2015) transformed JavaScript. These features are not optional — they are the standard way to write modern JS. If you are not using them, your code is harder to read and more error-prone than it needs to be.

Destructuring

Extract values from objects and arrays in a single, readable statement:

Object and array destructuring
// Object destructuring
const user = { name: 'Alice', email: 'alice@example.com', role: 'admin' };
const { name, email } = user;
console.log(name);  // 'Alice'

// Renaming during destructuring
const { name: fullName, role: userRole } = user;

// Array destructuring
const [first, second, ...rest] = [1, 2, 3, 4, 5];
console.log(first);  // 1
console.log(rest);   // [3, 4, 5]

// Function parameter destructuring
function greet({ name, role }) {
  return `Hello ${name} (${role})`;
}

Modules: import / export

Modules let you split code into separate files with explicit dependencies:

Module pattern
// utils.js
export function formatDate(date) { ... }
export const API_URL = 'https://api.example.com';

// Or export as default (one per file)
export default function logError(err) { ... }

// app.js
import logError, { formatDate, API_URL } from './utils.js';

Modules are the reason global scope pollution is no longer an issue. Everything is explicitly imported. If a file needs something, it imports it. If it does not, it is not available.

Array Methods: map, filter, reduce

These three methods replace most for loops. They express intent — you can tell at a glance what the code does:

map, filter, reduce in action
const numbers = [1, 2, 3, 4, 5];

// map — transform every element
const doubled = numbers.map(n => n * 2);
// [2, 4, 6, 8, 10]

// filter — keep elements that pass a test
const evens = numbers.filter(n => n % 2 === 0);
// [2, 4]

// reduce — accumulate a single value
const sum = numbers.reduce((acc, n) => acc + n, 0);
// 15

// Chained: get total of even numbers doubled
const total = numbers
  .filter(n => n % 2 === 0)
  .map(n => n * 2)
  .reduce((acc, n) => acc + n, 0);
// 12 (2*2 + 4*2)

Compare the chained version to the equivalent for loop — the chain communicates what it does (filter, then map, then sum) rather than how (create variable, loop, check condition, transform, accumulate).

Optional Chaining and Nullish Coalescing

Two modern operators that prevent the most common runtime error: "Cannot read property of undefined":

?. and ?? — safe access and default values
const user = { profile: { name: 'Alice' } };

// Optional chaining — safe nested access
const city = user?.profile?.address?.city;
// undefined instead of TypeError

// Nullish coalescing — default for null/undefined only
const name = user?.profile?.name ?? 'Anonymous';
// 'Alice' if name exists, 'Anonymous' if null/undefined

// Compare with || — treats 0 and '' as falsy
const count = 0;
console.log(count || 10);   // 10  (0 is falsy — wrong!)
console.log(count ?? 10);   // 0   (0 is not null/undefined — correct)

The critical difference: ?? only replaces null and undefined. || replaces any falsy value (0, "", false). Use ?? when 0 or "" are valid values.

Browser DevTools and Debugging

Writing code is only half the job — debugging is where you earn your reputation. Browser DevTools are your most important tool as a frontend developer. Learning to use them systematically separates engineers who fix bugs from those who chase them.

Opening DevTools

Every major browser has built-in developer tools. The shortcuts are muscle memory:

  • F12 or Ctrl+Shift+I (Windows/Linux)
  • Cmd+Option+I (Mac)
  • Right-click → Inspect on any element

The Console Panel

The console is your first line of defense. Use it for logging, testing expressions, and inspecting errors:

Console logging levels
console.log('General info');        // standard log
console.warn('Heads up');          // yellow warning
console.error('Something broke');  // red error with stack trace
console.table([{name: 'Alice'}, {name: 'Bob'}]);  // table format for arrays/objects
console.time('timer');             // start a timer
console.timeEnd('timer');          // end timer, prints duration

The console also shows runtime errors with a stack trace. Click the file link in the error to jump directly to the source line.

console.log is Not Debugging

Logging is a useful first step, but it's passive — you only see what you thought to log. Breakpoints are active: they stop execution and let you inspect everything. Use console.log for quick checks, breakpoints for real debugging.

The Sources Panel

The Sources panel (called "Debugger" in some browsers) is where breakpoints live. It shows all loaded files and lets you set breakpoints, step through code, and inspect variables.

Setting Breakpoints

Click a line number in the Sources panel to set a breakpoint. Execution pauses on that line, and you can:

  • Hover over variables to see their current value
  • View the full scope tree (Local, Closure, Global) in the right panel
  • Watch specific expressions update as you step through
  • View the call stack to see how you got to this point

Conditional Breakpoints

Right-click a line number and select "Add conditional breakpoint". Enter an expression — execution only pauses when the expression is true. Use this to skip unrelated iterations in a loop:

Conditional breakpoint scenario
// Instead of adding console.log inside a 1000-item loop:
for (let i = 0; i < 1000; i++) {
  process(items[i]);
}
// Set a conditional breakpoint on the process() line with: i === 500
// No code changes needed — pauses only on that iteration.

Breakpoint Types

TypeWhen to Use
Line breakpointPause on a specific line of code
Conditional breakpointPause only when an expression is true (loop iterations, specific state)
DOM breakpointPause when an element is modified, attribute changes, or subtree changes
XHR/fetch breakpointPause when a specific URL or any network request is made
Event listener breakpointPause when a specific event type fires (click, keydown, etc.)

Stepping Through Code

Once paused on a breakpoint, use the stepping controls to navigate execution:

Step controls (in order of frequency): ├── Step Over (F10) — execute current line, go to next ├── Step Into (F11) — enter a function call to debug its internals ├── Step Out (Shift+F11) — finish current function and return to caller └── Resume (F8) — continue execution until next breakpoint

DOM Breakpoints

Right-click any element in the Elements panel → Break on → and choose:

  • Subtree modifications — when children are added or removed
  • Attribute modifications — when attributes like class or style change
  • Node removal — when the element itself is removed from the DOM

This is invaluable when something is changing the DOM and you don't know what code is responsible. The debugger pauses on the exact line that made the change.

The debugger Statement

You can insert a debugger; statement directly in your code. When DevTools is open, execution pauses at that line. This is useful when you can't easily set a breakpoint in the Sources panel (e.g., code in a setTimeout or an event handler that's hard to reach).

debugger statement
function calculateTotal(items) {
  debugger;  // pause here when DevTools is open
  return items.reduce((sum, item) => sum + item.price, 0);
}

Remove debugger statements before committing — they'll pause execution for anyone with DevTools open.

Debugging Methodology

A systematic approach to debugging separates productive developers from frustrated ones. Follow this process every time:

Step 1: Reproduce Reliably

You cannot fix what you cannot trigger. Identify the exact steps that cause the bug. If it's intermittent, find the pattern (network speed, specific input, timing).

Step 2: Read the Error

The console gives you a file name, line number, and stack trace. Read it carefully. "Cannot read property of undefined" means something you expect to be an object is undefined — trace back to where it was set.

Step 3: Set a Breakpoint, Not a console.log

Go to the file and line the error points to. Set a breakpoint one line before the error. Reload. When execution pauses, inspect variables in the Scope panel. What is the actual state vs. what did you expect?

Step 4: Form a Hypothesis

Before changing code, state what you think is wrong: "The user object is undefined because the API call hasn't resolved yet." This prevents random guess-and-check debugging.

Step 5: Make One Change

Change one thing. Test. Did it fix the bug? Did it break anything else? Change multiple things at once and you won't know which one worked.

Step 6: Verify the Fix

Run the reproduction steps. The bug should be gone. Then test edge cases — what happens with empty input, null values, or unexpected data?

Debugging process: Reproduce → Read Error → Set Breakpoint → Inspect State → Form Hypothesis → Make One Change → Verify If not fixed → Refine Hypothesis → Repeat
The "Random Fix" Trap

Making random changes and reloading until the bug goes away is the most common junior mistake. You haven't fixed the bug — you've hidden it. It will reappear with different input. Always understand the root cause before making changes.

Common Debugging Scenarios

"X is not a function" or "Cannot read property of X"

Set a breakpoint at the failing line. Inspect what the variable actually contains. Trace back to where it should have been set. Common causes: API response missing a field, async code hasn't resolved, or a function parameter wasn't passed.

Event handler not firing

Check: Is the element selected correctly? Is the event name correct (it's click, not onclick)? Is the element dynamically added after the listener was attached? Use event delegation or attach the listener after the element exists. Check for JS errors before registration that may have stopped execution.

Async code executing in wrong order

Use breakpoints inside async functions to trace actual execution order. Common issue: forgetting await — the code continues synchronously while the promise is still pending.

State updates not reflected in UI

Verify you're modifying the correct variable. With objects or arrays, ensure you're creating a new reference rather than mutating. Check that the UI re-render is triggered.

Junior Developer

How a Junior Might Think

  • Uses var everywhere and doesn't understand scope issues
  • Writes deeply nested callbacks instead of using promises or async/await
  • Picks quote types at random, escapes unnecessarily
  • Concatenates strings with + even with variables
  • Uses click for everything, attaches events via HTML onclick attributes
  • Doesn't know the difference between blur, change, and input
  • Manipulates the DOM directly without considering performance
  • Doesn't handle promise rejections or network errors
  • Sprinkles console.log everywhere, never opens DevTools breakpoints
  • Makes random code changes until the bug "seems fixed"
Senior Developer

How a Senior Thinks

  • Uses const by default, let when reassignment is needed, never var
  • Writes async code with async/await and proper error handling
  • Uses single quotes by convention, switches to template literals for interpolation
  • Chooses the quote type that minimizes escaping based on string contents
  • Matches event type to intent: submit on forms, input with debounce, blur for validation
  • Uses event delegation for dynamic lists, addEventListener over HTML attributes
  • Always handles the error case — network failures, empty responses, timeouts
  • Uses breakpoints and systematic debugging methodology — reproduce, isolate, fix, verify
  • Reads stack traces and understands the root cause before changing code
Cheat Sheet
ConceptKey Takeaway
VariablesPrefer const. Use let for reassignment. Never use var.
FunctionsArrow functions inherit this from parent scope. Regular functions have their own this.
DOMUse querySelector/querySelectorAll. Prefer addEventListener over HTML attributes.
EventsMatch event to intent: input for live updates, change for committed values, blur for validation, submit on the form, not click on the button.
Asyncasync/await is syntactic sugar over promises. Always use try/catch.
StringsSingle quotes ' by convention. Template literals ` for interpolation/multi-line. Double quotes " for JSON. Choose the quote that needs no escaping.
Modern JSDestructure objects and arrays. Use map/filter/reduce instead of loops.
