Chapter 3

The Command Line & Local Dev

Terminal basics, errors, JSON, and getting your development environment running

Before you write your first line of code, you need to know how to talk to your computer. The terminal (also called command line, shell, or console) is how developers navigate files, run programs, and control their environment through text commands instead of clicking. This chapter covers everything you need to get started: terminal commands, understanding errors, working with JSON, and setting up a local development server.

What You'll Need First

Before we start, make sure you have these two things installed on your computer:

1. A Text Editor (Where You'll Write Code)

You need a program to write and edit code. We recommend Visual Studio Code — it's free, beginner-friendly, and works on Windows, Mac, and Linux.

  • Download: https://code.visualstudio.com/
  • Why VS Code? It has a built-in terminal, syntax highlighting (colors that make code easier to read), and thousands of extensions for different programming languages.

2. Git (For Version Control — Chapter 01)

Git tracks changes to your code. We'll use it in the next chapter, so install it now:

  • macOS: Open Terminal and run xcode-select --install
  • Windows: Download from git-scm.com
  • Linux: Run sudo apt install git in your terminal

3. How to Open Your Terminal

This is the #1 question from complete beginners — here's how:

  • macOS: Press Cmd + Space, type "Terminal", and press Enter. Or go to Applications → Utilities → Terminal.
  • Windows: Press Windows key + R, type "powershell", and press Enter. Or search for "PowerShell" in the Start menu.
  • Linux: Press Ctrl + Alt + T — this opens the terminal on most Linux distributions.

Pro tip: Pin Terminal to your taskbar or dock so it's easy to find. You'll open it every day.

Terminal Basics

The terminal (also called command line, shell, or console) is a text-based way to control your computer. On macOS it is called Terminal. On Windows use PowerShell or Windows Terminal. On Linux it is whatever terminal emulator you have installed. They all work the same way: you type a command and press Enter.

There are about five commands that cover 90% of what you will do:

CommandWhat It DoesExample
pwdPrint working directory — shows where you are/Users/you/projects
lsList files in the current directoryindex.html style.css js/
cdChange directory — move into a foldercd my-project
mkdirMake a new directory (folder)mkdir css creates a css/ folder
rmRemove a file (add -r for folders)rm old.txt deletes a file, rm -r old-folder/ deletes a folder

File Paths

Paths tell the computer where a file is. Two kinds:

  • Absolute path — starts from the root of your computer: /Users/you/projects/my-site/index.html. This always points to the same place regardless of where you are.
  • Relative path — starts from where you are right now: ./css/style.css means "the style.css inside the css folder next to me." The ./ means "current directory," ../ means "parent directory."

Tab completion: type the first few letters of a file or folder name and press Tab — the terminal will auto-complete it. If nothing happens, press Tab twice to see all options. This saves more time than any other terminal skill.

What Are File Extensions? (.html, .css, .js)

You've seen files like index.html, style.css, and app.js. The part after the dot (html, css, js) is the file extension — it tells your computer what type of file it is:

  • .html — an HTML web page
  • .css — a CSS stylesheet (how the page looks)
  • .js — a JavaScript file (how the page behaves)
  • .json — a JSON data file
  • .txt — plain text

Important: If you save a file as mypage.txt but the content is HTML, it won't render as a webpage when you open it in a browser. The extension tells the browser how to handle the file. Always use the correct extension!

Folder = Directory

You might see the words "folder" and "directory" used interchangeably. They mean the same thing — a container that holds files and other folders. In the terminal, "navigating to a directory" just means "going into that folder."

The Terminal Is Your Friend

Every developer spends hours in the terminal every day. It feels foreign at first because there are no visual cues — but that is its strength. You can do in one command what takes ten clicks in a GUI. Learn these five commands well, and the rest (cp, mv, grep, find) will come naturally as you need them.

"command not found" — You Are in the Wrong Folder

The most common terminal error. npm install fails with "command not found" because you are not in the project folder. python manage.py fails because you're in the wrong directory. Before running any command, always check pwd (where am I?) and ls (what's here?). If your project has a package.json, make sure you're in the same folder as it.

Reading Error Messages

Errors are not failures — they are information. The computer is telling you exactly what went wrong. Learning to read error messages is the single most important debugging skill.

The Anatomy of an Error

A typical JavaScript error
Uncaught TypeError: Cannot read properties of undefined (reading 'name')
    at Object.<anonymous> (script.js:10:15)
    at HTMLButtonElement.handleClick (script.js:22:5)
  1. Error type — TypeError, ReferenceError, SyntaxError. This tells you the category of problem.
  2. Error message — "Cannot read properties of undefined". You tried to access .name on something that is undefined.
  3. Stack trace — the list of files and line numbers showing the call chain. The top line is where the error happened. The lines below show what called what.
Error TypeMeaningCommon Cause
ReferenceError: X is not definedYou used a variable name that does not existTypo in variable name, forgot to declare with let/const, or scope issue
TypeError: Cannot read property of null/undefinedYou tried to access a property on a value that has noneAn API returned null, or you selected a DOM element that does not exist
SyntaxError: Unexpected tokenYour code has a typo in its structureMissing comma, extra bracket, unclosed string, using = instead of == or ===
404 Not FoundThe file or URL does not exist on the serverWrong file path, missing file, broken link
500 Internal Server ErrorThe server code crashedBackend error — check your server logs

The Debugging Workflow

  1. Read the error message — do not panic. Read it word by word. It tells you the file, the line number, and what went wrong.
  2. Look at the file and line — go to the line number mentioned. Look at what is on that line and the lines right before it.
  3. Add console.log() — log the values of variables right before the error. Is the variable what you expect?
  4. Search the error — copy the exact error message into Google (or your search engine of choice). Chances are thousands of people have had the exact same error. Stack Overflow will have an answer.
  5. Make one change — fix what you think is wrong, save, and reload. If it still errors, you eliminated one possibility. Try the next.
Every Error Is a Lesson

The first time you see "Cannot read property of undefined" it is confusing. The 50th time, you recognize it instantly. Experienced developers are not people who avoid errors — they are people who have seen every error a hundred times and know the fix without thinking. Make friends with your error messages. They are the most honest teachers you have.

JSON Syntax

JSON (JavaScript Object Notation) is the most common data format on the web. APIs return JSON. Configuration files use JSON. The SPA project in Chapter 09 is entirely driven by a config.json file. Understanding JSON syntax is essential — and the rules are strict.

Valid JSON — structure rules
{
  "name": "Alice",           // ✅ Keys MUST be double-quoted
  "age": 30,                 // ✅ Valid types: string, number, boolean, null, object, array
  "isAdmin": false,          // ✅ Booleans are lowercase true/false
  "bio": null,               // ✅ null is lowercase
  "tags": ["html", "css"],   // ✅ Arrays use square brackets
  "address": {               // ✅ Objects nested with curly braces
    "city": "New York"
  }                          // ❌ No trailing comma after last property!
}
RuleWhy It Catches Beginners
Keys must be double-quotedJavaScript objects don't need quotes on keys. JSON does. This is the #1 JSON mistake.
No trailing commas[1, 2, 3,] is invalid. JavaScript allows trailing commas in arrays/objects. JSON does not.
Strings must use double quotes'hello' is not valid JSON. Only "hello".
No commentsUnlike code, JSON does not support // or /* */ comments. Use a separate README or schema file.
Only six valid typesstring, number, boolean, null, object, array. No dates, no undefined, no functions.
Testing your JSON
# Validate a JSON file from the terminal
python -m json.tool config.json

# If the JSON is valid, it prints it neatly formatted.
# If it is invalid, it shows an error with the exact line and character.
JSON Is Everywhere

API responses, configuration files, database exports, and this textbook's content system — all use JSON. The rules are simple but unforgiving. A missing comma or an extra quote breaks the entire file. Use a JSON validator (jsonlint.com) whenever you get an error you can't find visually.

Setting Up Your Local Development Environment

Before you write your first line of code, you need to know how to run your work locally. There are two common scenarios:

Scenario 1: Static Site (No package.json)

A static site is just HTML, CSS, and JavaScript files. It needs no build tools. You can serve it with any simple HTTP server:

Start a static server
# Python (comes with most systems — try python or python3)
python -m http.server 8080

# Node.js (if you have it installed)
npx serve .

# Then open in your browser:
# http://localhost:8080

package.json is not required for static sites. It is only needed when you use Node.js tooling — bundlers (Vite, Webpack), frameworks (React, Next.js), or build scripts. If you don't have a package.json, you have a static site.

What Is localhost and :8080?

localhost means "this computer" — it's a special word for your own machine. When you see http://localhost:8080, it means "a website running on my computer, on port 8080." The :8080 is the port number — think of it as "which door" the server is using. Your computer can run many servers at once, each on a different port.

Scenario 2: Node Project (Has package.json)

When you start a new project from scratch, you create the package.json yourself:

Creating a new Node project
# Create a folder and enter it
mkdir my-project
cd my-project

# Create a package.json (answer the prompts or use --yes to skip them)
npm init
# or
npm init --yes    # skips all questions, uses defaults

# Your project now has a package.json — install dependencies:
npm install
# or install a specific package:
npm install express

# Start the dev server (command varies — check package.json "scripts")
npm run dev

When you clone an existing project (download from GitHub), it already has a package.json. You skip the npm init step and go straight to npm install:

Joining an existing Node project
# Install everything the project needs
npm install

# The node_modules/ folder appears automatically
# Start developing
npm run dev

Project Type Quick Decision

Look in your project folder. ├── Does it have a package.json? │ ├── YES → Node project. Run `npm install` then `npm run dev`. │ │ The dev server is already configured for you. │ └── NO → Static site. Use `python -m http.server 8080` │ or `npx serve .`. └── Not sure? Run `ls` and look for index.html. No package.json? It's a static site.

SPA Routing Fallback (Common Pitfall)

Single-page applications use client-side routing — the URL changes but the server always serves the same index.html. If you refresh a page like /about, the server looks for a file named about.html, doesn't find it, and returns a 404. The fix depends on your server:

  • Dev servers (Vite, Webpack Dev Server): configure history API fallback
  • Hash routing: use /#/about instead of /about. The hash portion is never sent to the server — the browser handles it client-side. This is the simplest fix and works everywhere.
  • Static hosting (Cloudflare Pages, Netlify, Vercel): configure redirect rules to serve index.html for all routes
Common Mistake

If your SPA works on npm run dev but breaks with a 404 after deploying, or breaks when you refresh a non-root page, this is almost always the SPA routing fallback problem. The solution is either hash routing or a server rewrite rule.

rm -rf Has No Undo — There Is No Trash Bin

When you run rm -rf foldername, the files are deleted immediately. There is no "undo." There is no "trash can." They are gone forever unless you have a backup or Git commit. The -rf flags mean "recursive" (delete everything inside) and "force" (don't ask for confirmation).

Beginner mistake: rm -rf * inside the wrong directory wipes everything. Always double-check pwd (print working directory) before any rm command. Tip: use rm -ri (interactive) to confirm each deletion, or test with ls first.

Case Sensitivity — File.txt Is Not file.txt

Linux/macOS file systems are case-sensitive. File.txt, file.txt, and FILE.txt are three different files. This trips up beginners who name a file MyFile.txt in their code but try to open myfile.txt.

Windows is case-insensitive (by default), so code that works on your machine breaks on deployment. Always match the exact case. Type ls to see what's actually there — the terminal won't guess what you meant.

Dev Environment Recovery

When you come back to a project after a break (or after cloning from GitHub), or when things stop working, follow this checklist:

Starting Fresh (Cloned from GitHub)

First time setting up a project
# 1. Clone the repository
git clone https://github.com/username/project.git
cd project

# 2. Install dependencies
npm install

# 3. Copy environment variables (if .env.example exists)
cp .env.example .env

# 4. Start the dev server
npm run dev

Returning to a Project

Quick restart
# 1. Navigate to your project
cd ~/path/to/project

# 2. Pull latest changes from GitHub (if working with a team)
git pull

# 3. What kind of project is it?
ls                      # look for index.html or package.json

# 4a. Static site — just serve it
python -m http.server 8080

# 4b. Node project — install and start
npm install             # dependencies may have changed
npm run dev

Restart Order (If Your System Has Multiple Parts)

  1. Backend / API — start any server first (Express, PHP, database)
  2. Frontend dev server — start your frontend server second
  3. Browser cache refresh — hard refresh (Ctrl+Shift+R / Cmd+Shift+R) to clear cached files

If things still don't work: Ctrl+C to stop the server, restart it, then hard refresh the browser. This solves the majority of "it was working yesterday" problems.

"It Worked on My Machine"

The most famous phrase in software development. Environment differences cause real bugs: different Node versions, missing dependencies, OS-specific paths. Use node --version to check your Node version, use nvm to manage multiple Node versions, and always commit your package-lock.json to ensure everyone gets the same dependency versions.

Junior Developer

How a Junior Might Think

  • Afraid of the terminal — prefers clicking in a GUI
  • Panics when an error appears instead of reading it
  • Doesn't know how to start a local server — reaches for a complex tool when python -m http.server would do
  • Ignores the stack trace and guesses at the fix
  • Doesn't understand the difference between npm install and npm ci
Senior Developer

How a Senior Thinks

  • Lives in the terminal — uses it for everything from git to file operations
  • Reads the full error message and stack trace before touching code
  • Can start a local dev server without thinking — chooses the right tool based on project type
  • Uses npm ci in CI and npm install for local development
  • Checks Node version and lockfile state when "it worked on my machine" strikes
