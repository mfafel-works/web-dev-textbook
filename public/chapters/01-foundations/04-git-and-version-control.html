Git & Version Control

Tracking changes, branching, collaborating, and undoing mistakes

Git is a version control system. It tracks changes to your files, lets you save snapshots (called commits), and makes it possible to collaborate with other developers without overwriting each other's work. This is not optional. Every developer uses Git from day one. This chapter covers everything from your first commit to collaborating on a team.

What Is Version Control?

Imagine writing a document without "undo." Every change is permanent. If you delete something, it is gone. If you want to try a different approach, you have to duplicate the file. That is what coding without Git is like.

Git gives you:

  • Snapshots — every commit is a save point you can return to
  • History — see who changed what and when
  • Branches — work on multiple features simultaneously without interference
  • Collaboration — merge changes from different people without losing work

Installing and Configuring Git

Check if Git is installed:

Check and install Git
# Check if Git is installed
git --version

# If not installed:
# macOS:  xcode-select --install
# Ubuntu: sudo apt install git
# Windows: https://git-scm.com/downloads

Configure your name and email — this is how your commits will be identified:

First-time Git setup
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# Check your configuration
git config --list

This is a one-time setup. The --global flag applies to all projects on your computer. You only need to change this if you use different identities for work and personal projects.

Your First Git Workflow

The basic cycle for any project:

Starting a new project with Git
# 1. Create and enter your project folder
mkdir my-first-site
cd my-first-site

# 2. Initialize Git (creates a hidden .git folder)
git init

# 3. Create some files (index.html, style.css, etc.)
# ...

# 4. See what Git notices has changed
git status

# 5. Stage files — tell Git "I want to save these"
git add index.html
git add .                  # add everything in the folder

# 6. Commit — save a snapshot with a message
git commit -m "Initial commit: basic HTML structure"

The Three States of Git

Every file in a Git repository is in one of three states:

Working Directory (files on your computer) ↓ git add Staging Area (files ready to be committed) ↓ git commit Local Repository (committed to history)
  • Modified — you changed the file but haven't staged it yet
  • Staged — you marked the file to be included in the next commit
  • Committed — the file is safely stored in Git's history

This three-stage system gives you control over what goes into each commit. You might edit three files but only commit two — the third is not ready yet.

Everyday Git Commands

CommandWhat It Does
git statusShow what files are changed, staged, or untracked

Commit Early, Commit Often

Each commit is a save point you can return to. Made a mess? git checkout -- . reverts to the last commit. Deleted something? Restore it from a previous commit. The safety net of Git is what lets you experiment without fear. Commit every time you get something working — even if it is not finished. A good commit message is present tense, imperative: "Add footer section," not "Added footer section" or "I added the footer."

Connecting to GitHub

Git runs on your computer. To back up your code and collaborate, you push it to a remote hosting service like GitHub, GitLab, or Bitbucket:

Push to GitHub
# 1. Create a new repository on github.com (don't add README — you have one)
# 2. Link your local repo to the remote:
git remote add origin https://github.com/your-username/my-first-site.git

# 3. Push your commits:
git push -u origin main    # -u sets the upstream, so next time just "git push"

Cloning an Existing Repository

To get a copy of someone else's project (or your own from another computer):

Clone a repository
git clone https://github.com/username/project.git
cd project
# You now have a full copy of the entire project and its history

Cloning automatically sets up the remote as origin, so you can push and pull immediately.

Branching and Merging

Branches let you work on multiple things at the same time without interfering. The default branch is main (or master on older repos). You create a new branch for each feature or bug fix.

Creating and Switching Branches

Branch workflow
# Create a new branch and switch to it
git checkout -b feature-navbar

# Make changes and commit them
git add .
git commit -m "Add responsive navbar"

# Switch back to main
git checkout main

# Merge the feature branch into main
git merge feature-navbar

# Delete the branch (cleanup after merging)
git branch -d feature-navbar

Why Branches Matter

Without branches, everyone on a team commits to main. If you are working on a feature that takes three days, your half-finished code is on main — breaking things for everyone. With branches, you work in isolation. When the feature is complete and tested, you merge it back. Branches are cheap and fast — create them liberally.

Branch Naming Conventions

Teams use naming patterns to keep branches organized: feature/add-login, bugfix/fix-header-overlap, chore/update-deps. This makes it clear what each branch does without reading the code. Some teams use ticket numbers: feature/PROJ-123-user-auth.

Merge Conflicts

When two branches modify the same line of the same file, Git cannot automatically merge. This is called a merge conflict. It is not a disaster — it is a normal part of collaboration.

Resolving a merge conflict
# Git tells you which files have conflicts
git status
# both modified:   index.html

# Open the file — you'll see conflict markers:
<<<<<<< HEAD
<h1>Welcome to my site</h1>
=======
<h1>Welcome to our site</h1>
>>>>>>> feature-branding

# Edit the file to keep the correct version, remove the markers:
<h1>Welcome to our site</h1>

# Stage the resolved file and commit
git add index.html
git commit -m "Resolve merge conflict in heading"

The <<<<<<< and >>>>>>> markers show you both versions. HEAD is your current branch. The other section is the branch being merged in. Edit, remove the markers, and commit.

Undoing Changes

Everyone makes mistakes. Git gives you several ways to undo, depending on what you need:

ScenarioCommandWhat It Does
Unstaged changes (haven't run git add)git checkout -- <file>Revert file to last committed version
Staged changes (ran git add but not commit)git reset HEAD <file>Unstage the file (changes remain in working directory)
Committed but not pushedgit commit --amend -m "new message"Change the last commit message or add forgotten files
Committed and pushedgit revert <commit-hash>Create a new commit that undoes the old one (safe — does not rewrite history)
Need to go back to an old stategit reset --hard <commit-hash>Discard all changes after that commit. Dangerous — only use locally
Never Rewrite Pushed History on Shared Branches

Commands like git reset --hard and git commit --amend rewrite Git history. If you have pushed your commits and someone else has pulled them, rewriting history creates chaos. Use git revert instead — it creates a new commit that undoes the old one without destroying history.

I Added node_modules to .gitignore But It's Still Committing

The .gitignore file only prevents untracked files from being added. If you already committed node_modules before adding it to .gitignore, Git keeps tracking it. The file was already in the history.

Fix: remove it from Git's tracking (but keep the files locally):

git rm -r --cached node_modules/

Then commit. The --cached flag removes files from tracking without deleting them from your folder. After this, .gitignore will work.

I Pushed and Got "Updates Were Rejected"

Git rejects your push when the remote has commits you don't have locally. This happens when: (1) you didn't pull before pushing, or (2) someone else pushed while you were working.

The fix is always:

git pull --rebase origin main
git push origin main

The --rebase flag puts your commits on top of the remote's, creating a clean linear history. If there are conflicts, Git will pause and ask you to resolve them — edit the files, then git add . and git rebase --continue.

The .gitignore File

Not everything in your project folder should be tracked by Git. The node_modules folder, compiled files, environment variables with secrets, and operating system files (.DS_Store on macOS) should be excluded:

Example .gitignore
# Dependencies
node_modules/

# Build output
dist/
build/

# Environment variables with secrets
.env
.env.local

# OS files
.DS_Store
Thumbs.db

# IDE files
.vscode/
.idea/

# Logs
*.log

Create a .gitignore file at the root of your project and commit it. The files listed in it will be ignored by Git — they won't appear in git status or be included in commits.

Check if a file is being ignored
# See why a file is being ignored
git check-ignore -v node_modules/package.json
The node_modules Golden Rule

Never commit node_modules/. It is huge (hundreds of megabytes), platform-specific, and entirely reproducible from package.json and package-lock.json. If you accidentally commit it, add it to .gitignore and remove it from tracking with git rm -r --cached node_modules.

Collaboration Workflows

GitHub Flow

The simplest and most common collaboration pattern:

  1. Create a branch off main for your feature or fix
  2. Make changes and commit on that branch
  3. Push the branch to GitHub
  4. Open a pull request (PR) — a request to merge your branch into main
  5. Team members review your code, leave comments, and request changes
  6. Make additional commits addressing feedback
  7. Merge the pull request
  8. Delete the feature branch
Full GitHub Flow example
# 1. Start from an up-to-date main
git checkout main
git pull

# 2. Create a feature branch
git checkout -b feature-add-search

# 3. Make changes and commit
git add .
git commit -m "Add search bar to header"

# 4. Push branch to GitHub
git push -u origin feature-add-search

# 5. Open a pull request on GitHub.com
# 6. Address review feedback
git add .
git commit -m "Fix search input width per review"
git push

# 7. Merge PR on GitHub, then delete branch
# 8. Update local main
git checkout main
git pull

Pull Request Etiquette

  • Keep PRs small — one feature or fix per PR. A 500-line PR is hard to review
  • Write a clear title and description explaining what and why
  • Reference the issue number if applicable: "Closes #42"
  • Reviewers: be constructive, not critical. Suggest, don't demand
  • Authors: don't take feedback personally — code review is about the code, not you
Junior Developer

How a Junior Might Think

  • Only knows git add, git commit, git push — panic-Googles everything else
  • Commits to main directly without branching
  • Writes vague commit messages like "fix stuff" or "update"
  • Panics at merge conflicts and asks for help instead of resolving them
  • Commits node_modules or .env by accident
  • Does not understand the difference between git pull and git fetch
Senior Developer

How a Senior Thinks

  • Uses branches liberally — every feature, fix, or experiment gets its own branch
  • Writes descriptive commit messages in present tense: "Add validation to contact form"
  • Resolves merge conflicts confidently by reading the conflicting code
  • Maintains a clean .gitignore and never commits generated files
  • Uses git revert for safe undo, never rewrites shared history
  • Reviews pull requests with constructive feedback and a collaborative tone
