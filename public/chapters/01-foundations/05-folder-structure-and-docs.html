Chapter 5

Folder Structure & Documentation

Project organization patterns, naming conventions, README standards, code comments, and API docs

A well-organized project communicates its architecture at a glance. Good documentation preserves knowledge and saves time. Together, they are the difference between a project that is a pleasure to work on and one that drains energy from everyone who touches it. This chapter covers both — because a perfect folder structure without documentation is still confusing, and perfect documentation without a clear structure is still hard to navigate.

Why Structure and Documentation Matter

The cost of neglect is invisible but real:

  • Duplicated code — without a clear structure, developers rewrite existing code because they cannot find it
  • Onboarding friction — new developers spend their first week learning where things are instead of learning how the system works
  • Lost knowledge — undocumented decisions live in people's heads. When they leave, the knowledge leaves with them
  • Context-switch tax — every question answered by documentation is an interruption that did not happen
Rule: a new developer should find any file within 30 seconds and understand any function's purpose within 2 minutes.

Small Project Structure (1-15 files)

my-project/ ├── index.html ├── about.html ├── css/ │ └── style.css ├── js/ │ └── main.js └── assets/ └── images/

Exacty one place for each file type. When the project grows, js/ becomes src/ with subdirectories. Grow the structure organically — add a folder only when the current one becomes crowded.

Medium Project Structure (15-100 files)

project/ ├── public/ │ ├── index.html │ └── favicon.ico ├── src/ │ ├── components/ │ │ ├── Header.jsx │ │ ├── Footer.jsx │ │ └── Button.jsx │ ├── pages/ │ │ ├── Home.jsx │ │ └── About.jsx │ ├── styles/ │ │ ├── global.css │ │ └── variables.css │ ├── utils/ │ │ ├── formatDate.js │ │ └── validate.js │ └── services/ │ └── api.js ├── tests/ │ └── components/ └── package.json

This is the standard type-based organization. The services/ directory isolates external system interactions (APIs, databases) — making it easy to test and swap implementations.

Enterprise / Feature-Based Structure

When src/components/ exceeds 30 files, switch to feature-based organization. Each feature is self-contained:

project/ ├── apps/ │ ├── web/ │ ├── admin/ │ └── api/ ├── packages/ │ ├── shared-ui/ │ └── shared-utils/ └── features/ ├── auth/ │ ├── components/ │ ├── hooks/ │ ├── services/ │ └── tests/ └── dashboard/ ├── components/ ├── hooks/ └── tests/

Each feature team owns their directory. Shared code is deliberately extracted to packages/. This scales to hundreds of developers because features can be understood in isolation.

Type-Based vs Feature-Based

FactorType-BasedFeature-Based
Size<100 files100+ files
Team1-5 developers5-100+ developers
Code sharingEasy — one utils folderRequires explicit extraction
Feature isolationCode spread across 6 foldersEverything in one directory

Start type-based. Switch to feature-based when components/ exceeds 30 files or you have 3+ developers on different features.

Naming Conventions

ConventionWhenExamples
kebab-caseFiles, folders, URLs, CSS classesuser-profile.js, style-guide.css
PascalCaseComponents, classesUserProfile.jsx, HttpClient.js
camelCaseVariables, functions, utilitiesformatDate.js, apiClient.js
UPPER_CASEConstants, env varsAPI_URL, MAX_RETRY_COUNT

A file name should tell you what it contains. If you have to open it to understand its purpose, the naming is wrong.

Practical Rules for Any Project

  • One thing per file. One component, one function, one class per file.
  • Co-locate what changes together. Put a component and its test in the same folder.
  • Max depth of 4. src/features/auth/components/Button.jsx is the deepest you should ever go.
  • Avoid generic names. utils/helpers.js is a dumping ground. Name by domain.
  • Index files for exports. Each folder re-exports its public API so consumers import from the folder, not individual files.

The Anatomy of a Great README

A README answers five questions, in order:

1. What is this project?

One sentence. Not "a web app for tasks" — "a collaborative task manager with real-time updates and offline support."

2. Why does it exist?

What problem does it solve? How is it different from alternatives?

3. How do I set it up?

Step-by-step from zero to running. Prerequisites, install commands, config steps. A new developer should produce visible output in under 2 minutes.

4. How do I use it?

Basic examples — most common commands, API calls, or configuration changes. Not every edge case — enough to be immediately productive.

5. How do I contribute?

Contribution guidelines, code style, PR process. This signals the project is maintained and welcoming.

README template
# Project Name

Short description: one paragraph about what this does and why.

## Prerequisites
- Node.js 18+
- PostgreSQL 14+

## Installation
```bash
git clone https://github.com/you/project.git
cd project
npm install
cp .env.example .env
npm run dev
```

## Usage
### Create a page
Add an entry to config.json:
```json
{ "route": "/blog", "page": "blog", "blocks": [...] }
```

### Deploy
```bash
npm run build && npm run deploy
```

## API Reference
`GET /api/posts` — Returns paginated posts.
| Param | Type | Default | Description |
|-------|------|---------|-------------|
| page  | number | 1 | Page number |

## Contributing
See CONTRIBUTING.md. PRs must include tests and docs.

Code Comments Done Right

Comment the why, not the what. Code tells you what it does. Comments tell you why it does it that way.

Bad vs good comments
// BAD: states the obvious
counter += 1;  // Increment counter by 1

// GOOD: explains intent
// Add retry delay to avoid rate-limit backoff avalanche.
// Without this, all retries fire simultaneously
// and the server sees the same load as the original burst.
await delay(1000 * attempt);

When to Comment

  • Non-obvious tradeoffs — why a Set was chosen over an Array
  • Business rules — "Orders over $500 require approval per policy FIN-401"
  • Workarounds — "Safari does not fire load for cached images (WebKit bug #xyz)"
  • Complex algorithms — a brief overview so the reader can study the details elsewhere

When NOT to Comment

  • Self-explanatory code
  • Dead code — delete it instead of commenting it out
  • TODO without a ticket number — "TODO: fix later" lives forever

Documentation-Driven Development

Write the docs first, then implement. This forces precision before coding.

  1. Define the contract — API spec, function signature, or component interface
  2. Review the contract — share with stakeholders before writing code
  3. Write the tests — against the contract (they will fail)
  4. Implement — write code until tests pass
  5. Document edge cases — update docs with anything discovered during implementation
Documentation Is an Investment

Every hour spent writing good documentation saves at least three hours of future debugging, onboarding, and answering questions. The best time to write it is when the code is fresh — ideally in the same PR that introduces the feature. Documentation that is added later is documentation that is never added.

Junior Developer

How a Junior Might Think

  • Creates files wherever is convenient — no consistent structure
  • Writes no README or a one-liner with just the project name
  • Comments restate the code ("increment counter by 1")
  • Does not refactor the structure as the project grows
  • Lets documentation become outdated immediately
Senior Developer

How a Senior Thinks

  • Plans structure around domain and team size — refactors proactively
  • Treats the README as the project's front door
  • Documents rationale, tradeoffs, and design decisions
  • Writes docs first, then implements
  • Keeps documentation in the same PR as code changes
